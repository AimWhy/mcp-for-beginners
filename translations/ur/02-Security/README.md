# بہترین سیکیورٹی طریقے

Model Context Protocol (MCP) کو اپنانا AI سے چلنے والی ایپلیکیشنز میں طاقتور نئی صلاحیتیں لاتا ہے، لیکن یہ روایتی سافٹ ویئر کے خطرات سے آگے بڑھ کر منفرد سیکیورٹی چیلنجز بھی پیدا کرتا ہے۔ محفوظ کوڈنگ، کم سے کم اختیارات، اور سپلائی چین سیکیورٹی جیسے قائم شدہ خدشات کے علاوہ، MCP اور AI کے کاموں کو نئے خطرات کا سامنا ہے جیسے کہ prompt injection، tool poisoning، اور dynamic tool modification۔ اگر انہیں مناسب طریقے سے منظم نہ کیا جائے تو یہ خطرات ڈیٹا کے اخراج، پرائیویسی کی خلاف ورزی، اور غیر ارادی نظامی رویے کا باعث بن سکتے ہیں۔

یہ سبق MCP سے جڑے سب سے متعلقہ سیکیورٹی خطرات کا جائزہ لیتا ہے—جس میں authentication، authorization، ضرورت سے زیادہ اجازتیں، بالواسطہ prompt injection، اور سپلائی چین کی کمزوریاں شامل ہیں—اور انہیں کم کرنے کے لیے عملی کنٹرولز اور بہترین طریقے فراہم کرتا ہے۔ آپ سیکھیں گے کہ Microsoft کے حل جیسے Prompt Shields، Azure Content Safety، اور GitHub Advanced Security کو کس طرح استعمال کرکے اپنے MCP نفاذ کو مضبوط بنایا جا سکتا ہے۔ ان کنٹرولز کو سمجھ کر اور نافذ کرکے، آپ سیکیورٹی خلاف ورزی کے امکانات کو نمایاں طور پر کم کر سکتے ہیں اور اپنے AI نظاموں کو مضبوط اور قابل اعتماد بنا سکتے ہیں۔

# سیکھنے کے مقاصد

اس سبق کے اختتام تک، آپ قابل ہوں گے:

- Model Context Protocol (MCP) کی وجہ سے پیدا ہونے والے منفرد سیکیورٹی خطرات کی نشاندہی اور وضاحت کرنا، جن میں prompt injection، tool poisoning، ضرورت سے زیادہ اجازتیں، اور سپلائی چین کی کمزوریاں شامل ہیں۔
- MCP سیکیورٹی خطرات کے مؤثر حفاظتی کنٹرولز کی وضاحت اور اطلاق کرنا، جیسے مضبوط authentication، کم سے کم اختیارات، محفوظ token مینجمنٹ، اور سپلائی چین کی تصدیق۔
- Microsoft کے حل جیسے Prompt Shields، Azure Content Safety، اور GitHub Advanced Security کو سمجھنا اور استعمال کرنا تاکہ MCP اور AI کے کاموں کی حفاظت کی جا سکے۔
- ٹول میٹا ڈیٹا کی تصدیق، dynamic تبدیلیوں کی نگرانی، اور بالواسطہ prompt injection حملوں کے خلاف دفاع کی اہمیت کو پہچاننا۔
- محفوظ کوڈنگ، سرور کو سخت کرنا، اور zero trust architecture جیسے قائم شدہ بہترین سیکیورٹی طریقوں کو اپنے MCP نفاذ میں شامل کرکے سیکیورٹی خلاف ورزیوں کے امکانات اور اثرات کو کم کرنا۔

# MCP سیکیورٹی کنٹرولز

کسی بھی نظام کو جو اہم وسائل تک رسائی رکھتا ہے، پوشیدہ سیکیورٹی چیلنجز کا سامنا ہوتا ہے۔ عام طور پر یہ چیلنجز بنیادی سیکیورٹی کنٹرولز اور تصورات کے صحیح اطلاق کے ذریعے حل کیے جا سکتے ہیں۔ چونکہ MCP ابھی نیا متعین کیا گیا ہے، اس کی وضاحت تیزی سے بدل رہی ہے اور جیسے جیسے پروٹوکول ارتقاء پذیر ہوتا ہے، اس میں شامل سیکیورٹی کنٹرولز بھی نکھرتے جائیں گے، جس سے کاروباری اور قائم شدہ سیکیورٹی آرکیٹیکچرز اور بہترین طریقوں کے ساتھ بہتر انضمام ممکن ہوگا۔

[Microsoft Digital Defense Report](https://aka.ms/mddr) میں شائع ہونے والی تحقیق کے مطابق، 98% رپورٹ شدہ خلاف ورزیاں مضبوط سیکیورٹی ہائجین کے ذریعے روکی جا سکتی ہیں اور کسی بھی قسم کی خلاف ورزی کے خلاف بہترین تحفظ یہ ہے کہ آپ اپنی بنیادی سیکیورٹی ہائجین، محفوظ کوڈنگ کے بہترین طریقے، اور سپلائی چین سیکیورٹی کو درست کریں — وہ آزمودہ طریقے جو ہم پہلے سے جانتے ہیں، اب بھی سیکیورٹی خطرات کو کم کرنے میں سب سے زیادہ مؤثر ہیں۔

آئیے دیکھتے ہیں کہ MCP اپنانے کے دوران آپ سیکیورٹی خطرات کو کیسے حل کرنا شروع کر سکتے ہیں۔

> **نوٹ:** درج ذیل معلومات **29 مئی 2025** تک درست ہیں۔ MCP پروٹوکول مسلسل ارتقاء پذیر ہے، اور مستقبل کے نفاذ میں نئے authentication پیٹرنز اور کنٹرولز شامل ہو سکتے ہیں۔ تازہ ترین اپ ڈیٹس اور رہنمائی کے لیے ہمیشہ [MCP Specification](https://spec.modelcontextprotocol.io/) اور سرکاری [MCP GitHub repository](https://github.com/modelcontextprotocol) اور [security best practice page](https://modelcontextprotocol.io/specification/draft/basic/security_best_practices) کو دیکھیں۔

### مسئلہ کا بیان  
اصل MCP وضاحت میں یہ فرض کیا گیا تھا کہ ڈیولپرز اپنا authentication سرور خود لکھیں گے۔ اس کے لیے OAuth اور متعلقہ سیکیورٹی پابندیوں کا علم ضروری تھا۔ MCP سرورز OAuth 2.0 Authorization Servers کے طور پر کام کرتے تھے، جو براہ راست صارف کی authentication کو منظم کرتے تھے بجائے اس کے کہ اسے Microsoft Entra ID جیسے بیرونی سروس کو تفویض کیا جائے۔ **26 اپریل 2025** سے MCP وضاحت میں ایک اپ ڈیٹ آئی ہے جو MCP سرورز کو صارف کی authentication کو بیرونی سروس کو تفویض کرنے کی اجازت دیتی ہے۔

### خطرات
- MCP سرور میں authorization منطق کی غلط ترتیب حساس ڈیٹا کے افشاء اور غلط طریقے سے نافذ کردہ رسائی کنٹرولز کا باعث بن سکتی ہے۔
- مقامی MCP سرور پر OAuth ٹوکن کی چوری۔ اگر چوری ہو جائے تو اس ٹوکن کو MCP سرور کی جگہ استعمال کرکے اس سروس کے وسائل اور ڈیٹا تک رسائی حاصل کی جا سکتی ہے جس کے لیے ٹوکن جاری کیا گیا ہے۔

#### Token Passthrough  
authorization وضاحت میں token passthrough واضح طور پر ممنوع ہے کیونکہ یہ کئی سیکیورٹی خطرات کو جنم دیتا ہے، جن میں شامل ہیں:

#### سیکیورٹی کنٹرولز کی بائی پاسنگ  
MCP سرور یا downstream APIs اہم سیکیورٹی کنٹرولز جیسے rate limiting، request validation، یا traffic monitoring نافذ کر سکتے ہیں جو ٹوکن کے audience یا دیگر credential پابندیوں پر منحصر ہوتے ہیں۔ اگر کلائنٹس براہ راست downstream APIs کے ساتھ ٹوکن حاصل کرکے استعمال کر سکتے ہیں بغیر MCP سرور کے انہیں درست طریقے سے validate کیے یا یہ یقینی بنائے کہ ٹوکن صحیح سروس کے لیے جاری کیے گئے ہیں، تو وہ ان کنٹرولز کو بائی پاس کر دیتے ہیں۔

#### احتساب اور آڈٹ ٹریل کے مسائل  
MCP سرور MCP کلائنٹس کی شناخت یا تمیز کرنے سے قاصر ہوگا جب کلائنٹس upstream-issued access token کے ساتھ کال کر رہے ہوں جو MCP سرور کے لیے opaque ہو سکتا ہے۔  
downstream Resource Server کے لاگز ایسی درخواستیں دکھا سکتے ہیں جو مختلف ماخذ یا شناخت سے آئی ہوں، بجائے اس کے کہ وہ MCP سرور ہو جو اصل میں ٹوکنز کو فارورڈ کر رہا ہو۔  
یہ دونوں عوامل واقعہ کی تحقیقات، کنٹرولز، اور آڈٹنگ کو مشکل بناتے ہیں۔  
اگر MCP سرور بغیر ٹوکن کے دعوے (جیسے کہ کردار، اختیارات، یا audience) یا دیگر میٹا ڈیٹا کی تصدیق کیے ٹوکنز کو پاس کرتا ہے، تو چوری شدہ ٹوکن رکھنے والا نقصان دہ فریق سرور کو ڈیٹا کے اخراج کے لیے پروکسی کے طور پر استعمال کر سکتا ہے۔

#### اعتماد کی حد کے مسائل  
downstream Resource Server مخصوص اداروں کو اعتماد دیتا ہے۔ یہ اعتماد ماخذ یا کلائنٹ کے رویے کے اندازوں پر مبنی ہو سکتا ہے۔ اس اعتماد کی حد کو توڑنا غیر متوقع مسائل کا باعث بن سکتا ہے۔  
اگر ٹوکن متعدد سروسز میں بغیر مناسب تصدیق کے قبول کیا جائے، تو ایک سروس کے سمجھوتہ ہونے والا حملہ آور اس ٹوکن کو دوسرے منسلک سروسز تک رسائی کے لیے استعمال کر سکتا ہے۔

#### مستقبل کی مطابقت کا خطرہ  
اگرچہ آج MCP سرور "خالص پراکسی" کے طور پر شروع ہو سکتا ہے، اسے بعد میں سیکیورٹی کنٹرولز شامل کرنے کی ضرورت پڑ سکتی ہے۔ ٹوکن audience کی مناسب علیحدگی سے سیکیورٹی ماڈل کو آگے بڑھانا آسان ہو جاتا ہے۔

### حفاظتی کنٹرولز

**MCP سرورز کو ایسے کسی بھی ٹوکن کو قبول نہیں کرنا چاہیے جو واضح طور پر MCP سرور کے لیے جاری نہ کیے گئے ہوں**

- **authorization منطق کا جائزہ لیں اور سخت کریں:** اپنے MCP سرور کی authorization نفاذ کا بغور آڈٹ کریں تاکہ صرف مطلوبہ صارفین اور کلائنٹس حساس وسائل تک رسائی حاصل کر سکیں۔ عملی رہنمائی کے لیے دیکھیں [Azure API Management Your Auth Gateway For MCP Servers | Microsoft Community Hub](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690) اور [Using Microsoft Entra ID To Authenticate With MCP Servers Via Sessions - Den Delimarsky](https://den.dev/blog/mcp-server-auth-entra-id-session/)۔  
- **محفوظ ٹوکن پریکٹس نافذ کریں:** [Microsoft کے بہترین طریقے برائے ٹوکن کی تصدیق اور عمر](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens) پر عمل کریں تاکہ access tokens کے غلط استعمال اور ٹوکن کی دوبارہ چوری کے خطرات کو کم کیا جا سکے۔  
- **ٹوکن اسٹوریج کی حفاظت کریں:** ہمیشہ ٹوکنز کو محفوظ طریقے سے ذخیرہ کریں اور انہیں آرام اور نقل و حمل کے دوران انکرپٹ کریں۔ عمل درآمد کی تجاویز کے لیے دیکھیں [Use secure token storage and encrypt tokens](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2)۔

# MCP سرورز کے لیے ضرورت سے زیادہ اجازتیں

### مسئلہ کا بیان  
MCP سرورز کو اس سروس/وسیلے کے لیے ضرورت سے زیادہ اجازتیں دی گئی ہو سکتی ہیں جس تک وہ رسائی حاصل کر رہے ہیں۔ مثال کے طور پر، ایک MCP سرور جو AI سیلز ایپلیکیشن کا حصہ ہے اور انٹرپرائز ڈیٹا اسٹور سے جڑتا ہے، اسے صرف سیلز ڈیٹا تک محدود رسائی ہونی چاہیے نہ کہ اسٹور کی تمام فائلوں تک۔ کم سے کم اختیارات کے اصول (جو قدیم ترین سیکیورٹی اصولوں میں سے ایک ہے) کے مطابق، کسی بھی وسیلے کو وہی اجازتیں ملنی چاہئیں جو اس کے کام کو انجام دینے کے لیے ضروری ہوں۔ AI اس میں اضافی چیلنج پیش کرتا ہے کیونکہ اسے لچکدار بنانے کے لیے درست اجازتوں کی تعریف کرنا مشکل ہو سکتا ہے۔

### خطرات  
- ضرورت سے زیادہ اجازتیں دینے سے ایسا ڈیٹا نکالنے یا ترمیم کرنے کی اجازت مل سکتی ہے جس تک MCP سرور کو رسائی نہیں دی گئی۔ اگر ڈیٹا ذاتی شناختی معلومات (PII) ہو تو یہ پرائیویسی کا مسئلہ بھی ہو سکتا ہے۔

### حفاظتی کنٹرولز  
- **کم سے کم اختیارات کا اصول نافذ کریں:** MCP سرور کو صرف وہی کم از کم اجازتیں دیں جو اس کے مطلوبہ کام انجام دینے کے لیے ضروری ہوں۔ ان اجازتوں کا باقاعدگی سے جائزہ لیں اور اپ ڈیٹ کریں تاکہ وہ ضرورت سے زیادہ نہ ہوں۔ تفصیلی رہنمائی کے لیے دیکھیں [Secure least-privileged access](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access)۔  
- **Role-Based Access Control (RBAC) استعمال کریں:** MCP سرور کو مخصوص وسائل اور کارروائیوں کے لیے سخت دائرہ کار والے رول تفویض کریں، وسیع یا غیر ضروری اجازتوں سے گریز کریں۔  
- **اجازتوں کی نگرانی اور آڈٹ کریں:** اجازتوں کے استعمال کی مسلسل نگرانی کریں اور رسائی کے لاگز کا آڈٹ کریں تاکہ ضرورت سے زیادہ یا غیر استعمال شدہ اختیارات کو جلد از جلد شناخت اور درست کیا جا سکے۔

# بالواسطہ prompt injection حملے

### مسئلہ کا بیان

نقصان دہ یا سمجھوتہ شدہ MCP سرورز صارف کے ڈیٹا کے افشاء یا غیر ارادی کارروائیوں کے ذریعے نمایاں خطرات پیدا کر سکتے ہیں۔ یہ خطرات خاص طور پر AI اور MCP پر مبنی کاموں میں اہم ہیں، جہاں:

- **Prompt Injection حملے:** حملہ آور prompts یا بیرونی مواد میں نقصان دہ ہدایات شامل کرتے ہیں، جس سے AI نظام غیر ارادی کارروائیاں کرتا ہے یا حساس ڈیٹا لیک ہو جاتا ہے۔ مزید جاننے کے لیے: [Prompt Injection](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/)  
- **Tool Poisoning:** حملہ آور ٹول میٹا ڈیٹا (جیسے وضاحتیں یا پیرامیٹرز) میں چالاکی سے تبدیلی کرتے ہیں تاکہ AI کے رویے کو متاثر کریں، ممکنہ طور پر سیکیورٹی کنٹرولز کو بائی پاس کریں یا ڈیٹا نکالیں۔ تفصیلات: [Tool Poisoning](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks)  
- **Cross-Domain Prompt Injection:** نقصان دہ ہدایات دستاویزات، ویب صفحات، یا ای میلز میں شامل کی جاتی ہیں، جو پھر AI کے ذریعے پروسیس ہوتی ہیں، جس سے ڈیٹا لیک یا چالاکی ہوتی ہے۔  
- **Dynamic Tool Modification (Rug Pulls):** ٹول کی تعریفیں صارف کی منظوری کے بعد تبدیل کی جا سکتی ہیں، بغیر صارف کی اطلاع کے نئے نقصان دہ رویے متعارف کراتے ہوئے۔

یہ کمزوریاں MCP سرورز اور ٹولز کو اپنے ماحول میں شامل کرتے وقت مضبوط تصدیق، نگرانی، اور سیکیورٹی کنٹرولز کی ضرورت کو اجاگر کرتی ہیں۔ مزید تفصیل کے لیے اوپر دیے گئے لنکس دیکھیں۔

**بالواسطہ Prompt Injection** (جسے cross-domain prompt injection یا XPIA بھی کہا جاتا ہے) جنریٹو AI نظاموں میں ایک سنگین کمزوری ہے، خاص طور پر ان میں جو Model Context Protocol (MCP) استعمال کرتے ہیں۔ اس حملے میں نقصان دہ ہدایات بیرونی مواد جیسے دستاویزات، ویب صفحات، یا ای میلز میں چھپی ہوتی ہیں۔ جب AI نظام اس مواد کو پروسیس کرتا ہے، تو وہ ان چھپی ہدایات کو جائز صارف کے کمانڈ کے طور پر سمجھ سکتا ہے، جس سے غیر ارادی کارروائیاں جیسے ڈیٹا لیک، نقصان دہ مواد کی تخلیق، یا صارف کے تعاملات کی چالاکی ہوتی ہے۔ تفصیلی وضاحت اور حقیقی مثالوں کے لیے دیکھیں [Prompt Injection](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/)۔

اس حملے کی خاص طور پر خطرناک شکل **Tool Poisoning** ہے۔ یہاں، حملہ آور MCP ٹولز کے میٹا ڈیٹا (جیسے ٹول کی وضاحت یا پیرامیٹرز) میں نقصان دہ ہدایات شامل کرتے ہیں۔ چونکہ بڑے زبان ماڈلز (LLMs) اس میٹا ڈیٹا پر انحصار کرتے ہیں کہ کون سے ٹولز کو کال کرنا ہے، اس لیے سمجھوتہ شدہ وضاحتیں ماڈل کو غیر مجاز ٹول کالز کرنے یا سیکیورٹی کنٹرولز کو بائی پاس کرنے پر مجبور کر سکتی ہیں۔ یہ چالاکیاں اکثر صارفین کو نظر نہیں آتیں لیکن AI نظام انہیں سمجھ کر عمل کرتا ہے۔ یہ خطرہ خاص طور پر ہوسٹڈ MCP سرور ماحول میں بڑھ جاتا ہے، جہاں صارف کی منظوری کے بعد ٹول کی تعریفیں اپ ڈیٹ کی جا سکتی ہیں — جسے بعض اوقات "[rug pull](https://www.wiz.io/blog/mcp-security-research-briefing#remote-servers-22)" کہا جاتا ہے۔ ایسے حالات میں، پہلے محفوظ ٹول بعد میں نقصان دہ کارروائیاں کرنے کے لیے تبدیل ہو سکتا ہے، جیسے ڈیٹا نکالنا یا نظامی رویے میں تبدیلی، بغیر صارف کی معلومات کے۔ اس حملے کے بارے میں مزید جاننے کے لیے دیکھیں [Tool Poisoning](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks)۔

## خطرات  
غیر ارادی AI کارروائیاں مختلف سیکیورٹی خطرات پیش کرتی ہیں جن میں ڈیٹا کا اخراج اور پرائیویسی کی خلاف ورزیاں شامل ہیں۔

### حفاظتی کنٹرولز  
### Indirect Prompt Injection حملوں کے خلاف Prompt Shields کا استعمال
-----------------------------------------------------------------------------

**AI Prompt Shields** مائیکروسافٹ کی جانب سے تیار کردہ حل ہیں جو براہ راست اور بالواسطہ prompt injection حملوں سے دفاع کرتے ہیں۔ یہ درج ذیل طریقوں سے مدد کرتے ہیں:

1.  **شناخت اور فلٹرنگ:** Prompt Shields جدید مشین لرننگ الگورتھمز اور قدرتی زبان کی پروسیسنگ استعمال کرتے ہیں تاکہ دستاویزات، ویب صفحات، یا ای میلز جیسے بیرونی مواد میں شامل نقصان دہ ہدایات کو شناخت اور فلٹر کیا جا سکے۔  
2.  **Spotlighting:** یہ تکنیک AI نظام کو جائز سسٹم ہدایات اور ممکنہ غیر معتبر بیرونی ان پٹس میں فرق کرنے میں مدد دیتی ہے۔ ان پٹ ٹیکسٹ کو اس طرح تبدیل کرکے جو ماڈل کے لیے زیادہ متعلقہ ہو، Spotlighting یقینی بناتی ہے کہ AI نقصان دہ ہدایات کو بہتر پہچان کر نظر انداز کر سکے۔  
3.  **Delimiters اور Datamarking:** سسٹم میسج میں delimiters شامل کرنا واضح طور پر ان پٹ ٹیکسٹ کی جگہ بتاتا ہے، جس سے AI نظام صارف کے ان پٹس کو ممکنہ نقصان دہ بیرونی مواد سے الگ کر سکتا ہے۔ Datamarking اس تصور کو بڑھاتے ہوئے خاص مارکرز کے ذریعے قابل اعتماد اور غیر قابل اعتماد ڈیٹا کی حد بندی کو نمایاں کرتا ہے۔  
4.  **مسلسل نگرانی اور اپ ڈیٹس:** Microsoft مسلسل Prompt Shields کی نگرانی اور اپ ڈیٹ کرتا ہے تاکہ نئے اور بدلتے ہوئے خطرات کا مقابلہ کیا جا سکے۔ یہ پیشگی حکمت عملی دفاع کو جدید حملہ تکنیکوں کے خلاف مؤثر رکھتی ہے۔  
5. **Azure Content Safety کے ساتھ انضمام:** Prompt Shields Azure AI Content Safety سوئٹ کا حصہ ہیں، جو jailbreak کی کوششوں، نقصان دہ مواد، اور AI ایپلیکیشنز میں دیگر سیکی
