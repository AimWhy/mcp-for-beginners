# सुरक्षा सर्वोत्तम अभ्यासहरू

Model Context Protocol (MCP) अपनाउँदा AI-चालित अनुप्रयोगहरूमा शक्तिशाली नयाँ क्षमताहरू प्राप्त हुन्छन्, तर यसले परम्परागत सफ्टवेयर जोखिमहरू भन्दा फरक विशिष्ट सुरक्षा चुनौतीहरू पनि ल्याउँछ। सुरक्षित कोडिङ, न्यूनतम अनुमति, र आपूर्ति श्रृंखला सुरक्षा जस्ता स्थापित चिन्ताहरूको अतिरिक्त, MCP र AI कार्यभारहरूलाई नयाँ खतराहरू जस्तै prompt injection, tool poisoning, र dynamic tool modification सामना गर्नुपर्छ। यी जोखिमहरूलाई सही तरिकाले व्यवस्थापन नगरिएमा डाटा चोरी, गोपनीयता उल्लङ्घन, र अनपेक्षित प्रणाली व्यवहार हुन सक्छ।

यो पाठले MCP सँग सम्बन्धित सबैभन्दा महत्वपूर्ण सुरक्षा जोखिमहरू—जस्तै प्रमाणीकरण, अधिक अनुमति, अप्रत्यक्ष prompt injection, र आपूर्ति श्रृंखला कमजोरीहरू—लाई अन्वेषण गर्छ र तिनीहरूलाई कम गर्नका लागि कार्यान्वयन योग्य नियन्त्रणहरू र सर्वोत्तम अभ्यासहरू प्रदान गर्छ। तपाईंले Microsoft का समाधानहरू जस्तै Prompt Shields, Azure Content Safety, र GitHub Advanced Security कसरी उपयोग गर्ने भनेर पनि सिक्नुहुनेछ जसले तपाईंको MCP कार्यान्वयनलाई बलियो बनाउँछ। यी नियन्त्रणहरू बुझेर र लागू गरेर, तपाईंले सुरक्षा उल्लङ्घनको सम्भावना उल्लेखनीय रूपमा घटाउन सक्नुहुन्छ र तपाईंका AI प्रणालीहरूलाई दिगो र विश्वासयोग्य बनाउन सक्नुहुन्छ।

# सिकाइ उद्देश्यहरू

यस पाठको अन्त्यसम्म, तपाईं सक्षम हुनुहुनेछ:

- Model Context Protocol (MCP) द्वारा ल्याइएका विशिष्ट सुरक्षा जोखिमहरू पहिचान र व्याख्या गर्न, जस्तै prompt injection, tool poisoning, अधिक अनुमति, र आपूर्ति श्रृंखला कमजोरीहरू।
- MCP सुरक्षा जोखिमहरूको लागि प्रभावकारी नियन्त्रणहरू वर्णन र लागू गर्न, जस्तै मजबुत प्रमाणीकरण, न्यूनतम अनुमति, सुरक्षित टोकन व्यवस्थापन, र आपूर्ति श्रृंखला प्रमाणीकरण।
- Microsoft का समाधानहरू जस्तै Prompt Shields, Azure Content Safety, र GitHub Advanced Security लाई MCP र AI कार्यभारहरूको सुरक्षा गर्न कसरी उपयोग गर्ने बुझ्न र प्रयोग गर्न।
- उपकरण मेटाडाटा प्रमाणीकरण, dynamic परिवर्तनहरूको अनुगमन, र अप्रत्यक्ष prompt injection आक्रमणहरू विरुद्ध सुरक्षा महत्त्वलाई चिन्हित गर्न।
- MCP कार्यान्वयनमा सुरक्षित कोडिङ, सर्भर कडाइ, र zero trust आर्किटेक्चर जस्ता स्थापित सुरक्षा सर्वोत्तम अभ्यासहरू समावेश गरेर सुरक्षा उल्लङ्घनको सम्भावना र प्रभाव घटाउन।

# MCP सुरक्षा नियन्त्रणहरू

महत्त्वपूर्ण स्रोतहरूमा पहुँच हुने कुनै पनि प्रणालीसँग सुरक्षा चुनौतीहरू स्वतः नै आउँछन्। सुरक्षा चुनौतीहरू सामान्यतया आधारभूत सुरक्षा नियन्त्रणहरू र अवधारणाहरूको सही प्रयोगबाट समाधान गर्न सकिन्छ। MCP भने नयाँ परिभाषित प्रोटोकल भएकाले यसको विनिर्देशन छिटो परिवर्तन हुँदैछ र प्रोटोकल विकास हुँदैछ। अन्ततः यसभित्रका सुरक्षा नियन्त्रणहरू परिपक्व हुनेछन् जसले उद्यम र स्थापित सुरक्षा वास्तुकला तथा सर्वोत्तम अभ्यासहरूसँग राम्रो एकीकरण सक्षम पार्नेछ।

[Microsoft Digital Defense Report](https://aka.ms/mddr) मा प्रकाशित अनुसन्धानले देखाएको छ कि रिपोर्ट गरिएका ९८% उल्लङ्घनहरू मजबुत सुरक्षा स्वच्छता द्वारा रोक्न सकिन्छ र कुनै पनि प्रकारको उल्लङ्घन विरुद्ध सबैभन्दा राम्रो सुरक्षा भनेको आधारभूत सुरक्षा स्वच्छता, सुरक्षित कोडिङ सर्वोत्तम अभ्यासहरू र आपूर्ति श्रृंखला सुरक्षा सही तरिकाले लागू गर्नु हो—यी परखिएका अभ्यासहरूले अझै पनि सुरक्षा जोखिम घटाउन सबैभन्दा ठूलो प्रभाव पार्छ।

MCP अपनाउँदा तपाईंले सुरक्षा जोखिमहरू कसरी सम्बोधन गर्न थाल्न सक्नुहुन्छ भन्ने केही तरिकाहरू हेरौं।

# MCP सर्भर प्रमाणीकरण (यदि तपाईंको MCP कार्यान्वयन २६ अप्रिल २०२५ भन्दा पहिलेको हो भने)

> **Note:** तलको जानकारी २६ अप्रिल २०२५ सम्म सही छ। MCP प्रोटोकल निरन्तर विकास भइरहेको छ, र भविष्यका कार्यान्वयनहरूले नयाँ प्रमाणीकरण ढाँचाहरू र नियन्त्रणहरू ल्याउन सक्छन्। सबैभन्दा नयाँ अपडेट र मार्गदर्शनका लागि सधैं [MCP Specification](https://spec.modelcontextprotocol.io/) र आधिकारिक [MCP GitHub repository](https://github.com/modelcontextprotocol) हेर्नुहोस्।

### समस्या विवरण  
मूल MCP विनिर्देशनले विकासकर्ताहरूले आफ्नै प्रमाणीकरण सर्भर लेख्ने अपेक्षा गरेको थियो। यसका लागि OAuth र सम्बन्धित सुरक्षा प्रतिबन्धहरूको ज्ञान आवश्यक थियो। MCP सर्भरहरूले OAuth 2.0 Authorization Servers को रूपमा काम गर्थे, आवश्यक प्रयोगकर्ता प्रमाणीकरणलाई सिधै व्यवस्थापन गर्थे, बाह्य सेवा जस्तै Microsoft Entra ID लाई प्रतिनिधित्व नगरी। २६ अप्रिल २०२५ सम्मको अपडेटले MCP सर्भरहरूलाई प्रयोगकर्ता प्रमाणीकरण बाह्य सेवामा सुम्पन अनुमति दिन्छ।

### जोखिमहरू
- MCP सर्भरमा गलत कन्फिगर गरिएको प्राधिकरण तर्कले संवेदनशील डाटा खुलाउन र पहुँच नियन्त्रणहरू गलत लागू हुन सक्छ।
- स्थानीय MCP सर्भरमा OAuth टोकन चोरी। चोरी भएमा, टोकनले MCP सर्भरको नक्कल गरेर सेवा र डाटामा पहुँच दिन सक्छ।

### नियन्त्रणहरू
- **प्राधिकरण तर्कको समीक्षा र कडाइ गर्नुहोस्:** तपाईंको MCP सर्भरको प्राधिकरण कार्यान्वयनलाई ध्यानपूर्वक अडिट गर्नुहोस् ताकि मात्र इच्छित प्रयोगकर्ता र क्लाइन्टहरूले संवेदनशील स्रोतहरूमा पहुँच पाउन सकून्। व्यावहारिक मार्गदर्शनका लागि [Azure API Management Your Auth Gateway For MCP Servers | Microsoft Community Hub](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690) र [Using Microsoft Entra ID To Authenticate With MCP Servers Via Sessions - Den Delimarsky](https://den.dev/blog/mcp-server-auth-entra-id-session/) हेर्नुहोस्।
- **सुरक्षित टोकन अभ्यासहरू लागू गर्नुहोस्:** पहुँच टोकनहरूको दुरुपयोग रोक्न र टोकन पुनःप्रयोग वा चोरी जोखिम घटाउन [Microsoft को टोकन मान्यकरण र अवधि सम्बन्धी सर्वोत्तम अभ्यासहरू](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens) पालना गर्नुहोस्।
- **टोकन भण्डारण सुरक्षित गर्नुहोस्:** टोकनहरू सँधै सुरक्षित रूपमा भण्डारण गर्नुहोस् र विश्राम र ट्रान्जिटमा तिनीहरूलाई इन्क्रिप्ट गर्नुहोस्। कार्यान्वयन सुझावका लागि [Use secure token storage and encrypt tokens](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2) हेर्नुहोस्।

# MCP सर्भरहरूको लागि अत्यधिक अनुमति

### समस्या विवरण  
MCP सर्भरहरूलाई उनीहरूले पहुँच गर्ने सेवा/स्रोतमा अत्यधिक अनुमति दिइएको हुन सक्छ। उदाहरणका लागि, AI बिक्री अनुप्रयोगको भाग हुने MCP सर्भरले उद्यम डाटा स्टोरमा पहुँच गर्दा केवल बिक्री डाटामा सीमित पहुँच हुनु पर्छ, स्टोरका सबै फाइलहरूमा होइन। न्यूनतम अनुमति सिद्धान्त (सबैभन्दा पुरानो सुरक्षा सिद्धान्तहरूमध्ये एक) लाई सम्झँदै, कुनै स्रोतलाई त्यसले गर्नुपर्ने कार्यभन्दा बढी अनुमति हुनु हुँदैन। AI को लचिलोपनका लागि आवश्यक अनुमति ठ्याक्कै निर्धारण गर्न चुनौतीपूर्ण हुन्छ।

### जोखिमहरू  
- अत्यधिक अनुमति दिँदा MCP सर्भरले पहुँच नपाउने डाटा चोरी वा संशोधन गर्न सक्छ। यदि डाटा व्यक्तिगत पहिचान योग्य जानकारी (PII) हो भने यो गोपनीयताको समस्या पनि हुन सक्छ।

### नियन्त्रणहरू  
- **न्यूनतम अनुमति सिद्धान्त लागू गर्नुहोस्:** MCP सर्भरलाई आवश्यक कार्यहरू गर्न न्यूनतम अनुमति मात्र दिनुहोस्। यी अनुमतिहरू नियमित रूपमा समीक्षा र अद्यावधिक गर्नुहोस् ताकि आवश्यकभन्दा बढी नहोस्। विस्तृत मार्गदर्शनका लागि [Secure least-privileged access](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access) हेर्नुहोस्।
- **Role-Based Access Control (RBAC) प्रयोग गर्नुहोस्:** MCP सर्भरलाई विशिष्ट स्रोत र कार्यहरूमा सीमित भूमिकाहरू असाइन गर्नुहोस्, व्यापक वा अनावश्यक अनुमतिहरूबाट बच्नुहोस्।
- **अनुमति अनुगमन र अडिट गर्नुहोस्:** अनुमति प्रयोग निरन्तर अनुगमन गर्नुहोस् र पहुँच लगहरू अडिट गरेर अत्यधिक वा प्रयोग नभएका अनुमतिहरू छिटो पत्ता लगाएर सुधार गर्नुहोस्।

# अप्रत्यक्ष prompt injection आक्रमणहरू

### समस्या विवरण

दुष्ट वा सम्झौता गरिएका MCP सर्भरहरूले ग्राहक डाटा खुलाउन वा अनपेक्षित क्रियाकलापहरू गर्न महत्वपूर्ण जोखिमहरू ल्याउन सक्छन्। यी जोखिमहरू AI र MCP-आधारित कार्यभारहरूमा विशेष गरी सान्दर्भिक छन्, जहाँ:

- **Prompt Injection आक्रमणहरू**: आक्रमणकारीहरूले prompts वा बाह्य सामग्रीमा दुष्ट निर्देशनहरू समावेश गर्छन्, जसले AI प्रणालीलाई अनपेक्षित कार्यहरू गर्न वा संवेदनशील डाटा चुहाउन प्रेरित गर्छ। थप जान्न: [Prompt Injection](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/)
- **Tool Poisoning**: आक्रमणकारीहरूले उपकरण मेटाडाटा (जस्तै विवरण वा प्यारामिटरहरू) मा हेरफेर गर्छन् जसले AI को व्यवहारमा प्रभाव पार्छ, सम्भावित रूपमा सुरक्षा नियन्त्रणहरू बाइपास गर्ने वा डाटा चुहाउने। विवरण: [Tool Poisoning](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks)
- **Cross-Domain Prompt Injection**: दुष्ट निर्देशनहरू कागजात, वेब पृष्ठ, वा इमेलमा समावेश गरिन्छ, जुन AI द्वारा प्रशोधित हुँदा डाटा चुहावट वा हेरफेर हुन्छ।
- **Dynamic Tool Modification (Rug Pulls)**: प्रयोगकर्ताको स्वीकृतिपछि उपकरण परिभाषाहरू परिवर्तन गर्न सकिन्छ, जसले नयाँ दुष्ट व्यवहारहरू ल्याउन सक्छ।

यी कमजोरीहरूले MCP सर्भर र उपकरणहरूलाई तपाईंको वातावरणमा समावेश गर्दा कडाइका साथ प्रमाणीकरण, अनुगमन, र सुरक्षा नियन्त्रण आवश्यक देखाउँछन्। थप विवरणका लागि माथि उल्लेखित लिंकहरू हेर्नुहोस्।

**अप्रत्यक्ष Prompt Injection** (जसलाई cross-domain prompt injection वा XPIA पनि भनिन्छ) उत्पादनात्मक AI प्रणालीहरूमा एक गम्भीर कमजोरी हो, जसमा Model Context Protocol (MCP) पनि समावेश छ। यस आक्रमणमा, दुष्ट निर्देशनहरू बाह्य सामग्रीमा लुकाइन्छ—जस्तै कागजात, वेब पृष्ठ, वा इमेल। जब AI प्रणालीले यो सामग्री प्रशोधित गर्छ, तब यसले लुकेको निर्देशनहरूलाई वैध प्रयोगकर्ता आदेशको रूपमा व्याख्या गर्न सक्छ, जसले डाटा चुहावट, हानिकारक सामग्री सिर्जना, वा प्रयोगकर्ता अन्तरक्रियाहरूमा हेरफेर जस्ता अनपेक्षित कार्यहरू निम्त्याउँछ। विस्तृत व्याख्या र वास्तविक उदाहरणहरूका लागि [Prompt Injection](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/) हेर्नुहोस्।

यस आक्रमणको विशेष खतरनाक रूप हो **Tool Poisoning**। यहाँ, आक्रमणकारीहरूले MCP उपकरणहरूको मेटाडाटामा दुष्ट निर्देशनहरू इंजेक्ट गर्छन् (जस्तै उपकरण विवरण वा प्यारामिटरहरू)। ठूलो भाषा मोडेलहरू (LLMs) यी मेटाडाटामा आधारित भएर कुन उपकरण कल गर्ने निर्णय गर्छन्, त्यसैले सम्झौता गरिएको विवरणहरूले मोडेललाई अनधिकृत उपकरण कलहरू गर्न वा सुरक्षा नियन्त्रणहरू बाइपास गर्न प्रलोभन दिन सक्छ। यी हेरफेरहरू प्रायः अन्त प्रयोगकर्ताहरूलाई देखिँदैनन् तर AI प्रणालीले व्याख्या गरी कार्यान्वयन गर्न सक्छ। यो जोखिम होस्ट गरिएको MCP सर्भर वातावरणहरूमा अझ बढी हुन्छ, जहाँ उपकरण परिभाषाहरू प्रयोगकर्ताको स्वीकृतिपछि परिवर्तन गर्न सकिन्छ—जसलाई कहिलेकाहीं "[rug pull](https://www.wiz.io/blog/mcp-security-research-briefing#remote-servers-22)" भनिन्छ। यस्तो अवस्थामा, पहिले सुरक्षित रहेको उपकरण पछि दुष्ट कार्यहरू गर्न परिवर्तन हुन सक्छ, जस्तै डाटा चोरी वा प्रणाली व्यवहार परिवर्तन, प्रयोगकर्ताको जानकारी बिना। यस आक्रमण बिन्दुका लागि थप जानकारी [Tool Poisoning](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks) मा छ।

## जोखिमहरू  
अनपेक्षित AI क्रियाकलापहरूले डाटा चुहावट र गोपनीयता उल्लङ्घन जस्ता विभिन्न सुरक्षा जोखिमहरू उत्पन्न गर्छन्।

### नियन्त्रणहरू  
### अप्रत्यक्ष Prompt Injection आक्रमणहरू विरुद्ध prompt shields को प्रयोग
-----------------------------------------------------------------------------

**AI Prompt Shields** Microsoft द्वारा विकास गरिएको समाधान हो जसले प्रत्यक्ष र अप्रत्यक्ष prompt injection आक्रमणहरूबाट सुरक्षा गर्दछ। यसले निम्न तरिकाले मद्दत गर्छ:

1.  **पत्ता लगाउने र फिल्टर गर्ने:** Prompt Shields ले उन्नत मेसिन लर्निङ एल्गोरिदम र प्राकृतिक भाषा प्रशोधन प्रयोग गरी दुष्ट निर्देशनहरू समावेश गरिएको बाह्य सामग्री जस्तै कागजात, वेब पृष्ठ, वा इमेलहरूलाई पत्ता लगाउँछ र फिल्टर गर्छ।
    
2.  **Spotlighting:** यस प्रविधिले AI प्रणालीलाई वैध प्रणाली निर्देशन र सम्भावित अविश्वसनीय बाह्य इनपुटबीच छुट्याउन मद्दत गर्छ। इनपुट टेक्स्टलाई मोडेलका लागि बढी सान्दर्भिक बनाउन रूपान्तरण गरेर Spotlighting ले AI लाई दुष्ट निर्देशनहरू पहिचान गरी बेवास्ता गर्न सक्षम बनाउँछ।
    
3.  **Delimiters र Datamarking:** प्रणाली सन्देशमा delimiters समावेश गर्दा इनपुट टेक्स्टको स्थान स्पष्ट रूपमा देखिन्छ, जसले AI प्रणालीलाई प्रयोगकर्ता इनपुट र सम्भावित हानिकारक बाह्य सामग्री छुट्याउन मद्दत गर्छ। Datamarking ले विशेष मार्करहरू प्रयोग गरेर विश्वासिलो र अविश्वसनीय डाटाको सीमाना हाइलाइट गर्छ।
    
4.  **निरन्तर अनुगमन र अद्यावधिक:** Microsoft ले नयाँ र विकसित हुने खतरा समाधान गर्न Prompt Shields लाई निरन्तर अनुगमन र अद्यावधिक गर्दछ। यस सक्रिय दृष्टिकोणले नवीनतम आक्रमण प्रविधिहरू विरुद्ध सुरक्षा प्रभावकारी रहन्छ।
    
5. **Azure Content Safety सँग एकीकरण:** Prompt Shields Azure AI Content Safety सूटको भाग हुन्, जसले AI अनुप्रयोगहरूमा jailbreak प्रयास, हानिकारक सामग्री, र अन्य सुरक्षा जोखिमहरू पत्ता लगाउन थप उपकरणहरू प्रदान गर्छ।

AI prompt shields सम्बन्धी थप जानकारीका लागि [Prompt Shields documentation](https://learn.microsoft.com/azure/ai-services/content-safety/concepts/jailbreak-detection) हेर्नुहोस्।

### आपूर्ति श्रृंखला सुरक्षा

AI युगमा आपूर्ति श्रृंखला सुरक्षा अझै पनि आधारभूत छ, तर तपाईंको आपूर्ति श्रृंखलाको परिधि विस्तार भएको छ। परम्परागत कोड प्याकेजहरूका साथै अब तपाईंले सबै AI-सँग सम्बन्धित कम्पोनेन्टहरू, जस्तै foundation models, embeddings सेवा, context providers, र तेस्रो पक्ष APIs कडाइका साथ प्रमाणीकरण र अनुगमन गर्नुपर्छ। यी प्रत्येकले उचित व्यवस्थापन नगरिएमा कमजोरी वा जोखिम ल्याउन सक्छ।

**AI र MCP का लागि प्रमुख आपूर्ति श्रृंखला सुरक्षा अभ्यासहरू:**  
- **एकीकरण अघि सबै कम्पोनेन्टहरू प्रमाणीकरण गर्नुहोस्:** यसमा खुला स्रोत पुस्तकालयहरू मात्र होइन, AI मोडेलहरू, डाटा स्रोतहरू, र बाह्य API हरू पनि समावेश छन्। सधैं मूल स्रोत, लाइसेन्स, र ज्ञात कमजोरीहरू जाँच्नुहोस्।  
- **सुरक्षित परिनियोजन पाइपलाइनहरू कायम राख्नुहोस्:** सुरक्षा स्क्यानिङ सहित स्वचालित CI/CD पाइपलाइनहरू प्रयोग गरेर समस्याहरू छिटो पत्ता लगाउनुहोस्। केवल विश्वासिलो आर्टिफ्याक्टहरू उत्पादनमा पठाउनुहोस्।  
- **निरन्तर अनुगमन र अडिट गर्नुहोस्:** सबै निर्भरताहरू, मोडेलहरू र डाटा सेवाहरू सहित, नयाँ कमजोरीहरू वा आपूर्ति श्रृंखला आक्रमणहरू पत्ता लगाउन निरन्तर अनुगमन लागू गर्नुहोस्।  
- **न्यूनतम अनुमति र पहुँच नियन्त्रण लागू गर्नुहोस्:** मोडेल, डाटा, र सेवाहरूमा MCP सर्भरलाई आवश्यक मात्र पहुँच दिनुहोस्।  
- **खतरा देखिँदा छिटो प्रतिक्रिया दिनुहोस्:** सम्झौता भएका कम्पोनेन्टहरू मर्मत वा प्रतिस्थापन गर्ने प्रक्रिया राख्नुहोस्, र उल्लङ्घन पत्ता लागेमा गोप्य जानकारी वा प्रमाणपत्रहरू परिवर्तन गर्नुहोस्।

[GitHub Advanced Security](https://github.com/security/advanced-security) ले secret scanning, dependency scanning, र CodeQL विश्लेषण जस्ता सुविधाहरू प्रदान गर्छ। यी उपकरणहरूले [Azure DevOps](https://azure.microsoft.com/en-us/products/devops) र [Azure Repos](https://azure.microsoft.com/en-us/products/devops/repos/) सँग एकीकृत भएर कोड र AI आपूर्ति श्रृंखला कम्पोनेन्टहरूमा कमजोरीहरू पहिचान र कम गर्न मद्दत गर्छन्।

Microsoft ले आफ्ना सबै उत्पादनहरूको लागि व्यापक आपूर्ति श्रृंखला सुरक्षा अभ्यासहरू आन्तरिक रूपमा लागू गरेको छ। थप जान्न [The Journey to Secure the Software Supply Chain at Microsoft](https://devblogs.microsoft.com/engineering-at-microsoft/the-journey-to-secure-the-software-supply-chain-at-microsoft/) पढ्नुहोस्।
