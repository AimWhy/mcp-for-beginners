# ਸੁਰੱਖਿਆ ਲਈ ਵਧੀਆ ਅਮਲ

Model Context Protocol (MCP) ਨੂੰ ਅਪਣਾਉਣ ਨਾਲ AI-ਚਲਿਤ ਐਪਲੀਕੇਸ਼ਨਾਂ ਵਿੱਚ ਸ਼ਕਤੀਸ਼ਾਲੀ ਨਵੀਆਂ ਸਮਰੱਥਾਵਾਂ ਆਉਂਦੀਆਂ ਹਨ, ਪਰ ਇਹ ਪਰੰਪਰਾਗਤ ਸਾਫਟਵੇਅਰ ਖ਼ਤਰਿਆਂ ਤੋਂ ਵੱਖਰੇ ਵਿਲੱਖਣ ਸੁਰੱਖਿਆ ਚੁਣੌਤੀਆਂ ਵੀ ਲਿਆਉਂਦਾ ਹੈ। ਸੁਰੱਖਿਅਤ ਕੋਡਿੰਗ, ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ ਅਤੇ ਸਪਲਾਈ ਚੇਨ ਸੁਰੱਖਿਆ ਵਰਗੀਆਂ ਮੌਜੂਦਾ ਚਿੰਤਾਵਾਂ ਦੇ ਨਾਲ-ਨਾਲ, MCP ਅਤੇ AI ਵਰਕਲੋਡਾਂ ਨੂੰ ਨਵੇਂ ਖ਼ਤਰੇ ਜਿਵੇਂ ਕਿ ਪ੍ਰਾਂਪਟ ਇੰਜੈਕਸ਼ਨ, ਟੂਲ ਜਹਿਰਲਾਪਣ, ਗਤੀਸ਼ੀਲ ਟੂਲ ਸੋਧ, ਸੈਸ਼ਨ ਹਾਈਜੈਕਿੰਗ, ਕਨਫਿਊਜ਼ਡ ਡਿਪਟੀ ਹਮਲੇ ਅਤੇ ਟੋਕਨ ਪਾਸਥਰੂ ਕਮਜ਼ੋਰੀਆਂ ਦਾ ਸਾਹਮਣਾ ਕਰਨਾ ਪੈਂਦਾ ਹੈ। ਜੇ ਇਹਨਾਂ ਖ਼ਤਰਿਆਂ ਨੂੰ ਠੀਕ ਤਰ੍ਹਾਂ ਸੰਭਾਲਿਆ ਨਾ ਜਾਵੇ ਤਾਂ ਇਹ ਡਾਟਾ ਚੋਰੀ, ਪ੍ਰਾਈਵੇਸੀ ਉਲੰਘਣਾ ਅਤੇ ਅਣਚਾਹੇ ਸਿਸਟਮ ਵਿਹਾਰ ਦਾ ਕਾਰਨ ਬਣ ਸਕਦੇ ਹਨ।

ਇਸ ਪਾਠ ਵਿੱਚ MCP ਨਾਲ ਜੁੜੇ ਸਭ ਤੋਂ ਮਹੱਤਵਪੂਰਨ ਸੁਰੱਖਿਆ ਖ਼ਤਰਿਆਂ ਦੀ ਚਰਚਾ ਕੀਤੀ ਗਈ ਹੈ—ਜਿਵੇਂ ਕਿ ਪ੍ਰਮਾਣਿਕਤਾ, ਅਧਿਕਾਰ, ਜ਼ਿਆਦਾ ਅਧਿਕਾਰ, ਅਪਰੋਕਸ਼ ਪ੍ਰਾਂਪਟ ਇੰਜੈਕਸ਼ਨ, ਸੈਸ਼ਨ ਸੁਰੱਖਿਆ, ਕਨਫਿਊਜ਼ਡ ਡਿਪਟੀ ਸਮੱਸਿਆਵਾਂ, ਟੋਕਨ ਪਾਸਥਰੂ ਕਮਜ਼ੋਰੀਆਂ ਅਤੇ ਸਪਲਾਈ ਚੇਨ ਕਮਜ਼ੋਰੀਆਂ—ਅਤੇ ਇਨ੍ਹਾਂ ਨੂੰ ਘਟਾਉਣ ਲਈ ਕਾਰਗਰ ਨਿਯੰਤਰਣ ਅਤੇ ਵਧੀਆ ਅਮਲ ਦਿੱਤੇ ਗਏ ਹਨ। ਤੁਸੀਂ Microsoft ਦੇ ਹੱਲਾਂ ਜਿਵੇਂ ਕਿ Prompt Shields, Azure Content Safety ਅਤੇ GitHub Advanced Security ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਆਪਣੇ MCP ਇੰਪਲੀਮੈਂਟੇਸ਼ਨ ਨੂੰ ਮਜ਼ਬੂਤ ਬਣਾਉਣ ਬਾਰੇ ਵੀ ਸਿੱਖੋਗੇ। ਇਹ ਨਿਯੰਤਰਣ ਸਮਝ ਕੇ ਅਤੇ ਲਾਗੂ ਕਰਕੇ, ਤੁਸੀਂ ਸੁਰੱਖਿਆ ਭੰਗ ਹੋਣ ਦੀ ਸੰਭਾਵਨਾ ਨੂੰ ਕਾਫੀ ਘਟਾ ਸਕਦੇ ਹੋ ਅਤੇ ਆਪਣੇ AI ਸਿਸਟਮਾਂ ਨੂੰ ਮਜ਼ਬੂਤ ਅਤੇ ਭਰੋਸੇਯੋਗ ਬਣਾ ਸਕਦੇ ਹੋ।

# ਸਿੱਖਣ ਦੇ ਲਕੜੇ

ਇਸ ਪਾਠ ਦੇ ਅੰਤ ਤੱਕ, ਤੁਸੀਂ ਸਮਰੱਥ ਹੋਵੋਗੇ:

- Model Context Protocol (MCP) ਵੱਲੋਂ ਲਿਆਂਦੇ ਗਏ ਵਿਲੱਖਣ ਸੁਰੱਖਿਆ ਖ਼ਤਰਿਆਂ ਦੀ ਪਛਾਣ ਅਤੇ ਵਿਆਖਿਆ ਕਰਨਾ, ਜਿਵੇਂ ਕਿ ਪ੍ਰਾਂਪਟ ਇੰਜੈਕਸ਼ਨ, ਟੂਲ ਜਹਿਰਲਾਪਣ, ਜ਼ਿਆਦਾ ਅਧਿਕਾਰ, ਸੈਸ਼ਨ ਹਾਈਜੈਕਿੰਗ, ਕਨਫਿਊਜ਼ਡ ਡਿਪਟੀ ਸਮੱਸਿਆਵਾਂ, ਟੋਕਨ ਪਾਸਥਰੂ ਕਮਜ਼ੋਰੀਆਂ ਅਤੇ ਸਪਲਾਈ ਚੇਨ ਕਮਜ਼ੋਰੀਆਂ।
- MCP ਸੁਰੱਖਿਆ ਖ਼ਤਰਿਆਂ ਲਈ ਪ੍ਰਭਾਵਸ਼ਾਲੀ ਨਿਯੰਤਰਣ ਲਾਗੂ ਕਰਨਾ, ਜਿਵੇਂ ਕਿ ਮਜ਼ਬੂਤ ਪ੍ਰਮਾਣਿਕਤਾ, ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ, ਸੁਰੱਖਿਅਤ ਟੋਕਨ ਪ੍ਰਬੰਧਨ, ਸੈਸ਼ਨ ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਅਤੇ ਸਪਲਾਈ ਚੇਨ ਦੀ ਜਾਂਚ।
- Microsoft ਦੇ ਹੱਲਾਂ ਜਿਵੇਂ ਕਿ Prompt Shields, Azure Content Safety ਅਤੇ GitHub Advanced Security ਦੀ ਵਰਤੋਂ ਕਰਕੇ MCP ਅਤੇ AI ਵਰਕਲੋਡਾਂ ਦੀ ਸੁਰੱਖਿਆ ਕਰਨਾ।
- ਟੂਲ ਮੈਟਾਡੇਟਾ ਦੀ ਸਹੀ ਜਾਂਚ ਕਰਨ, ਗਤੀਸ਼ੀਲ ਬਦਲਾਵਾਂ ਦੀ ਨਿਗਰਾਨੀ ਕਰਨ, ਅਪਰੋਕਸ਼ ਪ੍ਰਾਂਪਟ ਇੰਜੈਕਸ਼ਨ ਹਮਲਿਆਂ ਤੋਂ ਬਚਾਅ ਕਰਨ ਅਤੇ ਸੈਸ਼ਨ ਹਾਈਜੈਕਿੰਗ ਨੂੰ ਰੋਕਣ ਦੀ ਮਹੱਤਤਾ ਨੂੰ ਸਮਝਣਾ।
- ਸੁਰੱਖਿਅਤ ਕੋਡਿੰਗ, ਸਰਵਰ ਹਾਰਡਨਿੰਗ ਅਤੇ ਜ਼ੀਰੋ ਟਰੱਸਟ ਆਰਕੀਟੈਕਚਰ ਵਰਗੇ ਸਥਾਪਿਤ ਸੁਰੱਖਿਆ ਵਧੀਆ ਅਮਲਾਂ ਨੂੰ ਆਪਣੇ MCP ਇੰਪਲੀਮੈਂਟੇਸ਼ਨ ਵਿੱਚ ਸ਼ਾਮਲ ਕਰਕੇ ਸੁਰੱਖਿਆ ਭੰਗ ਹੋਣ ਦੀ ਸੰਭਾਵਨਾ ਅਤੇ ਪ੍ਰਭਾਵ ਨੂੰ ਘਟਾਉਣਾ।

# MCP ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ

ਜੋ ਕੋਈ ਵੀ ਸਿਸਟਮ ਮਹੱਤਵਪੂਰਨ ਸਰੋਤਾਂ ਤੱਕ ਪਹੁੰਚ ਰੱਖਦਾ ਹੈ, ਉਸਦੇ ਸਾਹਮਣੇ ਸੁਰੱਖਿਆ ਚੁਣੌਤੀਆਂ ਹੁੰਦੀਆਂ ਹਨ। ਇਹ ਚੁਣੌਤੀਆਂ ਆਮ ਤੌਰ 'ਤੇ ਮੂਲ ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਅਤੇ ਧਾਰਣਾਵਾਂ ਦੇ ਸਹੀ ਲਾਗੂ ਕਰਨ ਨਾਲ ਹੱਲ ਕੀਤੀਆਂ ਜਾ ਸਕਦੀਆਂ ਹਨ। MCP ਨਵਾਂ ਪਰਿਭਾਸ਼ਿਤ ਪ੍ਰੋਟੋਕੋਲ ਹੈ, ਇਸ ਲਈ ਇਸ ਦੀ ਵਿਸ਼ੇਸ਼ਤਾ ਤੇਜ਼ੀ ਨਾਲ ਬਦਲ ਰਹੀ ਹੈ ਅਤੇ ਜਿਵੇਂ ਜਿਵੇਂ ਪ੍ਰੋਟੋਕੋਲ ਵਿਕਸਤ ਹੋਵੇਗਾ, ਇਸਦੇ ਅੰਦਰਲੇ ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਵੀ ਪੱਕੇ ਹੋਣਗੇ, ਜੋ ਉਦਯੋਗ ਅਤੇ ਸਥਾਪਿਤ ਸੁਰੱਖਿਆ ਆਰਕੀਟੈਕਚਰਾਂ ਅਤੇ ਵਧੀਆ ਅਮਲਾਂ ਨਾਲ ਬਿਹਤਰ ਇੰਟਿਗ੍ਰੇਸ਼ਨ ਯੋਗ ਬਣਾਉਣਗੇ।

[Microsoft Digital Defense Report](https://aka.ms/mddr) ਵਿੱਚ ਪ੍ਰਕਾਸ਼ਿਤ ਖੋਜ ਦੱਸਦੀ ਹੈ ਕਿ 98% ਰਿਪੋਰਟ ਕੀਤੀਆਂ ਸੁਰੱਖਿਆ ਭੰਗਾਂ ਨੂੰ ਮਜ਼ਬੂਤ ਸੁਰੱਖਿਆ ਸਫਾਈ ਨਾਲ ਰੋਕਿਆ ਜਾ ਸਕਦਾ ਹੈ ਅਤੇ ਕਿਸੇ ਵੀ ਕਿਸਮ ਦੀ ਭੰਗ ਤੋਂ ਬਚਾਅ ਲਈ ਸਭ ਤੋਂ ਵਧੀਆ ਸੁਰੱਖਿਆ ਹੈ ਕਿ ਤੁਸੀਂ ਆਪਣੀ ਬੇਸਲਾਈਨ ਸੁਰੱਖਿਆ ਸਫਾਈ, ਸੁਰੱਖਿਅਤ ਕੋਡਿੰਗ ਵਧੀਆ ਅਮਲ ਅਤੇ ਸਪਲਾਈ ਚੇਨ ਸੁਰੱਖਿਆ ਨੂੰ ਠੀਕ ਢੰਗ ਨਾਲ ਲਾਗੂ ਕਰੋ—ਇਹ ਉਹ ਅਜ਼ਮਾਏ ਹੋਏ ਅਮਲ ਹਨ ਜੋ ਸੁਰੱਖਿਆ ਖ਼ਤਰੇ ਨੂੰ ਘਟਾਉਣ ਵਿੱਚ ਸਭ ਤੋਂ ਵੱਧ ਪ੍ਰਭਾਵਸ਼ਾਲੀ ਹਨ।

ਆਓ ਕੁਝ ਤਰੀਕੇ ਵੇਖੀਏ ਜਿਨ੍ਹਾਂ ਨਾਲ ਤੁਸੀਂ MCP ਅਪਣਾਉਂਦੇ ਸਮੇਂ ਸੁਰੱਖਿਆ ਖ਼ਤਰਿਆਂ ਨੂੰ ਸੰਭਾਲਣਾ ਸ਼ੁਰੂ ਕਰ ਸਕਦੇ ਹੋ।

> **Note:** ਹੇਠਾਂ ਦਿੱਤੀ ਜਾਣਕਾਰੀ **29 ਮਈ 2025** ਤੱਕ ਸਹੀ ਹੈ। MCP ਪ੍ਰੋਟੋਕੋਲ ਲਗਾਤਾਰ ਵਿਕਸਤ ਹੋ ਰਿਹਾ ਹੈ, ਅਤੇ ਭਵਿੱਖ ਵਿੱਚ ਨਵੇਂ ਪ੍ਰਮਾਣਿਕਤਾ ਪੈਟਰਨ ਅਤੇ ਨਿਯੰਤਰਣ ਆ ਸਕਦੇ ਹਨ। ਤਾਜ਼ਾ ਜਾਣਕਾਰੀ ਅਤੇ ਮਾਰਗਦਰਸ਼ਨ ਲਈ ਹਮੇਸ਼ਾ [MCP Specification](https://spec.modelcontextprotocol.io/) ਅਤੇ ਅਧਿਕਾਰਿਕ [MCP GitHub repository](https://github.com/modelcontextprotocol) ਅਤੇ [security best practice page](https://modelcontextprotocol.io/specification/draft/basic/security_best_practices) ਨੂੰ ਵੇਖੋ।

### ਸਮੱਸਿਆ ਦਾ ਬਿਆਨ  
ਮੂਲ MCP ਵਿਸ਼ੇਸ਼ਤਾ ਇਹ ਮੰਨਦੀ ਸੀ ਕਿ ਡਿਵੈਲਪਰ ਆਪਣੇ ਪ੍ਰਮਾਣਿਕਤਾ ਸਰਵਰ ਨੂੰ ਖੁਦ ਲਿਖਣਗੇ। ਇਸ ਲਈ OAuth ਅਤੇ ਸੰਬੰਧਿਤ ਸੁਰੱਖਿਆ ਸੀਮਾਵਾਂ ਦੀ ਜਾਣਕਾਰੀ ਲੋੜੀਂਦੀ ਸੀ। MCP ਸਰਵਰ OAuth 2.0 Authorization Servers ਵਜੋਂ ਕੰਮ ਕਰਦੇ ਸਨ, ਜੋ ਲੋੜੀਂਦੇ ਯੂਜ਼ਰ ਪ੍ਰਮਾਣਿਕਤਾ ਨੂੰ ਸਿੱਧਾ ਸੰਭਾਲਦੇ ਸਨ, ਬਜਾਏ ਇਸਦੇ ਕਿ ਇਸ ਨੂੰ ਕਿਸੇ ਬਾਹਰੀ ਸੇਵਾ ਜਿਵੇਂ Microsoft Entra ID ਨੂੰ ਸੌਂਪਿਆ ਜਾਵੇ। **26 ਅਪ੍ਰੈਲ 2025** ਤੋਂ MCP ਵਿਸ਼ੇਸ਼ਤਾ ਵਿੱਚ ਇੱਕ ਅੱਪਡੇਟ ਆਇਆ ਹੈ ਜੋ MCP ਸਰਵਰਾਂ ਨੂੰ ਯੂਜ਼ਰ ਪ੍ਰਮਾਣਿਕਤਾ ਬਾਹਰੀ ਸੇਵਾ ਨੂੰ ਸੌਂਪਣ ਦੀ ਆਗਿਆ ਦਿੰਦਾ ਹੈ।

### ਖ਼ਤਰੇ
- MCP ਸਰਵਰ ਵਿੱਚ ਗਲਤ ਤਰੀਕੇ ਨਾਲ ਅਧਿਕਾਰ ਲਾਜਿਕ ਲਾਗੂ ਹੋਣ ਨਾਲ ਸੰਵੇਦਨਸ਼ੀਲ ਡਾਟਾ ਦਾ ਪ੍ਰਕਾਸ਼ ਹੋ ਸਕਦਾ ਹੈ ਅਤੇ ਗਲਤ ਤਰੀਕੇ ਨਾਲ ਪਹੁੰਚ ਨਿਯੰਤਰਣ ਲਾਗੂ ਹੋ ਸਕਦੇ ਹਨ।
- ਸਥਾਨਕ MCP ਸਰਵਰ 'ਤੇ OAuth ਟੋਕਨ ਦੀ ਚੋਰੀ। ਜੇ ਟੋਕਨ ਚੋਰੀ ਹੋ ਜਾਵੇ, ਤਾਂ ਇਸਦਾ ਇਸਤੇਮਾਲ MCP ਸਰਵਰ ਦੀ ਨਕਲ ਕਰਨ ਅਤੇ ਉਸ ਸੇਵਾ ਤੋਂ ਸਰੋਤਾਂ ਅਤੇ ਡਾਟਾ ਤੱਕ ਪਹੁੰਚ ਲਈ ਕੀਤਾ ਜਾ ਸਕਦਾ ਹੈ ਜਿਸ ਲਈ OAuth ਟੋਕਨ ਜਾਰੀ ਕੀਤਾ ਗਿਆ ਹੈ।

#### Token Passthrough  
ਟੋਕਨ ਪਾਸਥਰੂ ਅਧਿਕਾਰ ਵਿਸ਼ੇਸ਼ਤਾ ਵਿੱਚ ਸਪਸ਼ਟ ਤੌਰ 'ਤੇ ਮਨਾਹੀ ਹੈ ਕਿਉਂਕਿ ਇਹ ਕਈ ਸੁਰੱਖਿਆ ਖ਼ਤਰਿਆਂ ਨੂੰ ਜਨਮ ਦਿੰਦਾ ਹੈ, ਜਿਨ੍ਹਾਂ ਵਿੱਚ ਸ਼ਾਮਲ ਹਨ:

#### ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਚਾਲਾਕੀ ਨਾਲ ਬਾਈਪਾਸ  
MCP ਸਰਵਰ ਜਾਂ ਡਾਊਨਸਟਰੀਮ API ਮਹੱਤਵਪੂਰਨ ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਜਿਵੇਂ ਕਿ ਰੇਟ ਲਿਮਿਟਿੰਗ, ਬੇਨਤੀ ਦੀ ਜਾਂਚ ਜਾਂ ਟ੍ਰੈਫਿਕ ਮਾਨੀਟਰਿੰਗ ਲਾਗੂ ਕਰ ਸਕਦੇ ਹਨ, ਜੋ ਟੋਕਨ ਦਰਸ਼ਕ ਜਾਂ ਹੋਰ ਪ੍ਰਮਾਣ ਪੱਤਰ ਸੀਮਾਵਾਂ 'ਤੇ ਨਿਰਭਰ ਕਰਦੇ ਹਨ। ਜੇ ਕਲਾਇੰਟ ਸਿੱਧਾ ਡਾਊਨਸਟਰੀਮ API ਨਾਲ ਟੋਕਨ ਪ੍ਰਾਪਤ ਕਰਕੇ ਵਰਤ ਸਕਦੇ ਹਨ ਬਿਨਾਂ MCP ਸਰਵਰ ਵੱਲੋਂ ਟੋਕਨਾਂ ਦੀ ਠੀਕ ਤਰ੍ਹਾਂ ਜਾਂਚ ਕੀਤੇ ਜਾਂ ਇਹ ਯਕੀਨੀ ਬਣਾਏ ਕਿ ਟੋਕਨ ਸਹੀ ਸੇਵਾ ਲਈ ਜਾਰੀ ਕੀਤੇ ਗਏ ਹਨ, ਤਾਂ ਉਹ ਇਹ ਨਿਯੰਤਰਣ ਬਾਈਪਾਸ ਕਰ ਲੈਂਦੇ ਹਨ।

#### ਜਵਾਬਦੇਹੀ ਅਤੇ ਆਡਿਟ ਟ੍ਰੇਲ ਸਮੱਸਿਆਵਾਂ  
ਜਦੋਂ ਕਲਾਇੰਟ ਉੱਪਰਲੇ ਪਾਸੇ ਜਾਰੀ ਕੀਤੇ ਗਏ ਐਕਸੈਸ ਟੋਕਨ ਨਾਲ ਕਾਲ ਕਰਦੇ ਹਨ ਜੋ MCP ਸਰਵਰ ਲਈ ਅਪਾਰਦਰਸ਼ੀ ਹੋ ਸਕਦਾ ਹੈ, ਤਾਂ MCP ਸਰਵਰ MCP ਕਲਾਇੰਟਾਂ ਦੀ ਪਛਾਣ ਜਾਂ ਵੱਖ-ਵੱਖ ਕਰਨ ਵਿੱਚ ਅਸਮਰੱਥ ਰਹੇਗਾ।  
ਡਾਊਨਸਟਰੀਮ ਰਿਸੋਰਸ ਸਰਵਰ ਦੇ ਲੌਗ ਅਜਿਹੀਆਂ ਬੇਨਤੀਆਂ ਦਿਖਾ ਸਕਦੇ ਹਨ ਜੋ ਕਿਸੇ ਹੋਰ ਸਰੋਤ ਜਾਂ ਵੱਖਰੀ ਪਛਾਣ ਵਾਲੇ ਸਰੋਤ ਤੋਂ ਆ ਰਹੀਆਂ ਹਨ, ਨਾ ਕਿ MCP ਸਰਵਰ ਤੋਂ ਜੋ ਟੋਕਨਾਂ ਨੂੰ ਅੱਗੇ ਭੇਜ ਰਿਹਾ ਹੈ।  
ਇਹ ਦੋਹਾਂ ਕਾਰਕ ਘਟਨਾ ਜਾਂਚ, ਨਿਯੰਤਰਣ ਅਤੇ ਆਡਿਟਿੰਗ ਨੂੰ ਮੁਸ਼ਕਲ ਬਣਾਉਂਦੇ ਹਨ।  
ਜੇ MCP ਸਰਵਰ ਟੋਕਨਾਂ ਦੇ ਦਾਅਵਿਆਂ (ਜਿਵੇਂ ਕਿ ਭੂਮਿਕਾਵਾਂ, ਅਧਿਕਾਰ ਜਾਂ ਦਰਸ਼ਕ) ਜਾਂ ਹੋਰ ਮੈਟਾਡੇਟਾ ਦੀ ਜਾਂਚ ਕੀਤੇ ਬਿਨਾਂ ਟੋਕਨ ਪਾਸ ਕਰਦਾ ਹੈ, ਤਾਂ ਚੋਰੀ ਹੋਏ ਟੋਕਨ ਵਾਲਾ ਕੋਈ ਬਦਮਾਸ਼ ਸਰਵਰ ਨੂੰ ਡਾਟਾ ਚੋਰੀ ਲਈ ਪ੍ਰਾਕਸੀ ਵਜੋਂ ਵਰਤ ਸਕਦਾ ਹੈ।

#### ਭਰੋਸਾ ਸੀਮਾ ਸਮੱਸਿਆਵਾਂ  
ਡਾਊਨਸਟਰੀਮ ਰਿਸੋਰਸ ਸਰਵਰ ਖਾਸ ਇਕਾਈਆਂ ਨੂੰ ਭਰੋਸਾ ਦਿੰਦਾ ਹੈ। ਇਹ ਭਰੋਸਾ ਮੂਲ ਜਾਂ ਕਲਾਇੰਟ ਦੇ ਵਿਹਾਰ ਦੇ ਨਮੂਨਿਆਂ ਬਾਰੇ ਧਾਰਣਾਵਾਂ ਸ਼ਾਮਲ ਕਰ ਸਕਦਾ ਹੈ। ਇਸ ਭਰੋਸਾ ਸੀਮਾ ਨੂੰ ਤੋੜਨਾ ਅਣਚਾਹੀਆਂ ਸਮੱਸਿਆਵਾਂ ਦਾ ਕਾਰਨ ਬਣ ਸਕਦਾ ਹੈ।  
ਜੇ ਟੋਕਨ ਬਿਨਾਂ ਠੀਕ ਜਾਂਚ ਦੇ ਕਈ ਸੇਵਾਵਾਂ ਵੱਲੋਂ ਸਵੀਕਾਰ ਕੀਤਾ ਜਾਂਦਾ ਹੈ, ਤਾਂ ਇੱਕ ਸੇਵਾ ਨੂੰ ਕਮਜ਼ੋਰ ਕਰਕੇ ਹਮਲਾਵਰ ਇਸ ਟੋਕਨ ਨਾਲ ਹੋਰ ਜੁੜੀਆਂ ਸੇਵਾਵਾਂ ਤੱਕ ਪਹੁੰਚ ਕਰ ਸਕਦਾ ਹੈ।

#### ਭਵਿੱਖੀ ਅਨੁਕੂਲਤਾ ਖ਼ਤਰਾ  
ਜੇ MCP ਸਰਵਰ ਅੱਜ "ਸ਼ੁੱਧ ਪ੍ਰਾਕਸੀ" ਵਜੋਂ ਸ਼ੁਰੂ ਹੁੰਦਾ ਹੈ, ਤਾਂ ਭਵਿੱਖ ਵਿੱਚ ਇਸਨੂੰ ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਸ਼ਾਮਲ ਕਰਨ ਦੀ ਲੋੜ ਪੈ ਸਕਦੀ ਹੈ। ਟੋਕਨ ਦਰਸ਼ਕ ਵੱਖਰਾ ਰੱਖਣਾ ਸੁਰੱਖਿਆ ਮਾਡਲ ਨੂੰ ਵਿਕਸਤ ਕਰਨ ਵਿੱਚ ਸਹੂਲਤ ਦਿੰਦਾ ਹੈ।

### ਘਟਾਉਣ ਵਾਲੇ ਨਿਯੰਤਰਣ

**MCP ਸਰਵਰ ਕਿਸੇ ਵੀ ਐਸੇ ਟੋਕਨ ਨੂੰ ਸਵੀਕਾਰ ਨਹੀਂ ਕਰ ਸਕਦਾ ਜੋ ਖਾਸ ਤੌਰ 'ਤੇ MCP ਸਰਵਰ ਲਈ ਜਾਰੀ ਨਾ ਕੀਤਾ ਗਿਆ ਹੋਵੇ**

- **ਅਧਿਕਾਰ ਲਾਜਿਕ ਦੀ ਸਮੀਖਿਆ ਅਤੇ ਮਜ਼ਬੂਤੀ:** ਆਪਣੇ MCP ਸਰਵਰ ਦੀ ਅਧਿਕਾਰ ਲਾਗੂਆਤ ਦੀ ਧਿਆਨ ਨਾਲ ਜਾਂਚ ਕਰੋ ਤਾਂ ਜੋ ਸਿਰਫ਼ ਮਨਜ਼ੂਰਸ਼ੁਦਾ ਯੂਜ਼ਰ ਅਤੇ ਕਲਾਇੰਟ ਸੰਵੇਦਨਸ਼ੀਲ ਸਰੋਤਾਂ ਤੱਕ ਪਹੁੰਚ ਸਕਣ। ਵਿਹਾਰਕ ਮਾਰਗਦਰਸ਼ਨ ਲਈ ਵੇਖੋ [Azure API Management Your Auth Gateway For MCP Servers | Microsoft Community Hub](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690) ਅਤੇ [Using Microsoft Entra ID To Authenticate With MCP Servers Via Sessions - Den Delimarsky](https://den.dev/blog/mcp-server-auth-entra-id-session/)।
- **ਸੁਰੱਖਿਅਤ ਟੋਕਨ ਅਮਲਾਂ ਨੂੰ ਲਾਗੂ ਕਰੋ:** [Microsoft ਦੇ ਟੋਕਨ ਵੈਰੀਫਿਕੇਸ਼ਨ ਅਤੇ ਲਾਈਫਟਾਈਮ ਲਈ ਵਧੀਆ ਅਮਲ](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens) ਦੀ ਪਾਲਣਾ ਕਰੋ ਤਾਂ ਜੋ ਐਕਸੈਸ ਟੋਕਨਾਂ ਦੀ ਗਲਤ ਵਰਤੋਂ ਅਤੇ ਟੋਕਨ ਰੀਪਲੇ ਜਾਂ ਚੋਰੀ ਦੇ ਖ਼ਤਰੇ ਨੂੰ ਘਟਾਇਆ ਜਾ ਸਕੇ।
- **ਟੋਕਨ ਸਟੋਰੇਜ ਦੀ ਸੁਰੱਖਿਆ ਕਰੋ:** ਹਮੇਸ਼ਾ ਟੋਕਨਾਂ ਨੂੰ ਸੁਰੱਖਿਅਤ ਢੰਗ ਨਾਲ ਸਟੋਰ ਕਰੋ ਅਤੇ ਇਨਕ੍ਰਿਪਸ਼ਨ ਦੀ ਵਰਤੋਂ ਕਰੋ ਤਾਂ ਜੋ ਇਹ ਸੁਰੱਖਿਅਤ ਰਹਿਣ, ਚਾਹੇ ਰੈਸਟ 'ਤੇ ਹੋਵੇ ਜਾਂ ਟ੍ਰਾਂਜ਼ਿਟ ਵਿੱਚ। ਲਾਗੂ ਕਰਨ ਲਈ ਸੁਝਾਅ ਲਈ ਵੇਖੋ [Use secure token storage and encrypt tokens](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2)।

# MCP ਸਰਵਰਾਂ ਲਈ ਜ਼ਿਆਦਾ ਅਧਿਕਾਰ

### ਸਮੱਸਿਆ ਦਾ ਬਿਆਨ  
MCP ਸਰਵਰਾਂ ਨੂੰ ਉਹ ਸੇਵਾ ਜਾਂ ਸਰੋਤ ਜਿਸ ਨਾਲ ਉਹ ਜੁੜੇ ਹਨ, ਉਸ ਲਈ ਜ਼ਿਆਦਾ ਅਧਿਕਾਰ ਦਿੱਤੇ ਗਏ ਹੋ ਸਕਦੇ ਹਨ। ਉਦਾਹਰਨ ਵਜੋਂ, ਇੱਕ MCP ਸਰਵਰ ਜੋ AI ਵਿਕਰੀ ਐਪਲੀਕੇਸ਼ਨ ਦਾ ਹਿੱਸਾ ਹੈ ਅਤੇ ਇੱਕ ਉਦਯੋਗਿਕ ਡਾਟਾ ਸਟੋਰ ਨਾਲ ਜੁੜਦਾ ਹੈ, ਉਸਨੂੰ ਸਿਰਫ਼ ਵਿਕਰੀ ਡਾਟਾ ਤੱਕ ਪਹੁੰਚ ਹੋਣੀ ਚਾਹੀਦੀ ਹੈ, ਨਾ ਕਿ ਸਟੋਰ ਵਿੱਚ ਸਾਰੇ ਫਾਇਲਾਂ ਤੱਕ। ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ ਦੇ ਸਿਧਾਂਤ (ਸਭ ਤੋਂ ਪੁਰਾਣਾ ਸੁਰੱਖਿਆ ਸਿਧਾਂਤ) ਨੂੰ ਯਾਦ ਰੱਖਦੇ ਹੋਏ, ਕਿਸੇ ਵੀ ਸਰੋਤ ਨੂੰ ਉਹਨਾਂ ਕੰਮਾਂ ਲਈ ਜ਼ਿਆਦਾ ਅਧਿਕਾਰ ਨਹੀਂ ਮਿਲਣੇ ਚਾਹੀਦੇ ਜੋ ਉਹ ਕਰਨਾ ਹੈ। AI ਇਸ ਖੇਤਰ ਵਿੱਚ ਵਧੇਰੇ ਚੁਣੌਤੀ ਪੈਦਾ ਕਰਦਾ ਹੈ ਕਿਉਂਕਿ ਇਸਨੂੰ ਲਚਕੀਲਾ ਬਣਾਉਣ ਲਈ, ਸਹੀ ਅਧਿਕਾਰਾਂ ਦੀ ਪਰਿਭਾਸ਼ਾ ਕਰਨਾ ਮੁਸ਼ਕਲ ਹੋ ਸਕਦਾ ਹੈ।

### ਖ਼ਤਰੇ  
- ਜ਼ਿਆਦਾ ਅਧਿਕਾਰ ਦੇਣ ਨਾਲ MCP ਸਰਵਰ ਨੂੰ ਐਸਾ ਡਾਟਾ ਬਾਹਰ ਕੱਢਣ ਜਾਂ ਸੋਧਣ ਦੀ ਆਗਿਆ ਮਿਲ ਸਕਦੀ ਹੈ ਜਿਸ ਤੱਕ ਉਸਨੂੰ ਪਹੁੰਚ ਨਹੀਂ ਹੋਣੀ ਚਾਹੀਦੀ। ਜੇ ਡਾਟਾ ਵਿਅਕਤੀਗਤ ਪਛਾਣਯੋਗ ਜਾਣਕਾਰੀ (PII) ਹੈ, ਤਾਂ ਇਹ ਪ੍ਰਾਈਵੇਸੀ ਸਮੱਸਿਆ ਵੀ ਬਣ ਸਕਦੀ ਹੈ।

### ਘਟਾਉਣ ਵਾਲੇ ਨਿਯੰਤਰਣ  
- **ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ ਦਾ ਸਿਧਾਂਤ ਲਾਗੂ ਕਰੋ:** MCP ਸਰਵਰ ਨੂੰ ਸਿਰਫ਼ ਉਹੀ ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ ਦਿਓ ਜੋ ਉਸਦੇ ਕੰਮ ਕਰਨ ਲਈ ਲੋੜੀਂਦੇ ਹਨ। ਨਿਯਮਤ ਤੌਰ 'ਤੇ ਇਹ ਅਧਿਕਾਰ ਸਮੀਖਿਆ ਕਰੋ ਅਤੇ ਅਪਡੇਟ ਕਰੋ ਤਾਂ ਜੋ ਇਹ ਜ਼ਰੂਰਤ ਤੋਂ ਵੱਧ ਨਾ ਹੋਣ। ਵਿਸਥਾਰ
ਮੁੜ-ਭੁੱਲਿਆ ਡਿਪਟੀ ਸਮੱਸਿਆ ਇੱਕ ਸੁਰੱਖਿਆ ਦੀ ਕਮਜ਼ੋਰੀ ਹੈ ਜੋ ਉਸ ਵੇਲੇ ਹੁੰਦੀ ਹੈ ਜਦੋਂ MCP ਸਰਵਰ MCP ਕਲਾਇੰਟਾਂ ਅਤੇ ਤੀਜੀ-ਪੱਖੀ APIs ਦੇ ਵਿਚਕਾਰ ਪ੍ਰਾਕਸੀ ਵਜੋਂ ਕੰਮ ਕਰਦਾ ਹੈ। ਇਹ ਕਮਜ਼ੋਰੀ ਉਸ ਸਮੇਂ ਫਾਇਦਾ ਉਠਾਈ ਜਾ ਸਕਦੀ ਹੈ ਜਦੋਂ MCP ਸਰਵਰ ਇੱਕ ਸਥਿਰ ਕਲਾਇੰਟ ID ਦੀ ਵਰਤੋਂ ਕਰਦਾ ਹੈ ਤੀਜੀ-ਪੱਖੀ ਅਥਾਰਟੀ ਸਰਵਰ ਨਾਲ ਪ੍ਰਮਾਣਿਕਤਾ ਲਈ, ਜੋ ਡਾਇਨਾਮਿਕ ਕਲਾਇੰਟ ਰਜਿਸਟ੍ਰੇਸ਼ਨ ਸਹਾਇਤਾ ਨਹੀਂ ਦਿੰਦਾ।

### ਖਤਰੇ

- **ਕੁਕੀ-ਆਧਾਰਿਤ ਸਹਿਮਤੀ ਬਾਈਪਾਸ**: ਜੇਕਰ ਯੂਜ਼ਰ ਪਹਿਲਾਂ MCP ਪ੍ਰਾਕਸੀ ਸਰਵਰ ਰਾਹੀਂ ਪ੍ਰਮਾਣਿਤ ਹੋ ਚੁੱਕਾ ਹੈ, ਤਾਂ ਤੀਜੀ-ਪੱਖੀ ਅਥਾਰਟੀ ਸਰਵਰ ਯੂਜ਼ਰ ਦੇ ਬ੍ਰਾਊਜ਼ਰ ਵਿੱਚ ਸਹਿਮਤੀ ਕੁਕੀ ਸੈੱਟ ਕਰ ਸਕਦਾ ਹੈ। ਹਮਲਾਵਰ ਇਸਦਾ ਫਾਇਦਾ ਉਠਾ ਕੇ ਯੂਜ਼ਰ ਨੂੰ ਇੱਕ ਖ਼ਤਰਨਾਕ ਲਿੰਕ ਭੇਜ ਸਕਦਾ ਹੈ ਜਿਸ ਵਿੱਚ ਇੱਕ ਬਣਾਇਆ ਹੋਇਆ ਅਥਾਰਟੀਕਰਨ ਬੇਨਤੀ ਅਤੇ ਖ਼ਤਰਨਾਕ ਰੀਡਾਇਰੈਕਟ URI ਹੁੰਦਾ ਹੈ।
- **ਅਥਾਰਟੀਕਰਨ ਕੋਡ ਦੀ ਚੋਰੀ**: ਜਦੋਂ ਯੂਜ਼ਰ ਖ਼ਤਰਨਾਕ ਲਿੰਕ 'ਤੇ ਕਲਿੱਕ ਕਰਦਾ ਹੈ, ਤੀਜੀ-ਪੱਖੀ ਅਥਾਰਟੀ ਸਰਵਰ ਮੌਜੂਦਾ ਕੁਕੀ ਕਾਰਨ ਸਹਿਮਤੀ ਸਕ੍ਰੀਨ ਨੂੰ ਛੱਡ ਸਕਦਾ ਹੈ, ਅਤੇ ਅਥਾਰਟੀਕਰਨ ਕੋਡ ਹਮਲਾਵਰ ਦੇ ਸਰਵਰ ਨੂੰ ਰੀਡਾਇਰੈਕਟ ਹੋ ਸਕਦਾ ਹੈ।
- **ਗੈਰ-ਅਧਿਕ੍ਰਿਤ API ਪਹੁੰਚ**: ਹਮਲਾਵਰ ਚੋਰੀ ਹੋਏ ਅਥਾਰਟੀਕਰਨ ਕੋਡ ਨੂੰ ਐਕਸੈਸ ਟੋਕਨ ਵਿੱਚ ਬਦਲ ਕੇ ਯੂਜ਼ਰ ਵਜੋਂ ਤੀਜੀ-ਪੱਖੀ API ਤੱਕ ਬਿਨਾਂ ਸਪਸ਼ਟ ਮਨਜ਼ੂਰੀ ਦੇ ਪਹੁੰਚ ਕਰ ਸਕਦਾ ਹੈ।

### ਰੋਕਥਾਮ ਦੇ ਉਪਾਅ

- **ਸਪਸ਼ਟ ਸਹਿਮਤੀ ਦੀ ਲੋੜ**: ਸਥਿਰ ਕਲਾਇੰਟ ID ਵਰਤਣ ਵਾਲੇ MCP ਪ੍ਰਾਕਸੀ ਸਰਵਰਾਂ ਨੂੰ ਹਰ ਡਾਇਨਾਮਿਕ ਰਜਿਸਟਰਡ ਕਲਾਇੰਟ ਲਈ ਯੂਜ਼ਰ ਦੀ ਸਹਿਮਤੀ ਲੈਣੀ **ਲਾਜ਼ਮੀ** ਹੈ, ਤੀਜੀ-ਪੱਖੀ ਅਥਾਰਟੀ ਸਰਵਰਾਂ ਨੂੰ ਅੱਗੇ ਭੇਜਣ ਤੋਂ ਪਹਿਲਾਂ।
- **ਠੀਕ OAuth ਲਾਗੂ ਕਰਨਾ**: OAuth 2.1 ਦੀਆਂ ਸੁਰੱਖਿਆ ਦੀਆਂ ਸਭ ਤੋਂ ਵਧੀਆ ਪ੍ਰਥਾਵਾਂ ਦੀ ਪਾਲਣਾ ਕਰੋ, ਜਿਸ ਵਿੱਚ ਅਥਾਰਟੀਕਰਨ ਬੇਨਤੀਆਂ ਲਈ ਕੋਡ ਚੈਲੈਂਜ (PKCE) ਦੀ ਵਰਤੋਂ ਸ਼ਾਮਲ ਹੈ, ਤਾਂ ਜੋ ਇੰਟਰਸੈਪਸ਼ਨ ਹਮਲਿਆਂ ਤੋਂ ਬਚਿਆ ਜਾ ਸਕੇ।
- **ਕਲਾਇੰਟ ਦੀ ਜਾਂਚ**: ਰੀਡਾਇਰੈਕਟ URI ਅਤੇ ਕਲਾਇੰਟ ਆਈਡੈਂਟੀਫਾਇਰ ਦੀ ਸਖ਼ਤ ਜਾਂਚ ਲਾਗੂ ਕਰੋ ਤਾਂ ਜੋ ਖ਼ਤਰਨਾਕ ਹਮਲਾਵਰਾਂ ਵੱਲੋਂ ਸ਼ੋਸ਼ਣ ਤੋਂ ਬਚਿਆ ਜਾ ਸਕੇ।

# ਟੋਕਨ ਪਾਸਥਰੂ ਕਮਜ਼ੋਰੀਆਂ

### ਸਮੱਸਿਆ ਦਾ ਬਿਆਨ

"ਟੋਕਨ ਪਾਸਥਰੂ" ਇੱਕ ਐਂਟੀ-ਪੈਟਰਨ ਹੈ ਜਿੱਥੇ MCP ਸਰਵਰ MCP ਕਲਾਇੰਟ ਤੋਂ ਟੋਕਨ ਲੈਂਦਾ ਹੈ ਬਿਨਾਂ ਇਹ ਜਾਂਚੇ ਕਿ ਟੋਕਨ ਸਹੀ ਤੌਰ 'ਤੇ MCP ਸਰਵਰ ਲਈ ਜਾਰੀ ਕੀਤੇ ਗਏ ਹਨ ਜਾਂ ਨਹੀਂ, ਅਤੇ ਫਿਰ ਉਹਨਾਂ ਨੂੰ ਡਾਊਨਸਟਰੀਮ APIs ਨੂੰ "ਪਾਸ ਕਰਦਾ" ਹੈ। ਇਹ ਅਮਲ MCP ਅਥਾਰਟੀਕਰਨ ਵਿਸ਼ੇਸ਼ਤਾ ਦਾ ਸਪਸ਼ਟ ਉਲੰਘਣ ਹੈ ਅਤੇ ਗੰਭੀਰ ਸੁਰੱਖਿਆ ਖਤਰਿਆਂ ਨੂੰ ਜਨਮ ਦਿੰਦਾ ਹੈ।

### ਖਤਰੇ

- **ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣਾਂ ਦੀ ਚਾਲਾਕੀ ਨਾਲ ਬਾਈਪਾਸਿੰਗ**: ਜੇਕਰ ਕਲਾਇੰਟ ਬਿਨਾਂ ਠੀਕ ਜਾਂਚ ਦੇ ਟੋਕਨ ਸਿੱਧਾ ਡਾਊਨਸਟਰੀਮ APIs ਨਾਲ ਵਰਤ ਸਕਦੇ ਹਨ, ਤਾਂ ਉਹ ਮਹੱਤਵਪੂਰਨ ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਜਿਵੇਂ ਕਿ ਰੇਟ ਲਿਮਿਟਿੰਗ, ਬੇਨਤੀ ਜਾਂਚ ਜਾਂ ਟ੍ਰੈਫਿਕ ਮਾਨੀਟਰਿੰਗ ਨੂੰ ਬਾਈਪਾਸ ਕਰ ਸਕਦੇ ਹਨ।
- **ਜਵਾਬਦੇਹੀ ਅਤੇ ਆਡਿਟ ਟ੍ਰੇਲ ਸਮੱਸਿਆਵਾਂ**: ਜਦੋਂ ਕਲਾਇੰਟ ਉੱਪਰਲੇ ਪਾਸੇ ਜਾਰੀ ਕੀਤੇ ਗਏ ਐਕਸੈਸ ਟੋਕਨ ਵਰਤਦੇ ਹਨ, MCP ਸਰਵਰ ਕਲਾਇੰਟਾਂ ਦੀ ਪਛਾਣ ਜਾਂ ਵੱਖ-ਵੱਖ ਕਰਨ ਵਿੱਚ ਅਸਮਰੱਥ ਹੋਵੇਗਾ, ਜਿਸ ਨਾਲ ਘਟਨਾ ਜਾਂਚ ਅਤੇ ਆਡਿਟਿੰਗ ਮੁਸ਼ਕਲ ਹੋ ਜਾਵੇਗੀ।
- **ਡਾਟਾ ਚੋਰੀ**: ਜੇ ਟੋਕਨ ਬਿਨਾਂ ਠੀਕ ਦਾਅਵਿਆਂ ਦੀ ਜਾਂਚ ਦੇ ਪਾਸ ਕੀਤੇ ਜਾਂਦੇ ਹਨ, ਤਾਂ ਇੱਕ ਖ਼ਤਰਨਾਕ ਹਮਲਾਵਰ ਚੋਰੀ ਹੋਏ ਟੋਕਨ ਨਾਲ ਸਰਵਰ ਨੂੰ ਡਾਟਾ ਚੋਰੀ ਲਈ ਪ੍ਰਾਕਸੀ ਵਜੋਂ ਵਰਤ ਸਕਦਾ ਹੈ।
- **ਭਰੋਸੇ ਦੀ ਸੀਮਾ ਦੀ ਉਲੰਘਣਾ**: ਡਾਊਨਸਟਰੀਮ ਸਰੋਤ ਸਰਵਰ ਖਾਸ ਇਕਾਈਆਂ ਨੂੰ ਉਤਪੱਤੀ ਜਾਂ ਵਰਤੋਂ ਦੇ ਰੁਝਾਨਾਂ ਬਾਰੇ ਧਾਰਣਾਵਾਂ ਦੇ ਨਾਲ ਭਰੋਸਾ ਦਿੰਦੇ ਹਨ। ਇਸ ਭਰੋਸੇ ਦੀ ਸੀਮਾ ਟੁੱਟਣ ਨਾਲ ਅਣਪਛਾਤੇ ਸੁਰੱਖਿਆ ਮੁੱਦੇ ਪੈਦਾ ਹੋ ਸਕਦੇ ਹਨ।
- **ਬਹੁ-ਸੇਵਾ ਟੋਕਨ ਗਲਤ ਵਰਤੋਂ**: ਜੇ ਟੋਕਨ ਬਿਨਾਂ ਠੀਕ ਜਾਂਚ ਦੇ ਕਈ ਸੇਵਾਵਾਂ ਵੱਲੋਂ ਸਵੀਕਾਰ ਕੀਤੇ ਜਾਂਦੇ ਹਨ, ਤਾਂ ਇੱਕ ਸੇਵਾ ਦੀ ਸੁਰੱਖਿਆ ਭੰਗ ਕਰਨ ਵਾਲਾ ਹਮਲਾਵਰ ਉਸ ਟੋਕਨ ਨਾਲ ਹੋਰ ਜੁੜੀਆਂ ਸੇਵਾਵਾਂ ਤੱਕ ਪਹੁੰਚ ਕਰ ਸਕਦਾ ਹੈ।

### ਰੋਕਥਾਮ ਦੇ ਉਪਾਅ

- **ਟੋਕਨ ਦੀ ਜਾਂਚ**: MCP ਸਰਵਰਾਂ ਨੂੰ **ਕਦੇ ਵੀ** ਉਹ ਟੋਕਨ ਸਵੀਕਾਰ ਨਹੀਂ ਕਰਨੇ ਚਾਹੀਦੇ ਜੋ ਸਪਸ਼ਟ ਤੌਰ 'ਤੇ MCP ਸਰਵਰ ਲਈ ਜਾਰੀ ਨਹੀਂ ਕੀਤੇ ਗਏ।
- **ਦਰਸ਼ਕ ਦੀ ਪੁਸ਼ਟੀ**: ਹਮੇਸ਼ਾ ਜਾਂਚ ਕਰੋ ਕਿ ਟੋਕਨਾਂ ਵਿੱਚ ਦਰਸ਼ਕ ਦਾ ਦਾਅਵਾ MCP ਸਰਵਰ ਦੀ ਪਹਚਾਣ ਨਾਲ ਮੇਲ ਖਾਂਦਾ ਹੈ।
- **ਠੀਕ ਟੋਕਨ ਜੀਵਨਚੱਕਰ ਪ੍ਰਬੰਧਨ**: ਛੋਟੇ ਸਮੇਂ ਵਾਲੇ ਐਕਸੈਸ ਟੋਕਨ ਅਤੇ ਟੋਕਨ ਰੋਟੇਸ਼ਨ ਦੀਆਂ ਸਹੀ ਪ੍ਰਥਾਵਾਂ ਲਾਗੂ ਕਰੋ ਤਾਂ ਜੋ ਟੋਕਨ ਚੋਰੀ ਅਤੇ ਗਲਤ ਵਰਤੋਂ ਦੇ ਖਤਰੇ ਨੂੰ ਘਟਾਇਆ ਜਾ ਸਕੇ।

# ਸੈਸ਼ਨ ਹਾਈਜੈਕਿੰਗ

### ਸਮੱਸਿਆ ਦਾ ਬਿਆਨ

ਸੈਸ਼ਨ ਹਾਈਜੈਕਿੰਗ ਇੱਕ ਹਮਲਾ ਹੈ ਜਿਸ ਵਿੱਚ ਸਰਵਰ ਵੱਲੋਂ ਕਲਾਇੰਟ ਨੂੰ ਦਿੱਤਾ ਗਿਆ ਸੈਸ਼ਨ ID ਗੈਰ-ਅਧਿਕ੍ਰਿਤ ਪੱਖ ਵੱਲੋਂ ਪ੍ਰਾਪਤ ਕਰਕੇ ਉਸੇ ਸੈਸ਼ਨ ID ਨਾਲ ਮੂਲ ਕਲਾਇੰਟ ਵਜੋਂ ਨਕਲ ਕਰਕੇ ਬਿਨਾਂ ਮਨਜ਼ੂਰੀ ਦੇ ਕਾਰਵਾਈਆਂ ਕੀਤੀਆਂ ਜਾਂਦੀਆਂ ਹਨ। ਇਹ ਖ਼ਾਸ ਕਰਕੇ ਸਟੇਟਫੁਲ HTTP ਸਰਵਰਾਂ ਵਿੱਚ ਜੋ MCP ਬੇਨਤੀਆਂ ਨੂੰ ਸੰਭਾਲਦੇ ਹਨ, ਚਿੰਤਾਜਨਕ ਹੈ।

### ਖਤਰੇ

- **ਸੈਸ਼ਨ ਹਾਈਜੈਕ ਪ੍ਰਾਂਪਟ ਇੰਜੈਕਸ਼ਨ**: ਜੇਕਰ ਹਮਲਾਵਰ ਸੈਸ਼ਨ ID ਪ੍ਰਾਪਤ ਕਰ ਲੈਂਦਾ ਹੈ, ਤਾਂ ਉਹ ਉਸ ਸਰਵਰ ਨੂੰ ਖ਼ਤਰਨਾਕ ਇਵੈਂਟ ਭੇਜ ਸਕਦਾ ਹੈ ਜੋ ਸੈਸ਼ਨ ਸਟੇਟ ਸਾਂਝਾ ਕਰਦਾ ਹੈ, ਜਿਸ ਨਾਲ ਨੁਕਸਾਨਦਾਇਕ ਕਾਰਵਾਈਆਂ ਜਾਂ ਸੰਵੇਦਨਸ਼ੀਲ ਡਾਟਾ ਤੱਕ ਪਹੁੰਚ ਹੋ ਸਕਦੀ ਹੈ।
- **ਸੈਸ਼ਨ ਹਾਈਜੈਕ ਨਕਲ**: ਚੋਰੀ ਹੋਏ ਸੈਸ਼ਨ ID ਨਾਲ ਹਮਲਾਵਰ MCP ਸਰਵਰ ਨੂੰ ਸਿੱਧਾ ਕਾਲ ਕਰ ਸਕਦਾ ਹੈ, ਪ੍ਰਮਾਣਿਕਤਾ ਨੂੰ ਬਾਈਪਾਸ ਕਰਦਾ ਅਤੇ ਮੂਲ ਯੂਜ਼ਰ ਵਜੋਂ ਵਰਤਿਆ ਜਾ ਸਕਦਾ ਹੈ।
- **ਖ਼ਰਾਬ ਰੀਸਮੂਏਬਲ ਸਟ੍ਰੀਮਜ਼**: ਜਦੋਂ ਸਰਵਰ ਰੀਡਿਲਿਵਰੀ ਜਾਂ ਰੀਸਮੂਏਬਲ ਸਟ੍ਰੀਮਜ਼ ਦਾ ਸਮਰਥਨ ਕਰਦਾ ਹੈ, ਤਾਂ ਹਮਲਾਵਰ ਬੇਨਤੀ ਨੂੰ ਅਕਾਲ ਮੋੜ ਸਕਦਾ ਹੈ, ਜਿਸ ਨਾਲ ਮੂਲ ਕਲਾਇੰਟ ਦੁਆਰਾ ਬਾਅਦ ਵਿੱਚ ਖ਼ਤਰਨਾਕ ਸਮੱਗਰੀ ਨਾਲ ਬੇਨਤੀ ਮੁੜ ਚਾਲੂ ਕੀਤੀ ਜਾ ਸਕਦੀ ਹੈ।

### ਰੋਕਥਾਮ ਦੇ ਉਪਾਅ

- **ਅਥਾਰਟੀਕਰਨ ਦੀ ਪੁਸ਼ਟੀ**: ਜਿਹੜੇ MCP ਸਰਵਰ ਅਥਾਰਟੀਕਰਨ ਲਾਗੂ ਕਰਦੇ ਹਨ, ਉਹ ਸਾਰੇ ਆਉਣ ਵਾਲੇ ਬੇਨਤੀਆਂ ਦੀ ਜਾਂਚ ਕਰਨੀ **ਲਾਜ਼ਮੀ** ਹੈ ਅਤੇ ਪ੍ਰਮਾਣਿਕਤਾ ਲਈ ਸੈਸ਼ਨ ਦੀ ਵਰਤੋਂ ਨਹੀਂ ਕਰਨੀ ਚਾਹੀਦੀ।
- **ਸੁਰੱਖਿਅਤ ਸੈਸ਼ਨ ID**: MCP ਸਰਵਰਾਂ ਨੂੰ ਸੁਰੱਖਿਅਤ, ਗੈਰ-ਪੂਰਵ ਨਿਰਧਾਰਿਤ ਸੈਸ਼ਨ ID ਵਰਤਣੇ **ਲਾਜ਼ਮੀ** ਹਨ ਜੋ ਸੁਰੱਖਿਅਤ ਰੈਂਡਮ ਨੰਬਰ ਜਨਰੇਟਰ ਨਾਲ ਬਣਾਏ ਗਏ ਹੋਣ। ਅਨੁਮਾਨਿਤ ਜਾਂ ਲੜੀਵਾਰ ਆਈਡੀ ਤੋਂ ਬਚੋ।
- **ਯੂਜ਼ਰ-ਵਿਸ਼ੇਸ਼ ਸੈਸ਼ਨ ਬਾਈਂਡਿੰਗ**: MCP ਸਰਵਰਾਂ ਨੂੰ ਚਾਹੀਦਾ ਹੈ ਕਿ ਸੈਸ਼ਨ ID ਨੂੰ ਯੂਜ਼ਰ-ਵਿਸ਼ੇਸ਼ ਜਾਣਕਾਰੀ ਨਾਲ ਜੋੜਨ, ਜਿਵੇਂ ਕਿ `<user_id>:<session_id>` ਫਾਰਮੈਟ ਵਿੱਚ, ਜੋ ਪ੍ਰਮਾਣਿਤ ਯੂਜ਼ਰ ਨਾਲ ਖਾਸ ਹੋਵੇ।
- **ਸੈਸ਼ਨ ਮਿਆਦ ਖ਼ਤਮ ਹੋਣਾ**: ਸੈਸ਼ਨ ਦੀ ਮਿਆਦ ਖ਼ਤਮ ਕਰਨ ਅਤੇ ਰੋਟੇਸ਼ਨ ਲਾਗੂ ਕਰੋ ਤਾਂ ਜੋ ਜੇ ਸੈਸ਼ਨ ID ਚੋਰੀ ਹੋ ਜਾਵੇ ਤਾਂ ਖਤਰੇ ਦਾ ਸਮਾਂ ਘੱਟ ਹੋਵੇ।
- **ਟ੍ਰਾਂਸਪੋਰਟ ਸੁਰੱਖਿਆ**: ਸਾਰੇ ਸੰਚਾਰ ਲਈ ਹਮੇਸ਼ਾ HTTPS ਦੀ ਵਰਤੋਂ ਕਰੋ ਤਾਂ ਜੋ ਸੈਸ਼ਨ ID ਦੀ ਚੋਰੀ ਤੋਂ ਬਚਿਆ ਜਾ ਸਕੇ।

# ਸਪਲਾਈ ਚੇਨ ਸੁਰੱਖਿਆ

ਸਪਲਾਈ ਚੇਨ ਸੁਰੱਖਿਆ AI ਯੁੱਗ ਵਿੱਚ ਬੁਨਿਆਦੀ ਰਹਿੰਦੀ ਹੈ, ਪਰ ਤੁਹਾਡੇ ਸਪਲਾਈ ਚੇਨ ਦੀ ਪਰਿਭਾਸ਼ਾ ਹੁਣ ਵਧ ਗਈ ਹੈ। ਰਵਾਇਤੀ ਕੋਡ ਪੈਕੇਜਾਂ ਦੇ ਇਲਾਵਾ, ਹੁਣ ਤੁਹਾਨੂੰ ਸਾਰੇ AI-ਸੰਬੰਧਿਤ ਹਿੱਸਿਆਂ ਦੀ ਕੜੀ ਜਾਂਚ ਅਤੇ ਨਿਗਰਾਨੀ ਕਰਨੀ ਚਾਹੀਦੀ ਹੈ, ਜਿਸ ਵਿੱਚ ਫਾਊਂਡੇਸ਼ਨ ਮਾਡਲ, ਐਂਬੈਡਿੰਗ ਸੇਵਾਵਾਂ, ਸੰਦਰਭ ਪ੍ਰਦਾਤਾ ਅਤੇ ਤੀਜੀ-ਪੱਖੀ APIs ਸ਼ਾਮਲ ਹਨ। ਜੇ ਇਹਨਾਂ ਨੂੰ ਠੀਕ ਤਰੀਕੇ ਨਾਲ ਪ੍ਰਬੰਧਿਤ ਨਾ ਕੀਤਾ ਜਾਵੇ ਤਾਂ ਇਹ ਖਤਰੇ ਜਾਂ ਕਮਜ਼ੋਰੀਆਂ ਲਿਆ ਸਕਦੇ ਹਨ।

**AI ਅਤੇ MCP ਲਈ ਮੁੱਖ ਸਪਲਾਈ ਚੇਨ ਸੁਰੱਖਿਆ ਅਭਿਆਸ:**
- **ਇੰਟਿਗ੍ਰੇਸ਼ਨ ਤੋਂ ਪਹਿਲਾਂ ਸਾਰੇ ਹਿੱਸਿਆਂ ਦੀ ਜਾਂਚ ਕਰੋ:** ਇਸ ਵਿੱਚ ਸਿਰਫ ਖੁੱਲ੍ਹੇ ਸਰੋਤ ਲਾਇਬ੍ਰੇਰੀਆਂ ਹੀ ਨਹੀਂ, ਸਗੋਂ AI ਮਾਡਲ, ਡਾਟਾ ਸਰੋਤ ਅਤੇ ਬਾਹਰੀ APIs ਵੀ ਸ਼ਾਮਲ ਹਨ। ਹਮੇਸ਼ਾ ਮੂਲ, ਲਾਇਸੈਂਸਿੰਗ ਅਤੇ ਜਾਣੇ-ਪਹਚਾਣੇ ਕਮਜ਼ੋਰੀਆਂ ਦੀ ਜਾਂਚ ਕਰੋ।
- **ਸੁਰੱਖਿਅਤ ਡਿਪਲੋਇਮੈਂਟ ਪਾਈਪਲਾਈਨ ਬਣਾਓ:** ਆਟੋਮੇਟਿਕ CI/CD ਪਾਈਪਲਾਈਨ ਵਰਤੋਂ ਜੋ ਸੁਰੱਖਿਆ ਸਕੈਨਿੰਗ ਨਾਲ ਜੁੜੀ ਹੋਈ ਹੋਵੇ ਤਾਂ ਜੋ ਮੁੱਦੇ ਜਲਦੀ ਪਤਾ ਲੱਗਣ। ਸਿਰਫ ਭਰੋਸੇਯੋਗ ਆਰਟੀਫੈਕਟ ਪ੍ਰੋਡਕਸ਼ਨ ਵਿੱਚ ਡਿਪਲੋਇ ਕਰੋ।
- **ਲਗਾਤਾਰ ਨਿਗਰਾਨੀ ਅਤੇ ਆਡਿਟਿੰਗ ਕਰੋ:** ਸਾਰੇ ਨਿਰਭਰਤਾਵਾਂ, ਮਾਡਲਾਂ ਅਤੇ ਡਾਟਾ ਸੇਵਾਵਾਂ ਦੀ ਲਗਾਤਾਰ ਨਿਗਰਾਨੀ ਕਰੋ ਤਾਂ ਜੋ ਨਵੀਆਂ ਕਮਜ਼ੋਰੀਆਂ ਜਾਂ ਸਪਲਾਈ ਚੇਨ ਹਮਲਿਆਂ ਦਾ ਪਤਾ ਲੱਗ ਸਕੇ।
- **ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ ਅਤੇ ਪਹੁੰਚ ਨਿਯੰਤਰਣ ਲਾਗੂ ਕਰੋ:** ਮਾਡਲਾਂ, ਡਾਟਾ ਅਤੇ ਸੇਵਾਵਾਂ ਤੱਕ ਪਹੁੰਚ ਸਿਰਫ਼ ਉਸੇ ਹੱਦ ਤੱਕ ਸੀਮਿਤ ਕਰੋ ਜੋ MCP ਸਰਵਰ ਲਈ ਜ਼ਰੂਰੀ ਹੈ।
- **ਖ਼ਤਰਿਆਂ ਦਾ ਜਲਦੀ ਜਵਾਬ ਦਿਓ:** ਜੇ ਕੋਈ ਹਿੱਸਾ ਭੰਗ ਹੋ ਜਾਵੇ ਤਾਂ ਉਸਨੂੰ ਪੈਚ ਕਰਨ ਜਾਂ ਬਦਲਣ ਲਈ ਪ੍ਰਕਿਰਿਆ ਬਣਾਓ, ਅਤੇ ਗੁਪਤ ਚਾਬੀਆਂ ਜਾਂ ਪ੍ਰਮਾਣ ਪੱਤਰਾਂ ਨੂੰ ਰੋਟੇਟ ਕਰਨ ਲਈ ਵੀ ਤਿਆਰ ਰਹੋ।

[GitHub Advanced Security](https://github.com/security/advanced-security) ਵਿੱਚ ਸਿਕ੍ਰੇਟ ਸਕੈਨਿੰਗ, ਡਿਪੈਂਡੈਂਸੀ ਸਕੈਨਿੰਗ ਅਤੇ CodeQL ਵਿਸ਼ਲੇਸ਼ਣ ਵਰਗੀਆਂ ਵਿਸ਼ੇਸ਼ਤਾਵਾਂ ਹਨ। ਇਹ ਟੂਲ [Azure DevOps](https://azure.microsoft.com/en-us/products/devops) ਅਤੇ [Azure Repos](https://azure.microsoft.com/en-us/products/devops/repos/) ਨਾਲ ਇੰਟਿਗ੍ਰੇਟ ਹੁੰਦੇ ਹਨ, ਜੋ ਟੀਮਾਂ ਨੂੰ ਕੋਡ ਅਤੇ AI ਸਪਲਾਈ ਚੇਨ ਹਿੱਸਿਆਂ ਵਿੱਚ ਕਮਜ਼ੋਰੀਆਂ ਦੀ ਪਹਚਾਣ ਅਤੇ ਰੋਕਥਾਮ ਵਿੱਚ ਮਦਦ ਕਰਦੇ ਹਨ।

ਮਾਈਕ੍ਰੋਸਾਫਟ ਵੀ ਆਪਣੇ ਸਾਰੇ ਉਤਪਾਦਾਂ ਲਈ ਅੰਦਰੂਨੀ ਤੌਰ 'ਤੇ ਵਿਆਪਕ ਸਪਲਾਈ ਚੇਨ ਸੁਰੱਖਿਆ ਅਭਿਆਸ ਲਾਗੂ ਕਰਦਾ ਹੈ। ਹੋਰ ਜਾਣਕਾਰੀ ਲਈ [The Journey to Secure the Software Supply Chain at Microsoft](https://devblogs.microsoft.com/engineering-at-microsoft/the-journey-to-secure-the-software-supply-chain-at-microsoft/) ਵੇਖੋ।
