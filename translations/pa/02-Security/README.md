# ਸੁਰੱਖਿਆ ਲਈ ਵਧੀਆ ਅਮਲ

Model Context Protocol (MCP) ਨੂੰ ਅਪਣਾਉਣ ਨਾਲ AI-ਚਲਿਤ ਐਪਲੀਕੇਸ਼ਨਾਂ ਵਿੱਚ ਸ਼ਕਤੀਸ਼ਾਲੀ ਨਵੀਆਂ ਸਮਰੱਥਾਵਾਂ ਆਉਂਦੀਆਂ ਹਨ, ਪਰ ਇਹ ਪਰੰਪਰਾਗਤ ਸਾਫਟਵੇਅਰ ਖਤਰਨਾਂ ਤੋਂ ਵੱਖਰੇ ਵਿਲੱਖਣ ਸੁਰੱਖਿਆ ਚੁਣੌਤੀਆਂ ਵੀ ਲਿਆਉਂਦਾ ਹੈ। ਸੁਰੱਖਿਅਤ ਕੋਡਿੰਗ, ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ ਅਤੇ ਸਪਲਾਈ ਚੇਨ ਸੁਰੱਖਿਆ ਵਰਗੀਆਂ ਮੌਜੂਦਾ ਚਿੰਤਾਵਾਂ ਦੇ ਨਾਲ-ਨਾਲ, MCP ਅਤੇ AI ਵਰਕਲੋਡਾਂ ਨੂੰ ਨਵੇਂ ਖਤਰੇ ਜਿਵੇਂ ਕਿ ਪ੍ਰਾਂਪਟ ਇੰਜੈਕਸ਼ਨ, ਟੂਲ ਜਹਿਰੀਲਾ ਬਣਾਉਣਾ ਅਤੇ ਗਤੀਸ਼ੀਲ ਟੂਲ ਸੋਧ ਦਾ ਸਾਹਮਣਾ ਕਰਨਾ ਪੈਂਦਾ ਹੈ। ਜੇ ਇਹਨਾਂ ਖਤਰਿਆਂ ਨੂੰ ਠੀਕ ਤਰ੍ਹਾਂ ਸੰਭਾਲਿਆ ਨਾ ਜਾਵੇ ਤਾਂ ਇਹ ਡਾਟਾ ਚੋਰੀ, ਪ੍ਰਾਈਵੇਸੀ ਉਲੰਘਣਾ ਅਤੇ ਅਣਚਾਹੇ ਸਿਸਟਮ ਵਿਹਾਰ ਦਾ ਕਾਰਨ ਬਣ ਸਕਦੇ ਹਨ।

ਇਸ ਪਾਠ ਵਿੱਚ MCP ਨਾਲ ਜੁੜੇ ਸਭ ਤੋਂ ਮਹੱਤਵਪੂਰਨ ਸੁਰੱਖਿਆ ਖਤਰਿਆਂ ਦੀ ਚਰਚਾ ਕੀਤੀ ਗਈ ਹੈ—ਜਿਵੇਂ ਕਿ ਪ੍ਰਮਾਣਿਕਤਾ, ਅਧਿਕਾਰ, ਜ਼ਿਆਦਾ ਅਧਿਕਾਰ, ਅਪਰੋਕਸ਼ ਪ੍ਰਾਂਪਟ ਇੰਜੈਕਸ਼ਨ ਅਤੇ ਸਪਲਾਈ ਚੇਨ ਦੀਆਂ ਕਮਜ਼ੋਰੀਆਂ—ਅਤੇ ਉਨ੍ਹਾਂ ਨੂੰ ਘਟਾਉਣ ਲਈ ਕਾਰਗਰ ਨਿਯੰਤਰਣ ਅਤੇ ਵਧੀਆ ਅਮਲ ਦਿੱਤੇ ਗਏ ਹਨ। ਤੁਸੀਂ Microsoft ਦੇ ਹੱਲਾਂ ਜਿਵੇਂ ਕਿ Prompt Shields, Azure Content Safety ਅਤੇ GitHub Advanced Security ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਆਪਣੇ MCP ਇੰਪਲੀਮੈਂਟੇਸ਼ਨ ਨੂੰ ਮਜ਼ਬੂਤ ਕਰਨ ਬਾਰੇ ਵੀ ਸਿੱਖੋਗੇ। ਇਹ ਨਿਯੰਤਰਣ ਸਮਝ ਕੇ ਅਤੇ ਲਾਗੂ ਕਰਕੇ, ਤੁਸੀਂ ਸੁਰੱਖਿਆ ਭੰਗ ਹੋਣ ਦੀ ਸੰਭਾਵਨਾ ਨੂੰ ਕਾਫੀ ਘਟਾ ਸਕਦੇ ਹੋ ਅਤੇ ਆਪਣੇ AI ਸਿਸਟਮਾਂ ਨੂੰ ਮਜ਼ਬੂਤ ਅਤੇ ਭਰੋਸੇਯੋਗ ਬਣਾ ਸਕਦੇ ਹੋ।

# ਸਿੱਖਣ ਦੇ ਲਕੜੇ

ਇਸ ਪਾਠ ਦੇ ਅੰਤ ਤੱਕ, ਤੁਸੀਂ ਸਮਰੱਥ ਹੋਵੋਗੇ:

- Model Context Protocol (MCP) ਵੱਲੋਂ ਲਿਆਂਦੇ ਗਏ ਵਿਲੱਖਣ ਸੁਰੱਖਿਆ ਖਤਰਿਆਂ ਦੀ ਪਛਾਣ ਅਤੇ ਵਿਆਖਿਆ ਕਰਨਾ, ਜਿਵੇਂ ਕਿ ਪ੍ਰਾਂਪਟ ਇੰਜੈਕਸ਼ਨ, ਟੂਲ ਜਹਿਰੀਲਾ ਬਣਾਉਣਾ, ਜ਼ਿਆਦਾ ਅਧਿਕਾਰ ਅਤੇ ਸਪਲਾਈ ਚੇਨ ਦੀਆਂ ਕਮਜ਼ੋਰੀਆਂ।
- MCP ਸੁਰੱਖਿਆ ਖਤਰਿਆਂ ਲਈ ਪ੍ਰਭਾਵਸ਼ਾਲੀ ਨਿਯੰਤਰਣ ਲਾਗੂ ਕਰਨਾ, ਜਿਵੇਂ ਕਿ ਮਜ਼ਬੂਤ ਪ੍ਰਮਾਣਿਕਤਾ, ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ, ਸੁਰੱਖਿਅਤ ਟੋਕਨ ਪ੍ਰਬੰਧਨ ਅਤੇ ਸਪਲਾਈ ਚੇਨ ਦੀ ਜਾਂਚ।
- Microsoft ਦੇ ਹੱਲਾਂ ਜਿਵੇਂ ਕਿ Prompt Shields, Azure Content Safety ਅਤੇ GitHub Advanced Security ਦੀ ਵਰਤੋਂ ਕਰਕੇ MCP ਅਤੇ AI ਵਰਕਲੋਡਾਂ ਦੀ ਸੁਰੱਖਿਆ ਕਰਨਾ।
- ਟੂਲ ਮੈਟਾਡੇਟਾ ਦੀ ਜਾਂਚ ਕਰਨ, ਗਤੀਸ਼ੀਲ ਬਦਲਾਵਾਂ ਦੀ ਨਿਗਰਾਨੀ ਕਰਨ ਅਤੇ ਅਪਰੋਕਸ਼ ਪ੍ਰਾਂਪਟ ਇੰਜੈਕਸ਼ਨ ਹਮਲਿਆਂ ਤੋਂ ਬਚਾਅ ਕਰਨ ਦੀ ਮਹੱਤਤਾ ਨੂੰ ਸਮਝਣਾ।
- ਸੁਰੱਖਿਆ ਦੇ ਸਥਾਪਿਤ ਵਧੀਆ ਅਮਲਾਂ—ਜਿਵੇਂ ਕਿ ਸੁਰੱਖਿਅਤ ਕੋਡਿੰਗ, ਸਰਵਰ ਮਜ਼ਬੂਤੀ ਅਤੇ ਜ਼ੀਰੋ ਟਰੱਸਟ ਆਰਕੀਟੈਕਚਰ—ਨੂੰ ਆਪਣੇ MCP ਇੰਪਲੀਮੈਂਟੇਸ਼ਨ ਵਿੱਚ ਸ਼ਾਮਲ ਕਰਕੇ ਸੁਰੱਖਿਆ ਭੰਗ ਹੋਣ ਦੀ ਸੰਭਾਵਨਾ ਅਤੇ ਪ੍ਰਭਾਵ ਨੂੰ ਘਟਾਉਣਾ।

# MCP ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ

ਜੋ ਵੀ ਸਿਸਟਮ ਮਹੱਤਵਪੂਰਨ ਸਰੋਤਾਂ ਤੱਕ ਪਹੁੰਚ ਰੱਖਦਾ ਹੈ, ਉਸਦੇ ਸੁਰੱਖਿਆ ਚੁਣੌਤੀਆਂ ਹੁੰਦੀਆਂ ਹਨ। ਇਹ ਚੁਣੌਤੀਆਂ ਆਮ ਤੌਰ 'ਤੇ ਮੂਲ ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਅਤੇ ਧਾਰਣਾਵਾਂ ਦੇ ਸਹੀ ਲਾਗੂ ਕਰਨ ਨਾਲ ਹੱਲ ਕੀਤੀਆਂ ਜਾ ਸਕਦੀਆਂ ਹਨ। MCP ਨਵਾਂ ਪਰਿਭਾਸ਼ਿਤ ਪ੍ਰੋਟੋਕੋਲ ਹੈ, ਇਸ ਲਈ ਇਸਦੀ ਵਿਸ਼ੇਸ਼ਤਾਵਾਂ ਤੇਜ਼ੀ ਨਾਲ ਬਦਲ ਰਹੀਆਂ ਹਨ ਅਤੇ ਜਿਵੇਂ ਜਿਵੇਂ ਪ੍ਰੋਟੋਕੋਲ ਵਿਕਸਤ ਹੋਵੇਗਾ, ਇਸਦੇ ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਵੀ ਪੱਕੇ ਹੋਣਗੇ, ਜੋ ਕਿ ਉਦਯੋਗ ਅਤੇ ਸਥਾਪਿਤ ਸੁਰੱਖਿਆ ਆਰਕੀਟੈਕਚਰਾਂ ਅਤੇ ਵਧੀਆ ਅਮਲਾਂ ਨਾਲ ਬਿਹਤਰ ਇੰਟਿਗ੍ਰੇਸ਼ਨ ਯੋਗ ਬਣਾਉਣਗੇ।

[Microsoft Digital Defense Report](https://aka.ms/mddr) ਵਿੱਚ ਪ੍ਰਕਾਸ਼ਿਤ ਖੋਜ ਮੁਤਾਬਕ, 98% ਰਿਪੋਰਟ ਕੀਤੀਆਂ ਗਈਆਂ ਸੁਰੱਖਿਆ ਭੰਗਾਂ ਨੂੰ ਮਜ਼ਬੂਤ ਸੁਰੱਖਿਆ ਸਫਾਈ ਨਾਲ ਰੋਕਿਆ ਜਾ ਸਕਦਾ ਹੈ ਅਤੇ ਕਿਸੇ ਵੀ ਕਿਸਮ ਦੀ ਭੰਗ ਤੋਂ ਬਚਾਅ ਲਈ ਸਭ ਤੋਂ ਵਧੀਆ ਤਰੀਕਾ ਹੈ ਆਪਣੀ ਬੇਸਲਾਈਨ ਸੁਰੱਖਿਆ ਸਫਾਈ, ਸੁਰੱਖਿਅਤ ਕੋਡਿੰਗ ਦੇ ਵਧੀਆ ਅਮਲ ਅਤੇ ਸਪਲਾਈ ਚੇਨ ਸੁਰੱਖਿਆ ਨੂੰ ਠੀਕ ਕਰਨਾ — ਇਹਨਾਂ ਪਰਖੇ ਹੋਏ ਅਮਲਾਂ ਦਾ ਸੁਰੱਖਿਆ ਖਤਰੇ ਨੂੰ ਘਟਾਉਣ ਵਿੱਚ ਸਭ ਤੋਂ ਵੱਡਾ ਪ੍ਰਭਾਵ ਹੁੰਦਾ ਹੈ।

ਆਓ ਵੇਖੀਏ ਕਿ MCP ਅਪਣਾਉਂਦੇ ਸਮੇਂ ਤੁਸੀਂ ਸੁਰੱਖਿਆ ਖਤਰਿਆਂ ਨੂੰ ਕਿਵੇਂ ਸੰਭਾਲ ਸਕਦੇ ਹੋ।

> **Note:** ਹੇਠਾਂ ਦਿੱਤੀ ਜਾਣਕਾਰੀ **29 ਮਈ 2025** ਤੱਕ ਸਹੀ ਹੈ। MCP ਪ੍ਰੋਟੋਕੋਲ ਲਗਾਤਾਰ ਵਿਕਸਤ ਹੋ ਰਿਹਾ ਹੈ, ਅਤੇ ਭਵਿੱਖ ਵਿੱਚ ਨਵੇਂ ਪ੍ਰਮਾਣਿਕਤਾ ਪੈਟਰਨ ਅਤੇ ਨਿਯੰਤਰਣ ਆ ਸਕਦੇ ਹਨ। ਤਾਜ਼ਾ ਜਾਣਕਾਰੀ ਅਤੇ ਮਾਰਗਦਰਸ਼ਨ ਲਈ ਹਮੇਸ਼ਾ [MCP Specification](https://spec.modelcontextprotocol.io/) ਅਤੇ ਅਧਿਕਾਰਕ [MCP GitHub repository](https://github.com/modelcontextprotocol) ਅਤੇ [security best practice page](https://modelcontextprotocol.io/specification/draft/basic/security_best_practices) ਨੂੰ ਵੇਖੋ।

### ਸਮੱਸਿਆ ਦਾ ਬਿਆਨ  
ਮੂਲ MCP ਵਿਸ਼ੇਸ਼ਤਾ ਇਹ ਮੰਨਦੀ ਸੀ ਕਿ ਡਿਵੈਲਪਰ ਆਪਣੇ ਪ੍ਰਮਾਣਿਕਤਾ ਸਰਵਰ ਨੂੰ ਖੁਦ ਲਿਖਣਗੇ। ਇਸ ਲਈ OAuth ਅਤੇ ਸੰਬੰਧਿਤ ਸੁਰੱਖਿਆ ਸੀਮਾਵਾਂ ਦੀ ਜਾਣਕਾਰੀ ਲੋੜੀਂਦੀ ਸੀ। MCP ਸਰਵਰ OAuth 2.0 Authorization Servers ਵਜੋਂ ਕੰਮ ਕਰਦੇ ਸਨ, ਜੋ ਸਿੱਧਾ ਯੂਜ਼ਰ ਪ੍ਰਮਾਣਿਕਤਾ ਸੰਭਾਲਦੇ ਸਨ, ਬਜਾਏ ਕਿਸੇ ਬਾਹਰੀ ਸੇਵਾ ਜਿਵੇਂ Microsoft Entra ID ਨੂੰ ਸੌਂਪਣ ਦੇ। **26 ਅਪ੍ਰੈਲ 2025** ਤੋਂ, MCP ਵਿਸ਼ੇਸ਼ਤਾ ਵਿੱਚ ਇੱਕ ਅੱਪਡੇਟ ਆਇਆ ਹੈ ਜੋ MCP ਸਰਵਰਾਂ ਨੂੰ ਯੂਜ਼ਰ ਪ੍ਰਮਾਣਿਕਤਾ ਬਾਹਰੀ ਸੇਵਾ ਨੂੰ ਸੌਂਪਣ ਦੀ ਆਗਿਆ ਦਿੰਦਾ ਹੈ।

### ਖਤਰੇ
- MCP ਸਰਵਰ ਵਿੱਚ ਗਲਤ ਤਰੀਕੇ ਨਾਲ ਅਧਿਕਾਰ ਲਾਜਿਕ ਲਾਗੂ ਹੋਣ ਨਾਲ ਸੰਵੇਦਨਸ਼ੀਲ ਡਾਟਾ ਦਾ ਪ੍ਰਕਾਸ਼ ਹੋ ਸਕਦਾ ਹੈ ਅਤੇ ਗਲਤ ਤਰੀਕੇ ਨਾਲ ਪਹੁੰਚ ਨਿਯੰਤਰਣ ਲਾਗੂ ਹੋ ਸਕਦੇ ਹਨ।
- ਸਥਾਨਕ MCP ਸਰਵਰ 'ਤੇ OAuth ਟੋਕਨ ਦੀ ਚੋਰੀ। ਜੇ ਟੋਕਨ ਚੋਰੀ ਹੋ ਜਾਵੇ, ਤਾਂ ਇਸਦਾ ਇਸਤੇਮਾਲ MCP ਸਰਵਰ ਦੀ ਨਕਲ ਕਰਨ ਅਤੇ ਉਸ ਸੇਵਾ ਤੋਂ ਸਰੋਤਾਂ ਅਤੇ ਡਾਟਾ ਤੱਕ ਪਹੁੰਚ ਲਈ ਕੀਤਾ ਜਾ ਸਕਦਾ ਹੈ ਜਿਸ ਲਈ ਟੋਕਨ ਜਾਰੀ ਕੀਤਾ ਗਿਆ ਸੀ।

#### Token Passthrough  
Authorization ਵਿਸ਼ੇਸ਼ਤਾ ਵਿੱਚ ਟੋਕਨ ਪਾਸਥਰੂ ਸਪਸ਼ਟ ਤੌਰ 'ਤੇ ਮਨਾਹੀ ਹੈ ਕਿਉਂਕਿ ਇਹ ਕਈ ਸੁਰੱਖਿਆ ਖਤਰਿਆਂ ਨੂੰ ਜਨਮ ਦਿੰਦਾ ਹੈ, ਜਿਵੇਂ:

#### ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਚੱਕਰਵਾਤ  
MCP ਸਰਵਰ ਜਾਂ ਡਾਊਨਸਟਰੀਮ APIs ਮਹੱਤਵਪੂਰਨ ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਜਿਵੇਂ ਕਿ ਰੇਟ ਲਿਮਿਟਿੰਗ, ਬੇਨਤੀ ਦੀ ਜਾਂਚ ਜਾਂ ਟ੍ਰੈਫਿਕ ਨਿਗਰਾਨੀ ਲਾਗੂ ਕਰ ਸਕਦੇ ਹਨ, ਜੋ ਟੋਕਨ ਦਰਸ਼ਕ ਜਾਂ ਹੋਰ ਪ੍ਰਮਾਣ ਪੱਤਰ ਸੀਮਾਵਾਂ 'ਤੇ ਨਿਰਭਰ ਕਰਦੇ ਹਨ। ਜੇ ਕਲਾਇੰਟ ਸਿੱਧਾ ਡਾਊਨਸਟਰੀਮ APIs ਨਾਲ ਟੋਕਨ ਪ੍ਰਾਪਤ ਕਰਕੇ ਅਤੇ ਵਰਤ ਕੇ MCP ਸਰਵਰ ਵੱਲੋਂ ਠੀਕ ਤਰੀਕੇ ਨਾਲ ਜਾਂਚ ਨਾ ਕਰਵਾਉਣ, ਜਾਂ ਇਹ ਯਕੀਨੀ ਨਾ ਬਣਾਉਣ ਕਿ ਟੋਕਨ ਸਹੀ ਸੇਵਾ ਲਈ ਜਾਰੀ ਕੀਤਾ ਗਿਆ ਹੈ, ਤਾਂ ਉਹ ਇਹ ਨਿਯੰਤਰਣ ਬਾਈਪਾਸ ਕਰ ਲੈਂਦੇ ਹਨ।

#### ਜਵਾਬਦੇਹੀ ਅਤੇ ਆਡਿਟ ਟ੍ਰੇਲ ਸਮੱਸਿਆਵਾਂ  
ਜਦੋਂ ਕਲਾਇੰਟ ਉੱਪਰਲੇ ਜਾਰੀ ਕੀਤੇ ਗਏ ਐਕਸੈਸ ਟੋਕਨ ਨਾਲ ਕਾਲ ਕਰਦੇ ਹਨ ਜੋ MCP ਸਰਵਰ ਲਈ ਅਸਪਸ਼ਟ ਹੋ ਸਕਦਾ ਹੈ, ਤਾਂ MCP ਸਰਵਰ MCP ਕਲਾਇੰਟਾਂ ਦੀ ਪਛਾਣ ਜਾਂ ਵੱਖ-ਵੱਖ ਕਰਨ ਵਿੱਚ ਅਸਮਰੱਥ ਰਹੇਗਾ।  
ਡਾਊਨਸਟਰੀਮ ਰਿਸੋਰਸ ਸਰਵਰ ਦੇ ਲੌਗ ਅਜਿਹੀਆਂ ਬੇਨਤੀਆਂ ਦਰਸਾ ਸਕਦੇ ਹਨ ਜੋ ਕਿਸੇ ਹੋਰ ਸਰੋਤ ਜਾਂ ਵੱਖਰੀ ਪਛਾਣ ਵਾਲੀ ਲੱਗਦੀਆਂ ਹਨ, ਨਾ ਕਿ MCP ਸਰਵਰ ਜੋ ਟੋਕਨ ਅੱਗੇ ਭੇਜ ਰਿਹਾ ਹੈ।  
ਇਹ ਦੋਹਾਂ ਕਾਰਕ ਘਟਨਾ ਜਾਂਚ, ਨਿਯੰਤਰਣ ਅਤੇ ਆਡਿਟਿੰਗ ਨੂੰ ਮੁਸ਼ਕਲ ਬਣਾਉਂਦੇ ਹਨ।  
ਜੇ MCP ਸਰਵਰ ਟੋਕਨਾਂ ਦੀਆਂ ਦਾਵਿਆਂ (ਜਿਵੇਂ ਕਿ ਭੂਮਿਕਾਵਾਂ, ਅਧਿਕਾਰ ਜਾਂ ਦਰਸ਼ਕ) ਜਾਂ ਹੋਰ ਮੈਟਾਡੇਟਾ ਦੀ ਜਾਂਚ ਕੀਤੇ ਬਿਨਾਂ ਟੋਕਨ ਪਾਸ ਕਰਦਾ ਹੈ, ਤਾਂ ਚੋਰੀ ਹੋਏ ਟੋਕਨ ਵਾਲਾ ਕੋਈ ਬਦਮਾਸ਼ ਸਰਵਰ ਨੂੰ ਡਾਟਾ ਚੋਰੀ ਲਈ ਪ੍ਰਾਕਸੀ ਵਜੋਂ ਵਰਤ ਸਕਦਾ ਹੈ।

#### ਭਰੋਸਾ ਸੀਮਾ ਸਮੱਸਿਆਵਾਂ  
ਡਾਊਨਸਟਰੀਮ ਰਿਸੋਰਸ ਸਰਵਰ ਖਾਸ ਇਕਾਈਆਂ ਨੂੰ ਭਰੋਸਾ ਦਿੰਦਾ ਹੈ। ਇਹ ਭਰੋਸਾ ਮੂਲ ਜਾਂ ਕਲਾਇੰਟ ਦੇ ਵਿਹਾਰ ਦੇ ਪੈਟਰਨ ਬਾਰੇ ਧਾਰਣਾਵਾਂ ਸ਼ਾਮਲ ਕਰ ਸਕਦਾ ਹੈ। ਇਸ ਭਰੋਸਾ ਸੀਮਾ ਨੂੰ ਤੋੜਨਾ ਅਣਚਾਹੀਆਂ ਸਮੱਸਿਆਵਾਂ ਦਾ ਕਾਰਨ ਬਣ ਸਕਦਾ ਹੈ।  
ਜੇ ਟੋਕਨ ਬਿਨਾਂ ਠੀਕ ਜਾਂਚ ਦੇ ਕਈ ਸੇਵਾਵਾਂ ਵੱਲੋਂ ਸਵੀਕਾਰ ਕੀਤਾ ਜਾਂਦਾ ਹੈ, ਤਾਂ ਇੱਕ ਸੇਵਾ ਨੂੰ ਕਮਪ੍ਰੋਮਾਈਜ਼ ਕਰਨ ਵਾਲਾ ਹਮਲਾਵਰ ਇਸ ਟੋਕਨ ਦੀ ਵਰਤੋਂ ਹੋਰ ਜੁੜੀਆਂ ਸੇਵਾਵਾਂ ਤੱਕ ਪਹੁੰਚ ਲਈ ਕਰ ਸਕਦਾ ਹੈ।

#### ਭਵਿੱਖੀ ਅਨੁਕੂਲਤਾ ਦਾ ਖਤਰਾ  
ਜੇ MCP ਸਰਵਰ ਅੱਜ "ਪਿਊਰ ਪ੍ਰਾਕਸੀ" ਵਜੋਂ ਸ਼ੁਰੂ ਹੁੰਦਾ ਹੈ, ਤਾਂ ਭਵਿੱਖ ਵਿੱਚ ਉਸਨੂੰ ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਸ਼ਾਮਲ ਕਰਨ ਦੀ ਲੋੜ ਪੈ ਸਕਦੀ ਹੈ। ਸਹੀ ਟੋਕਨ ਦਰਸ਼ਕ ਵੰਡ ਨਾਲ ਸ਼ੁਰੂਆਤ ਕਰਨ ਨਾਲ ਸੁਰੱਖਿਆ ਮਾਡਲ ਨੂੰ ਵਿਕਸਤ ਕਰਨਾ ਆਸਾਨ ਹੁੰਦਾ ਹੈ।

### ਘਟਾਉਣ ਵਾਲੇ ਨਿਯੰਤਰਣ

**MCP ਸਰਵਰ ਕਿਸੇ ਵੀ ਐਸੇ ਟੋਕਨ ਨੂੰ ਸਵੀਕਾਰ ਨਹੀਂ ਕਰ ਸਕਦਾ ਜੋ ਸਪਸ਼ਟ ਤੌਰ 'ਤੇ MCP ਸਰਵਰ ਲਈ ਜਾਰੀ ਨਾ ਕੀਤਾ ਗਿਆ ਹੋਵੇ।**

- **ਅਧਿਕਾਰ ਲਾਜਿਕ ਦੀ ਸਮੀਖਿਆ ਅਤੇ ਮਜ਼ਬੂਤੀ:** ਆਪਣੇ MCP ਸਰਵਰ ਦੀ ਅਧਿਕਾਰ ਲਾਗੂਆਤ ਦੀ ਧਿਆਨ ਨਾਲ ਜਾਂਚ ਕਰੋ ਤਾਂ ਜੋ ਸਿਰਫ਼ ਮਨਜ਼ੂਰਸ਼ੁਦਾ ਯੂਜ਼ਰ ਅਤੇ ਕਲਾਇੰਟ ਸੰਵੇਦਨਸ਼ੀਲ ਸਰੋਤਾਂ ਤੱਕ ਪਹੁੰਚ ਸਕਣ। ਵਿਹਾਰਕ ਮਾਰਗਦਰਸ਼ਨ ਲਈ ਵੇਖੋ [Azure API Management Your Auth Gateway For MCP Servers | Microsoft Community Hub](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690) ਅਤੇ [Using Microsoft Entra ID To Authenticate With MCP Servers Via Sessions - Den Delimarsky](https://den.dev/blog/mcp-server-auth-entra-id-session/)।
- **ਸੁਰੱਖਿਅਤ ਟੋਕਨ ਅਮਲਾਂ ਨੂੰ ਲਾਗੂ ਕਰੋ:** [Microsoft ਦੇ ਟੋਕਨ ਵੈਰੀਫਿਕੇਸ਼ਨ ਅਤੇ ਲਾਈਫਟਾਈਮ ਲਈ ਵਧੀਆ ਅਮਲ](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens) ਦੀ ਪਾਲਣਾ ਕਰੋ ਤਾਂ ਜੋ ਟੋਕਨਾਂ ਦੀ ਗਲਤ ਵਰਤੋਂ ਅਤੇ ਟੋਕਨ ਰੀਪਲੇ ਜਾਂ ਚੋਰੀ ਦੇ ਖਤਰੇ ਨੂੰ ਘਟਾਇਆ ਜਾ ਸਕੇ।
- **ਟੋਕਨ ਸਟੋਰੇਜ ਦੀ ਸੁਰੱਖਿਆ ਕਰੋ:** ਹਮੇਸ਼ਾ ਟੋਕਨਾਂ ਨੂੰ ਸੁਰੱਖਿਅਤ ਢੰਗ ਨਾਲ ਸਟੋਰ ਕਰੋ ਅਤੇ ਉਨ੍ਹਾਂ ਨੂੰ ਰੱਖਣ ਅਤੇ ਟ੍ਰਾਂਜ਼ਿਟ ਦੌਰਾਨ ਇਨਕ੍ਰਿਪਸ਼ਨ ਨਾਲ ਸੁਰੱਖਿਅਤ ਕਰੋ। ਲਾਗੂ ਕਰਨ ਲਈ ਸੁਝਾਅ ਵੇਖੋ [Use secure token storage and encrypt tokens](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2)।

# MCP ਸਰਵਰਾਂ ਲਈ ਜ਼ਿਆਦਾ ਅਧਿਕਾਰ

### ਸਮੱਸਿਆ ਦਾ ਬਿਆਨ  
MCP ਸਰਵਰਾਂ ਨੂੰ ਉਹਨਾਂ ਸੇਵਾ/ਸਰੋਤਾਂ ਲਈ ਜ਼ਿਆਦਾ ਅਧਿਕਾਰ ਦਿੱਤੇ ਗਏ ਹੋ ਸਕਦੇ ਹਨ ਜਿਨ੍ਹਾਂ ਤੱਕ ਉਹ ਪਹੁੰਚ ਰਹੇ ਹਨ। ਉਦਾਹਰਨ ਵਜੋਂ, ਇੱਕ AI ਸੇਲਜ਼ ਐਪਲੀਕੇਸ਼ਨ ਦਾ MCP ਸਰਵਰ ਜੋ ਇੱਕ ਉਦਯੋਗਿਕ ਡਾਟਾ ਸਟੋਰ ਨਾਲ ਜੁੜਿਆ ਹੈ, ਉਸਨੂੰ ਸਿਰਫ ਸੇਲਜ਼ ਡਾਟਾ ਤੱਕ ਪਹੁੰਚ ਹੋਣੀ ਚਾਹੀਦੀ ਹੈ, ਨਾ ਕਿ ਸਟੋਰ ਵਿੱਚ ਸਾਰੇ ਫਾਇਲਾਂ ਤੱਕ। ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ ਦੇ ਸਿਧਾਂਤ (ਜੋ ਸਭ ਤੋਂ ਪੁਰਾਣਾ ਸੁਰੱਖਿਆ ਸਿਧਾਂਤ ਹੈ) ਮੁਤਾਬਕ, ਕਿਸੇ ਵੀ ਸਰੋਤ ਨੂੰ ਉਹਨਾਂ ਕੰਮਾਂ ਲਈ ਜ਼ਿਆਦਾ ਅਧਿਕਾਰ ਨਹੀਂ ਮਿਲਣੇ ਚਾਹੀਦੇ ਜੋ ਉਹ ਕਰਨਾ ਹੈ। AI ਇਸ ਖੇਤਰ ਵਿੱਚ ਵਧੇਰੇ ਚੁਣੌਤੀ ਪੈਦਾ ਕਰਦਾ ਹੈ ਕਿਉਂਕਿ ਇਸਨੂੰ ਲਚਕੀਲਾ ਬਣਾਉਣ ਲਈ ਸਹੀ ਅਧਿਕਾਰਾਂ ਦੀ ਪਰਿਭਾਸ਼ਾ ਕਰਨਾ ਮੁਸ਼ਕਲ ਹੁੰਦਾ ਹੈ।

### ਖਤਰੇ  
- ਜ਼ਿਆਦਾ ਅਧਿਕਾਰ ਦੇਣ ਨਾਲ MCP ਸਰਵਰ ਨੂੰ ਐਸਾ ਡਾਟਾ ਬਾਹਰ ਕੱਢਣ ਜਾਂ ਸੋਧਣ ਦੀ ਆਗਿਆ ਮਿਲ ਸਕਦੀ ਹੈ ਜਿਸ ਤੱਕ ਉਸਨੂੰ ਪਹੁੰਚ ਨਹੀਂ ਹੋਣੀ ਚਾਹੀਦੀ। ਜੇ ਡਾਟਾ ਵਿਅਕਤੀਗਤ ਪਛਾਣਯੋਗ ਜਾਣਕਾਰੀ (PII) ਹੈ, ਤਾਂ ਇਹ ਪ੍ਰਾਈਵੇਸੀ ਸਮੱਸਿਆ ਵੀ ਬਣ ਸਕਦੀ ਹੈ।

### ਘਟਾਉਣ ਵਾਲੇ ਨਿਯੰਤਰਣ  
- **ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ ਦਾ ਸਿਧਾਂਤ ਲਾਗੂ ਕਰੋ:** MCP ਸਰਵਰ ਨੂੰ ਸਿਰਫ਼ ਉਹਨਾਂ ਕੰਮਾਂ ਲਈ ਲੋੜੀਂਦੇ ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ ਦਿਓ। ਇਹਨਾਂ ਅਧਿਕਾਰਾਂ ਦੀ ਨਿਯਮਤ ਸਮੀਖਿਆ ਅਤੇ ਅਪਡੇਟ ਕਰੋ ਤਾਂ ਜੋ ਇਹ ਜ਼ਰੂਰਤ ਤੋਂ ਵੱਧ ਨਾ ਹੋਣ। ਵਿਸਥਾਰ ਲਈ ਵੇਖੋ [Secure least-privileged access](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access)।
- **ਰੋਲ-ਅਧਾਰਿਤ ਪਹੁੰਚ ਨਿਯੰਤਰਣ (RBAC) ਦੀ ਵਰਤੋਂ ਕਰੋ:** MCP ਸਰਵਰ ਨੂੰ ਐਸੇ ਰੋਲ ਸੌਂਪੋ ਜੋ ਖਾਸ ਸਰੋਤਾਂ ਅਤੇ ਕਾਰਵਾਈਆਂ ਲਈ ਸੀਮਿਤ ਹੋਣ, ਅਤੇ ਵਿਆਪਕ ਜਾਂ ਬੇਕਾਰ ਅਧਿਕਾਰਾਂ ਤੋਂ ਬਚੋ।
- **ਅਧਿਕਾਰਾਂ ਦੀ ਨਿਗਰਾਨੀ ਅਤੇ ਆਡਿਟ ਕਰੋ:** ਅਧਿਕਾਰਾਂ ਦੀ ਵਰਤੋਂ ਨੂੰ ਲਗਾਤਾਰ ਨਿਗਰਾਨੀ ਕਰੋ ਅਤੇ ਪਹੁੰਚ ਲੌਗਾਂ ਦੀ ਆਡਿਟਿੰਗ ਕਰਕੇ ਜ਼ਿਆਦਾ ਜਾਂ ਬੇਕਾਰ ਅਧਿਕਾਰਾਂ ਨੂੰ ਜਲਦੀ ਪਛਾਣੋ ਅਤੇ ਠੀਕ ਕਰੋ।

# ਅਪਰੋਕਸ਼ ਪ੍ਰਾਂਪਟ ਇੰਜੈਕਸ਼ਨ ਹਮਲੇ

### ਸਮੱਸਿਆ ਦਾ ਬਿਆਨ
ਸਪਲਾਈ ਚੇਨ ਸੁਰੱਖਿਆ AI ਯੁੱਗ ਵਿੱਚ ਬੁਨਿਆਦੀ ਰਹਿੰਦੀ ਹੈ, ਪਰ ਤੁਹਾਡੇ ਸਪਲਾਈ ਚੇਨ ਦੀ ਪਰਿਭਾਸ਼ਾ ਹੁਣ ਵਧ ਗਈ ਹੈ। ਰਵਾਇਤੀ ਕੋਡ ਪੈਕੇਜਾਂ ਦੇ ਨਾਲ-ਨਾਲ, ਹੁਣ ਤੁਹਾਨੂੰ ਸਾਰੇ AI-ਸੰਬੰਧਿਤ ਹਿੱਸਿਆਂ ਦੀ ਕੜੀ ਤਰ੍ਹਾਂ ਜਾਂਚ ਅਤੇ ਨਿਗਰਾਨੀ ਕਰਨੀ ਚਾਹੀਦੀ ਹੈ, ਜਿਸ ਵਿੱਚ ਫਾਊਂਡੇਸ਼ਨ ਮਾਡਲ, ਐਂਬੈਡਿੰਗ ਸੇਵਾਵਾਂ, ਸੰਦਰਭ ਪ੍ਰਦਾਤਾ ਅਤੇ ਤੀਜੀ ਪੱਖੀ APIs ਸ਼ਾਮਲ ਹਨ। ਜੇ ਇਹਨਾਂ ਨੂੰ ਠੀਕ ਤਰੀਕੇ ਨਾਲ ਪ੍ਰਬੰਧਿਤ ਨਾ ਕੀਤਾ ਗਿਆ ਤਾਂ ਇਹ ਕਮਜ਼ੋਰੀਆਂ ਜਾਂ ਖਤਰੇ ਲਿਆ ਸਕਦੇ ਹਨ।

**AI ਅਤੇ MCP ਲਈ ਮੁੱਖ ਸਪਲਾਈ ਚੇਨ ਸੁਰੱਖਿਆ ਅਭਿਆਸ:**
- **ਇੰਟੀਗ੍ਰੇਸ਼ਨ ਤੋਂ ਪਹਿਲਾਂ ਸਾਰੇ ਹਿੱਸਿਆਂ ਦੀ ਜਾਂਚ ਕਰੋ:** ਇਸ ਵਿੱਚ ਸਿਰਫ ਖੁੱਲ੍ਹੇ ਸਰੋਤ ਲਾਇਬ੍ਰੇਰੀਆਂ ਹੀ ਨਹੀਂ, ਸਗੋਂ AI ਮਾਡਲ, ਡਾਟਾ ਸਰੋਤ ਅਤੇ ਬਾਹਰੀ APIs ਵੀ ਸ਼ਾਮਲ ਹਨ। ਹਮੇਸ਼ਾ ਮੂਲ, ਲਾਇਸੈਂਸਿੰਗ ਅਤੇ ਜਾਣੇ-ਪਹਚਾਣੇ ਕਮਜ਼ੋਰੀਆਂ ਦੀ ਜਾਂਚ ਕਰੋ।
- **ਸੁਰੱਖਿਅਤ ਡਿਪਲੋਇਮੈਂਟ ਪਾਈਪਲਾਈਨਾਂ ਬਣਾਓ:** ਆਟੋਮੇਟਿਡ CI/CD ਪਾਈਪਲਾਈਨਾਂ ਵਰਤੋ ਜਿਨ੍ਹਾਂ ਵਿੱਚ ਸੁਰੱਖਿਆ ਸਕੈਨਿੰਗ ਸ਼ਾਮਲ ਹੋਵੇ ਤਾਂ ਜੋ ਸਮੱਸਿਆਵਾਂ ਨੂੰ ਜਲਦੀ ਪਤਾ ਲੱਗ ਸਕੇ। ਯਕੀਨੀ ਬਣਾਓ ਕਿ ਸਿਰਫ ਭਰੋਸੇਯੋਗ ਆਰਟੀਫੈਕਟ ਪ੍ਰੋਡਕਸ਼ਨ ਵਿੱਚ ਡਿਪਲੋਇ ਕੀਤੇ ਜਾਣ।
- **ਲਗਾਤਾਰ ਨਿਗਰਾਨੀ ਅਤੇ ਆਡਿਟ ਕਰੋ:** ਸਾਰੇ ਡਿਪੈਂਡੈਂਸੀਜ਼, ਮਾਡਲਾਂ ਅਤੇ ਡਾਟਾ ਸੇਵਾਵਾਂ ਦੀ ਲਗਾਤਾਰ ਨਿਗਰਾਨੀ ਕਰੋ ਤਾਂ ਜੋ ਨਵੀਆਂ ਕਮਜ਼ੋਰੀਆਂ ਜਾਂ ਸਪਲਾਈ ਚੇਨ ਹਮਲਿਆਂ ਦਾ ਪਤਾ ਲੱਗ ਸਕੇ।
- **ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ ਅਤੇ ਪਹੁੰਚ ਨਿਯੰਤਰਣ ਲਾਗੂ ਕਰੋ:** ਮਾਡਲਾਂ, ਡਾਟਾ ਅਤੇ ਸੇਵਾਵਾਂ ਤੱਕ ਪਹੁੰਚ ਸਿਰਫ ਉਸੇ ਹੱਦ ਤੱਕ ਸੀਮਿਤ ਕਰੋ ਜੋ ਤੁਹਾਡੇ MCP ਸਰਵਰ ਲਈ ਜ਼ਰੂਰੀ ਹੈ।
- **ਖਤਰਿਆਂ ਦਾ ਜਲਦੀ ਜਵਾਬ ਦਿਓ:** ਜੇ ਕੋਈ ਹਿੱਸਾ ਖਰਾਬ ਹੋ ਜਾਵੇ ਤਾਂ ਉਸਨੂੰ ਪੈਚ ਕਰਨ ਜਾਂ ਬਦਲਣ ਲਈ ਪ੍ਰਕਿਰਿਆ ਬਣਾਓ, ਅਤੇ ਜੇ ਸੁਰੱਖਿਆ ਭੰਗ ਹੋਵੇ ਤਾਂ ਸਿਕ੍ਰੇਟ ਜਾਂ ਪ੍ਰਮਾਣ ਪੱਤਰਾਂ ਨੂੰ ਰੋਟੇਟ ਕਰੋ।

[GitHub Advanced Security](https://github.com/security/advanced-security) ਸਿਕ੍ਰੇਟ ਸਕੈਨਿੰਗ, ਡਿਪੈਂਡੈਂਸੀ ਸਕੈਨਿੰਗ ਅਤੇ CodeQL ਵਿਸ਼ਲੇਸ਼ਣ ਵਰਗੀਆਂ ਵਿਸ਼ੇਸ਼ਤਾਵਾਂ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ। ਇਹ ਟੂਲ [Azure DevOps](https://azure.microsoft.com/en-us/products/devops) ਅਤੇ [Azure Repos](https://azure.microsoft.com/en-us/products/devops/repos/) ਨਾਲ ਇੰਟੀਗ੍ਰੇਟ ਹੁੰਦੇ ਹਨ ਤਾਂ ਜੋ ਟੀਮਾਂ ਕੋਡ ਅਤੇ AI ਸਪਲਾਈ ਚੇਨ ਹਿੱਸਿਆਂ ਵਿੱਚ ਕਮਜ਼ੋਰੀਆਂ ਦੀ ਪਹਿਚਾਣ ਅਤੇ ਨਿਵਾਰਨ ਕਰ ਸਕਣ।

ਮਾਈਕ੍ਰੋਸਾਫਟ ਆਪਣੇ ਸਾਰੇ ਉਤਪਾਦਾਂ ਲਈ ਅੰਦਰੂਨੀ ਤੌਰ 'ਤੇ ਵੀ ਵਿਆਪਕ ਸਪਲਾਈ ਚੇਨ ਸੁਰੱਖਿਆ ਅਭਿਆਸ ਲਾਗੂ ਕਰਦਾ ਹੈ। ਹੋਰ ਜਾਣਕਾਰੀ ਲਈ ਵੇਖੋ [The Journey to Secure the Software Supply Chain at Microsoft](https://devblogs.microsoft.com/engineering-at-microsoft/the-journey-to-secure-the-software-supply-chain-at-microsoft/)।

# ਸਥਾਪਿਤ ਸੁਰੱਖਿਆ ਦੇ ਸਭ ਤੋਂ ਵਧੀਆ ਅਭਿਆਸ ਜੋ ਤੁਹਾਡੇ MCP ਇੰਪਲੀਮੈਂਟੇਸ਼ਨ ਦੀ ਸੁਰੱਖਿਆ ਸਥਿਤੀ ਨੂੰ ਮਜ਼ਬੂਤ ਕਰਨਗੇ

ਕੋਈ ਵੀ MCP ਇੰਪਲੀਮੈਂਟੇਸ਼ਨ ਉਸ ਸੰਗਠਨ ਦੇ ਮੌਜੂਦਾ ਸੁਰੱਖਿਆ ਮਾਹੌਲ ਦੀ ਸਥਿਤੀ ਨੂੰ ਵਿਰਾਸਤ ਵਿੱਚ ਲੈਂਦਾ ਹੈ ਜਿਸ 'ਤੇ ਇਹ ਬਣਾਇਆ ਗਿਆ ਹੈ, ਇਸ ਲਈ ਜਦੋਂ ਤੁਸੀਂ MCP ਨੂੰ ਆਪਣੇ ਕੁੱਲ AI ਸਿਸਟਮਾਂ ਦੇ ਹਿੱਸੇ ਵਜੋਂ ਦੇਖਦੇ ਹੋ ਤਾਂ ਇਹ ਸਿਫਾਰਸ਼ ਕੀਤੀ ਜਾਂਦੀ ਹੈ ਕਿ ਤੁਸੀਂ ਆਪਣੀ ਮੌਜੂਦਾ ਸੁਰੱਖਿਆ ਸਥਿਤੀ ਨੂੰ ਮਜ਼ਬੂਤ ਕਰੋ। ਹੇਠਾਂ ਦਿੱਤੇ ਸਥਾਪਿਤ ਸੁਰੱਖਿਆ ਨਿਯੰਤਰਣ ਖਾਸ ਤੌਰ 'ਤੇ ਮਹੱਤਵਪੂਰਨ ਹਨ:

-   ਤੁਹਾਡੇ AI ਐਪਲੀਕੇਸ਼ਨ ਵਿੱਚ ਸੁਰੱਖਿਅਤ ਕੋਡਿੰਗ ਦੇ ਸਭ ਤੋਂ ਵਧੀਆ ਅਭਿਆਸ - [OWASP Top 10](https://owasp.org/www-project-top-ten/) ਅਤੇ [OWASP Top 10 for LLMs](https://genai.owasp.org/download/43299/?tmstv=1731900559) ਤੋਂ ਬਚਾਅ, ਸਿਕ੍ਰੇਟ ਅਤੇ ਟੋਕਨ ਲਈ ਸੁਰੱਖਿਅਤ ਵੌਲਟਾਂ ਦੀ ਵਰਤੋਂ, ਸਾਰੇ ਐਪਲੀਕੇਸ਼ਨ ਹਿੱਸਿਆਂ ਵਿਚਕਾਰ ਅੰਤ-ਤੱਕ ਸੁਰੱਖਿਅਤ ਸੰਚਾਰ ਲਾਗੂ ਕਰਨਾ ਆਦਿ।
-   ਸਰਵਰ ਨੂੰ ਮਜ਼ਬੂਤ ਬਣਾਉਣਾ -- ਜਿੱਥੇ ਸੰਭਵ ਹੋ MFA ਵਰਤੋਂ, ਪੈਚਿੰਗ ਨੂੰ ਅਪ-ਟੂ-ਡੇਟ ਰੱਖੋ, ਪਹੁੰਚ ਲਈ ਤੀਜੀ ਪੱਖੀ ਆਈਡੈਂਟੀਟੀ ਪ੍ਰਦਾਤਾ ਨਾਲ ਸਰਵਰ ਨੂੰ ਇੰਟੀਗ੍ਰੇਟ ਕਰੋ ਆਦਿ।
-   ਡਿਵਾਈਸ, ਇੰਫਰਾਸਟਰੱਕਚਰ ਅਤੇ ਐਪਲੀਕੇਸ਼ਨਾਂ ਨੂੰ ਪੈਚਾਂ ਨਾਲ ਅਪ-ਟੂ-ਡੇਟ ਰੱਖੋ।
-   ਸੁਰੱਖਿਆ ਨਿਗਰਾਨੀ -- AI ਐਪਲੀਕੇਸ਼ਨ (ਜਿਸ ਵਿੱਚ MCP ਕਲਾਇੰਟ/ਸਰਵਰ ਵੀ ਸ਼ਾਮਲ ਹਨ) ਦੀ ਲੌਗਿੰਗ ਅਤੇ ਨਿਗਰਾਨੀ ਲਾਗੂ ਕਰੋ ਅਤੇ ਉਹਨਾਂ ਲੌਗਾਂ ਨੂੰ ਕੇਂਦਰੀ SIEM ਨੂੰ ਭੇਜੋ ਤਾਂ ਜੋ ਅਸਧਾਰਣ ਗਤੀਵਿਧੀਆਂ ਦਾ ਪਤਾ ਲੱਗ ਸਕੇ।
-   ਜ਼ੀਰੋ ਟਰੱਸਟ ਆਰਕੀਟੈਕਚਰ -- ਨੈੱਟਵਰਕ ਅਤੇ ਆਈਡੈਂਟੀਟੀ ਨਿਯੰਤਰਣਾਂ ਰਾਹੀਂ ਹਿੱਸਿਆਂ ਨੂੰ ਤਰਕਸੰਗਤ ਢੰਗ ਨਾਲ ਅਲੱਗ ਕਰਨਾ ਤਾਂ ਜੋ ਜੇ ਕੋਈ AI ਐਪਲੀਕੇਸ਼ਨ ਖਰਾਬ ਹੋ ਜਾਵੇ ਤਾਂ ਲੈਟਰਲ ਮੂਵਮੈਂਟ ਘੱਟ ਤੋਂ ਘੱਟ ਹੋਵੇ।

# ਮੁੱਖ ਨਤੀਜੇ

- ਸੁਰੱਖਿਆ ਦੇ ਮੂਲ ਤੱਤ ਅਹੰਕਾਰਪੂਰਕ ਹਨ: ਸੁਰੱਖਿਅਤ ਕੋਡਿੰਗ, ਘੱਟੋ-ਘੱਟ ਅਧਿਕਾਰ, ਸਪਲਾਈ ਚੇਨ ਦੀ ਜਾਂਚ ਅਤੇ ਲਗਾਤਾਰ ਨਿਗਰਾਨੀ MCP ਅਤੇ AI ਵਰਕਲੋਡ ਲਈ ਜ਼ਰੂਰੀ ਹਨ।
- MCP ਨਵੇਂ ਖਤਰਿਆਂ ਨੂੰ ਲਿਆਉਂਦਾ ਹੈ—ਜਿਵੇਂ ਕਿ ਪ੍ਰਾਂਪਟ ਇੰਜੈਕਸ਼ਨ, ਟੂਲ ਪੋਇਜ਼ਨਿੰਗ ਅਤੇ ਬੇਹੱਦ ਅਧਿਕਾਰ—ਜਿਨ੍ਹਾਂ ਲਈ ਰਵਾਇਤੀ ਅਤੇ AI-ਖਾਸ ਨਿਯੰਤਰਣ ਦੋਹਾਂ ਦੀ ਲੋੜ ਹੈ।
- ਮਜ਼ਬੂਤ ਪ੍ਰਮਾਣਿਕਤਾ, ਅਧਿਕਾਰ ਅਤੇ ਟੋਕਨ ਪ੍ਰਬੰਧਨ ਅਭਿਆਸ ਵਰਤੋਂ, ਜਿੱਥੇ ਸੰਭਵ ਹੋ Microsoft Entra ID ਵਰਗੇ ਬਾਹਰੀ ਆਈਡੈਂਟੀਟੀ ਪ੍ਰਦਾਤਾ ਦੀ ਵਰਤੋਂ ਕਰੋ।
- ਟੂਲ ਮੈਟਾਡੇਟਾ ਦੀ ਜਾਂਚ ਕਰਕੇ, ਗਤੀਸ਼ੀਲ ਬਦਲਾਵਾਂ ਦੀ ਨਿਗਰਾਨੀ ਕਰਕੇ ਅਤੇ Microsoft Prompt Shields ਵਰਗੇ ਹੱਲਾਂ ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਅਪਰੋਕਸੀਮੈਟ ਪ੍ਰਾਂਪਟ ਇੰਜੈਕਸ਼ਨ ਅਤੇ ਟੂਲ ਪੋਇਜ਼ਨਿੰਗ ਤੋਂ ਬਚਾਅ ਕਰੋ।
- ਆਪਣੇ AI ਸਪਲਾਈ ਚੇਨ ਵਿੱਚ ਸਾਰੇ ਹਿੱਸਿਆਂ—ਮਾਡਲ, ਐਂਬੈਡਿੰਗ ਅਤੇ ਸੰਦਰਭ ਪ੍ਰਦਾਤਾ ਸਮੇਤ—ਨੂੰ ਕੋਡ ਡਿਪੈਂਡੈਂਸੀਜ਼ ਵਾਂਗ ਹੀ ਸਖਤੀ ਨਾਲ ਸੰਭਾਲੋ।
- MCP ਵਿਸ਼ੇਸ਼ਤਾਵਾਂ ਵਿੱਚ ਹੋ ਰਹੇ ਬਦਲਾਵਾਂ ਨਾਲ ਅਪ-ਟੂ-ਡੇਟ ਰਹੋ ਅਤੇ ਸੁਰੱਖਿਅਤ ਮਿਆਰ ਬਣਾਉਣ ਲਈ ਕਮਿਊਨਿਟੀ ਵਿੱਚ ਯੋਗਦਾਨ ਪਾਓ।
