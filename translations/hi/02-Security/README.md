# Security Best Practices

Model Context Protocol (MCP) को अपनाने से AI-चालित अनुप्रयोगों में शक्तिशाली नई क्षमताएँ आती हैं, लेकिन इसके साथ ही पारंपरिक सॉफ़्टवेयर खतरों से अलग कुछ विशिष्ट सुरक्षा चुनौतियाँ भी सामने आती हैं। सुरक्षित कोडिंग, न्यूनतम विशेषाधिकार, और सप्लाई चेन सुरक्षा जैसे स्थापित मुद्दों के अलावा, MCP और AI वर्कलोड्स को नए खतरों का सामना करना पड़ता है जैसे कि प्रॉम्प्ट इंजेक्शन, टूल पॉइज़निंग, और डायनामिक टूल संशोधन। यदि इन्हें सही तरीके से प्रबंधित न किया जाए तो ये खतरे डेटा चोरी, गोपनीयता उल्लंघन, और अनचाहे सिस्टम व्यवहार का कारण बन सकते हैं।

यह पाठ MCP से जुड़े सबसे महत्वपूर्ण सुरक्षा जोखिमों—जैसे प्रमाणीकरण, प्राधिकरण, अत्यधिक अनुमतियाँ, अप्रत्यक्ष प्रॉम्प्ट इंजेक्शन, और सप्लाई चेन कमजोरियाँ—की समीक्षा करता है और इन्हें कम करने के लिए व्यावहारिक नियंत्रण और सर्वोत्तम प्रथाएँ प्रदान करता है। आप Microsoft के समाधान जैसे Prompt Shields, Azure Content Safety, और GitHub Advanced Security का उपयोग करके अपनी MCP कार्यान्वयन को मजबूत करना भी सीखेंगे। इन नियंत्रणों को समझकर और लागू करके, आप सुरक्षा उल्लंघनों की संभावना को काफी हद तक कम कर सकते हैं और अपने AI सिस्टम को मजबूत और भरोसेमंद बना सकते हैं।

# Learning Objectives

इस पाठ के अंत तक, आप सक्षम होंगे:

- Model Context Protocol (MCP) से जुड़े विशिष्ट सुरक्षा जोखिमों की पहचान और व्याख्या करना, जिनमें प्रॉम्प्ट इंजेक्शन, टूल पॉइज़निंग, अत्यधिक अनुमतियाँ, और सप्लाई चेन कमजोरियाँ शामिल हैं।
- MCP सुरक्षा जोखिमों के लिए प्रभावी निवारक नियंत्रणों को समझना और लागू करना, जैसे मजबूत प्रमाणीकरण, न्यूनतम विशेषाधिकार, सुरक्षित टोकन प्रबंधन, और सप्लाई चेन सत्यापन।
- Microsoft के समाधान जैसे Prompt Shields, Azure Content Safety, और GitHub Advanced Security का उपयोग करके MCP और AI वर्कलोड्स की सुरक्षा करना।
- टूल मेटाडेटा की सत्यापन, डायनामिक बदलावों की निगरानी, और अप्रत्यक्ष प्रॉम्प्ट इंजेक्शन हमलों से बचाव के महत्व को समझना।
- स्थापित सुरक्षा सर्वोत्तम प्रथाओं—जैसे सुरक्षित कोडिंग, सर्वर हार्डनिंग, और ज़ीरो ट्रस्ट आर्किटेक्चर—को अपनी MCP कार्यान्वयन में शामिल करके सुरक्षा उल्लंघनों की संभावना और प्रभाव को कम करना।

# MCP security controls

किसी भी सिस्टम को महत्वपूर्ण संसाधनों तक पहुँच होने पर सुरक्षा चुनौतियाँ होती हैं। इन चुनौतियों का समाधान आमतौर पर मौलिक सुरक्षा नियंत्रणों और अवधारणाओं को सही ढंग से लागू करके किया जाता है। चूंकि MCP अभी नया परिभाषित हुआ है, इसकी विशिष्टता तेजी से बदल रही है और जैसे-जैसे प्रोटोकॉल विकसित होगा, इसमें सुरक्षा नियंत्रण भी परिपक्व होंगे, जिससे इसे एंटरप्राइज और स्थापित सुरक्षा आर्किटेक्चर एवं सर्वोत्तम प्रथाओं के साथ बेहतर एकीकरण मिलेगा।

[Microsoft Digital Defense Report](https://aka.ms/mddr) में प्रकाशित शोध के अनुसार, रिपोर्ट किए गए 98% उल्लंघनों को मजबूत सुरक्षा स्वच्छता द्वारा रोका जा सकता है, और किसी भी प्रकार के उल्लंघन के खिलाफ सर्वोत्तम सुरक्षा आपके बेसलाइन सुरक्षा स्वच्छता, सुरक्षित कोडिंग सर्वोत्तम प्रथाओं और सप्लाई चेन सुरक्षा को सही तरीके से लागू करना है—वे आजमाए हुए उपाय जो सुरक्षा जोखिम को कम करने में सबसे प्रभावी साबित होते हैं।

आइए देखें कि MCP को अपनाते समय आप सुरक्षा जोखिमों को कैसे संबोधित करना शुरू कर सकते हैं।

# MCP server authentication (if your MCP implementation was before 26th April 2025)

> **Note:** निम्न जानकारी 26 अप्रैल 2025 तक सही है। MCP प्रोटोकॉल लगातार विकसित हो रहा है, और भविष्य के कार्यान्वयन में नए प्रमाणीकरण पैटर्न और नियंत्रण आ सकते हैं। नवीनतम अपडेट और मार्गदर्शन के लिए हमेशा [MCP Specification](https://spec.modelcontextprotocol.io/) और आधिकारिक [MCP GitHub repository](https://github.com/modelcontextprotocol) देखें।

### समस्या विवरण  
मूल MCP विशिष्टता यह मानती थी कि डेवलपर्स अपना प्रमाणीकरण सर्वर खुद बनाएंगे। इसके लिए OAuth और संबंधित सुरक्षा प्रतिबंधों का ज्ञान आवश्यक था। MCP सर्वर OAuth 2.0 Authorization Servers के रूप में कार्य करते थे, सीधे उपयोगकर्ता प्रमाणीकरण प्रबंधित करते थे बजाय इसे Microsoft Entra ID जैसे बाहरी सेवा को सौंपने के। 26 अप्रैल 2025 से, MCP विशिष्टता के अपडेट के तहत MCP सर्वर अब उपयोगकर्ता प्रमाणीकरण को बाहरी सेवा को सौंप सकते हैं।

### जोखिम
- MCP सर्वर में गलत कॉन्फ़िगर की गई प्राधिकरण लॉजिक संवेदनशील डेटा के खुलासे और गलत एक्सेस नियंत्रण का कारण बन सकती है।
- स्थानीय MCP सर्वर पर OAuth टोकन चोरी। यदि टोकन चोरी हो जाता है, तो इसका उपयोग MCP सर्वर का नकल करने और उस सेवा के संसाधनों तथा डेटा तक पहुँचने के लिए किया जा सकता है जिसके लिए OAuth टोकन है।

### निवारक नियंत्रण
- **प्राधिकरण लॉजिक की समीक्षा और सख्ती करें:** अपने MCP सर्वर के प्राधिकरण कार्यान्वयन का सावधानीपूर्वक ऑडिट करें ताकि केवल इच्छित उपयोगकर्ता और क्लाइंट ही संवेदनशील संसाधनों तक पहुँच सकें। व्यावहारिक मार्गदर्शन के लिए देखें [Azure API Management Your Auth Gateway For MCP Servers | Microsoft Community Hub](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690) और [Using Microsoft Entra ID To Authenticate With MCP Servers Via Sessions - Den Delimarsky](https://den.dev/blog/mcp-server-auth-entra-id-session/)।
- **सुरक्षित टोकन प्रथाओं को लागू करें:** [Microsoft के टोकन सत्यापन और जीवनकाल के सर्वोत्तम अभ्यास](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens) का पालन करें ताकि एक्सेस टोकन के दुरुपयोग और टोकन रिप्ले या चोरी का खतरा कम हो।
- **टोकन स्टोरेज की सुरक्षा करें:** हमेशा टोकन को सुरक्षित रूप से स्टोर करें और उन्हें आराम और ट्रांसमिशन दोनों में एन्क्रिप्ट करें। कार्यान्वयन सुझावों के लिए देखें [Use secure token storage and encrypt tokens](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2)।

# Excessive permissions for MCP servers

### समस्या विवरण  
MCP सर्वरों को जिस सेवा/संसाधन तक पहुँच मिली है, उसके लिए अत्यधिक अनुमतियाँ दी गई हो सकती हैं। उदाहरण के लिए, एक AI सेल्स एप्लिकेशन का MCP सर्वर जो एंटरप्राइज डेटा स्टोर से जुड़ा है, उसे केवल सेल्स डेटा तक सीमित पहुँच होनी चाहिए, न कि स्टोर की सभी फाइलों तक। न्यूनतम विशेषाधिकार के सिद्धांत (जो सबसे पुराने सुरक्षा सिद्धांतों में से एक है) के अनुसार, किसी भी संसाधन को केवल उतनी ही अनुमतियाँ मिलनी चाहिए जितनी उसे आवश्यक कार्यों को पूरा करने के लिए चाहिए। AI इस मामले को चुनौतीपूर्ण बनाता है क्योंकि इसे लचीला बनाने के लिए आवश्यक अनुमतियों को सटीक रूप से परिभाषित करना मुश्किल हो सकता है।

### जोखिम  
- अत्यधिक अनुमतियाँ देने से MCP सर्वर को ऐसे डेटा तक पहुँच या संशोधन की अनुमति मिल सकती है जिसे वह एक्सेस करने के लिए अभिप्रेत नहीं था। यदि डेटा व्यक्तिगत पहचान योग्य जानकारी (PII) है तो यह गोपनीयता समस्या भी हो सकती है।

### निवारक नियंत्रण  
- **न्यूनतम विशेषाधिकार का सिद्धांत लागू करें:** MCP सर्वर को केवल आवश्यक न्यूनतम अनुमतियाँ दें। नियमित रूप से इन अनुमतियों की समीक्षा और अद्यतन करें ताकि वे आवश्यक से अधिक न हों। विस्तृत मार्गदर्शन के लिए देखें [Secure least-privileged access](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access)।
- **Role-Based Access Control (RBAC) का उपयोग करें:** MCP सर्वर को ऐसे रोल असाइन करें जो विशिष्ट संसाधनों और क्रियाओं तक सीमित हों, और व्यापक या अनावश्यक अनुमतियों से बचें।
- **अनुमतियों की निगरानी और ऑडिट करें:** अनुमतियों के उपयोग की निरंतर निगरानी करें और एक्सेस लॉग का ऑडिट करें ताकि अत्यधिक या अप्रयुक्त विशेषाधिकारों का तुरंत पता चल सके और उन्हें ठीक किया जा सके।

# Indirect prompt injection attacks

### समस्या विवरण

दुष्ट या समझौता किए गए MCP सर्वर ग्राहक डेटा उजागर कर सकते हैं या अनपेक्षित क्रियाएँ करवा सकते हैं, जिससे महत्वपूर्ण जोखिम उत्पन्न होते हैं। ये जोखिम विशेष रूप से AI और MCP-आधारित वर्कलोड्स में प्रासंगिक हैं, जहाँ:

- **Prompt Injection Attacks**: हमलावर प्रॉम्प्ट या बाहरी सामग्री में दुर्भावनापूर्ण निर्देश छुपाते हैं, जिससे AI सिस्टम अनचाहे कार्य करता है या संवेदनशील डेटा लीक करता है। अधिक जानें: [Prompt Injection](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/)
- **Tool Poisoning**: हमलावर टूल मेटाडेटा (जैसे विवरण या पैरामीटर) को बदलकर AI के व्यवहार को प्रभावित करते हैं, जिससे सुरक्षा नियंत्रणों को बायपास या डेटा चोरी हो सकती है। विवरण: [Tool Poisoning](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks)
- **Cross-Domain Prompt Injection**: दस्तावेज़ों, वेब पेजों या ईमेल में दुर्भावनापूर्ण निर्देश छुपाए जाते हैं, जिन्हें AI प्रोसेस करता है, जिससे डेटा लीक या हेरफेर होता है।
- **Dynamic Tool Modification (Rug Pulls)**: उपयोगकर्ता की मंजूरी के बाद टूल परिभाषाएँ बदली जा सकती हैं, जिससे नए दुर्भावनापूर्ण व्यवहार बिना उपयोगकर्ता के ज्ञान के जुड़ जाते हैं।

ये कमजोरियाँ MCP सर्वरों और टूल्स को आपके वातावरण में जोड़ते समय मजबूत सत्यापन, निगरानी, और सुरक्षा नियंत्रणों की आवश्यकता को दर्शाती हैं। गहराई से जानने के लिए ऊपर दिए गए लिंक देखें।

**Indirect Prompt Injection** (जिसे cross-domain prompt injection या XPIA भी कहा जाता है) जेनरेटिव AI सिस्टम्स में एक गंभीर कमजोरि है, जिनमें Model Context Protocol (MCP) का उपयोग होता है। इस हमले में, दुर्भावनापूर्ण निर्देश बाहरी सामग्री—जैसे दस्तावेज़, वेब पेज, या ईमेल—में छिपाए जाते हैं। जब AI सिस्टम इस सामग्री को प्रोसेस करता है, तो यह छिपे हुए निर्देशों को वैध उपयोगकर्ता आदेश समझ सकता है, जिससे डेटा लीक, हानिकारक सामग्री निर्माण, या उपयोगकर्ता इंटरैक्शन में हेरफेर जैसे अनचाहे परिणाम हो सकते हैं। विस्तृत व्याख्या और वास्तविक उदाहरणों के लिए देखें [Prompt Injection](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/)।

इस हमले का एक विशेष रूप है **Tool Poisoning**। इसमें हमलावर MCP टूल्स के मेटाडेटा (जैसे टूल विवरण या पैरामीटर) में दुर्भावनापूर्ण निर्देश इंजेक्ट करते हैं। चूंकि बड़े भाषा मॉडल (LLMs) इस मेटाडेटा पर निर्भर करते हैं यह तय करने के लिए कि कौन से टूल कॉल करने हैं, समझौता किए गए विवरण मॉडल को अनधिकृत टूल कॉल करने या सुरक्षा नियंत्रणों को बायपास करने के लिए भ्रमित कर सकते हैं। ये हेरफेर अक्सर उपयोगकर्ताओं को दिखाई नहीं देते लेकिन AI सिस्टम द्वारा समझे और कार्यान्वित किए जाते हैं। यह जोखिम होस्टेड MCP सर्वर वातावरण में बढ़ जाता है, जहां उपयोगकर्ता की मंजूरी के बाद टूल परिभाषाएँ बदली जा सकती हैं—जिसे कभी-कभी "[rug pull](https://www.wiz.io/blog/mcp-security-research-briefing#remote-servers-22)" कहा जाता है। ऐसे मामलों में, एक टूल जो पहले सुरक्षित था, बाद में बिना उपयोगकर्ता के ज्ञान के दुर्भावनापूर्ण कार्य कर सकता है, जैसे डेटा चोरी या सिस्टम व्यवहार में बदलाव। इस हमले के बारे में अधिक जानकारी के लिए देखें [Tool Poisoning](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks)।

## जोखिम
अनपेक्षित AI क्रियाएँ कई प्रकार के सुरक्षा जोखिम प्रस्तुत करती हैं जिनमें डेटा चोरी और गोपनीयता उल्लंघन शामिल हैं।

### निवारक नियंत्रण
### अप्रत्यक्ष प्रॉम्प्ट इंजेक्शन हमलों से बचाव के लिए Prompt Shields का उपयोग
-----------------------------------------------------------------------------

**AI Prompt Shields** Microsoft द्वारा विकसित एक समाधान है जो सीधे और अप्रत्यक्ष दोनों प्रकार के प्रॉम्प्ट इंजेक्शन हमलों से बचाव करता है। ये निम्नलिखित तरीकों से मदद करते हैं:

1.  **डिटेक्शन और फिल्टरिंग**: Prompt Shields उन्नत मशीन लर्निंग एल्गोरिदम और प्राकृतिक भाषा प्रसंस्करण का उपयोग करके दस्तावेज़ों, वेब पेजों, या ईमेल जैसी बाहरी सामग्री में छिपे दुर्भावनापूर्ण निर्देशों का पता लगाते और उन्हें फिल्टर करते हैं।
    
2.  **Spotlighting**: यह तकनीक AI सिस्टम को वैध सिस्टम निर्देशों और संभावित अविश्वसनीय बाहरी इनपुट के बीच अंतर करने में मदद करती है। इनपुट टेक्स्ट को इस तरह परिवर्तित करना जिससे मॉडल के लिए यह अधिक प्रासंगिक हो जाता है, Spotlighting AI को दुर्भावनापूर्ण निर्देशों की पहचान और उन्हें नजरअंदाज करने में सक्षम बनाता है।
    
3.  **डेलिमीटर्स और डेटामार्किंग**: सिस्टम संदेश में डेलिमीटर्स शामिल करने से इनपुट टेक्स्ट के स्थान को स्पष्ट रूप से दर्शाया जाता है, जिससे AI सिस्टम उपयोगकर्ता इनपुट और संभावित हानिकारक बाहरी सामग्री को अलग पहचान सके। डेटामार्किंग इस अवधारणा को आगे बढ़ाते हुए विशेष मार्करों का उपयोग करता है ताकि भरोसेमंद और अविश्वसनीय डेटा की सीमाओं को हाइलाइट किया जा सके।
    
4.  **निरंतर निगरानी और अपडेट्स**: Microsoft लगातार Prompt Shields की निगरानी और अपडेट करता है ताकि नए और विकसित होते खतरों का मुकाबला किया जा सके। यह सक्रिय दृष्टिकोण सुनिश्चित करता है कि सुरक्षा उपाय नवीनतम हमले तकनीकों के खिलाफ प्रभावी बने रहें।
    
5. **Azure Content Safety के साथ एकीकरण:** Prompt Shields व्यापक Azure AI Content Safety सूट का हिस्सा हैं, जो AI अनुप्रयोगों में जेलब्रेक प्रयासों, हानिकारक सामग्री, और अन्य सुरक्षा जोखिमों का पता लगाने के लिए अतिरिक्त उपकरण प्रदान करता है।

AI Prompt Shields के बारे में अधिक पढ़ने के लिए देखें [Prompt Shields documentation](https://learn.microsoft.com/azure/ai-services/content-safety/concepts/jailbreak-detection)।

### Supply chain security

AI युग में सप्लाई चेन सुरक्षा मौलिक बनी हुई है, लेकिन अब आपकी सप्लाई चेन की परिभाषा का दायरा बढ़ गया है। पारंपरिक कोड पैकेजों के अलावा, अब आपको सभी AI-संबंधित घटकों जैसे फाउंडेशन मॉडल, एम्बेडिंग सेवाएं, कॉन्टेक्स्ट प्रोवाइडर्स, और थर्ड-पार्टी API की भी कड़ाई से जांच और निगरानी करनी होगी। इनमें से प्रत्येक, यदि सही तरीके से प्रबंधित न किया जाए, तो कमजोरियाँ या जोखिम पैदा कर सकता है।

**AI और MCP के लिए मुख्य सप्लाई चेन सुरक्षा प्रथाएँ:**
- **एकीकरण से पहले सभी घटकों का सत्यापन करें:** इसमें केवल ओपन-सोर्स लाइब्रेरी ही नहीं, बल्कि AI मॉडल, डेटा स्रोत, और बाहरी API भी शामिल हैं। हमेशा उत्पत्ति, लाइसेंसिंग, और ज्ञात कमजोरियों की जांच करें।
- **सुरक्षित डिप्लॉयमेंट पाइपलाइनों को बनाए रखें:** स्वचालित CI/CD पाइपलाइन का उपयोग करें जिसमें सुरक्षा स्कैनिंग शामिल हो ताकि समस्याओं का जल्दी पता चल सके। सुनिश्चित करें कि केवल विश्वसनीय आर्टिफैक्ट्स ही प्रोडक्शन में डिप्लॉय हों।
- **निरंतर निगरानी और ऑडिट करें:** सभी निर्भरताओं, मॉडल और डेटा सेवाओं की लगातार निगरानी करें ताकि नए कमजोरियों या सप्लाई चेन हमलों का पता चल सके।
- **न्यूनतम विशेषाधिकार और एक्सेस नियंत्रण लागू करें:** मॉडल, डेटा, और सेवाओं तक पहुँच को केवल MCP सर्वर के आवश्यक कार्यों तक सीमित रखें।
- **खतरों का शीघ्र प्रतिक्रिया दें:** समझौता किए गए घटकों को पैच या बदलने की प्रक्रिया रखें, और यदि उल्लंघन पाया जाए तो सीक्रेट्स या क्रेडेंशियल्स को रोटेट करें।

[GitHub Advanced Security](https://github.com/security/advanced-security) में सीक्रेट स्कैनिंग, निर्भरता स्कैनिंग, और CodeQL विश्लेषण जैसे फीचर्स हैं। ये उपकरण [Azure DevOps](https://azure.microsoft.com/en-us/products/devops) और [Azure Repos](https://azure.microsoft.com/en-us/products/devops/repos/) के साथ एकीकृत होकर टीमों को कोड और AI सप्लाई चेन घटकों में कमजोरियों की पहचान और निवारण में मदद करते हैं।

Microsoft भी आंतरिक रूप से सभी उत्पादों के लिए व्यापक सप्लाई चेन सुरक्षा प्रथाओं को लागू करता है। इसके बारे में अधिक जानने के लिए देखें [The Journey to Secure the Software Supply Chain at Microsoft](https://devblogs.microsoft.com/engineering-at-microsoft/the-journey-to-secure-the-software-supply-chain-at-microsoft/)।
