# सुरक्षा सर्वोत्तम पद्धती

Model Context Protocol (MCP) स्वीकारल्याने AI-चालित अनुप्रयोगांना शक्तिशाली नवीन क्षमता मिळतात, पण त्याचबरोबर पारंपरिक सॉफ्टवेअर धोके ओलांडणाऱ्या अनोख्या सुरक्षा आव्हानांचीही निर्मिती होते. सुरक्षित कोडिंग, किमान अधिकार, आणि पुरवठा साखळी सुरक्षा यांसारख्या प्रस्थापित चिंतांव्यतिरिक्त, MCP आणि AI कार्यभारांना नवीन धोके जसे की prompt injection, tool poisoning, आणि dynamic tool modification यांचा सामना करावा लागतो. योग्य व्यवस्थापन न झाल्यास हे धोके डेटा चोरी, गोपनीयतेचे उल्लंघन, आणि अनपेक्षित प्रणाली वर्तन यांना कारणीभूत ठरू शकतात.

हा धडा MCP शी संबंधित सर्वात महत्त्वाच्या सुरक्षा धोके—जसे की प्रमाणीकरण, अधिकृतता, जास्त अधिकार, अप्रत्यक्ष prompt injection, आणि पुरवठा साखळीतील असुरक्षितता—यांचा अभ्यास करतो आणि त्यांना कमी करण्यासाठी उपयुक्त नियंत्रण व सर्वोत्तम पद्धती प्रदान करतो. तुम्हाला Microsoft च्या Prompt Shields, Azure Content Safety, आणि GitHub Advanced Security सारख्या उपायांचा वापर करून तुमच्या MCP अंमलबजावणीला कसे मजबूत करायचे हे देखील शिकवले जाईल. या नियंत्रणांचा समज आणि अंमलबजावणी करून तुम्ही सुरक्षा भंग होण्याची शक्यता लक्षणीयरीत्या कमी करू शकता आणि तुमची AI प्रणाली मजबूत व विश्वासार्ह ठेवू शकता.

# शिकण्याचे उद्दिष्ट

या धड्याच्या शेवटी, तुम्ही सक्षम असाल:

- Model Context Protocol (MCP) मुळे उद्भवणाऱ्या अनोख्या सुरक्षा धोके ओळखणे आणि समजावून सांगणे, ज्यात prompt injection, tool poisoning, जास्त अधिकार, आणि पुरवठा साखळीतील असुरक्षितता यांचा समावेश आहे.
- MCP सुरक्षा धोके कमी करण्यासाठी प्रभावी नियंत्रण जसे की मजबूत प्रमाणीकरण, किमान अधिकार, सुरक्षित टोकन व्यवस्थापन, आणि पुरवठा साखळीची पडताळणी यांचा वापर करणे.
- Microsoft च्या Prompt Shields, Azure Content Safety, आणि GitHub Advanced Security सारख्या उपायांचा वापर करून MCP आणि AI कार्यभारांचे संरक्षण करणे.
- टूल मेटाडेटाची पडताळणी करण्याचे महत्त्व ओळखणे, डायनॅमिक बदलांवर लक्ष ठेवणे, आणि अप्रत्यक्ष prompt injection हल्ल्यांपासून संरक्षण करणे.
- सुरक्षित कोडिंग, सर्व्हर हार्डनिंग, आणि झिरो ट्रस्ट आर्किटेक्चर यांसारख्या प्रस्थापित सुरक्षा सर्वोत्तम पद्धती MCP अंमलबजावणीत समाकलित करून सुरक्षा भंग होण्याची शक्यता आणि परिणाम कमी करणे.

# MCP सुरक्षा नियंत्रण

कोणत्याही प्रणालीला महत्त्वाच्या संसाधनांवर प्रवेश असल्यास सुरक्षा आव्हाने असतात. सुरक्षा आव्हाने सामान्यतः मूलभूत सुरक्षा नियंत्रण आणि संकल्पनांच्या योग्य वापराने हाताळली जातात. MCP हे नव्याने परिभाषित केलेले असल्याने, त्याचा तपशील वेगाने बदलत आहे आणि प्रोटोकॉल विकसित होत आहे. शेवटी त्यातील सुरक्षा नियंत्रण परिपक्व होतील, ज्यामुळे ते एंटरप्राइझ आणि प्रस्थापित सुरक्षा आर्किटेक्चर व सर्वोत्तम पद्धतींसह चांगले एकत्रित होतील.

[Microsoft Digital Defense Report](https://aka.ms/mddr) मध्ये प्रकाशित संशोधनानुसार, नोंदवलेल्या ९८% सुरक्षा भंगांना मजबूत सुरक्षा स्वच्छता आणि मूलभूत सुरक्षा पद्धतींचा योग्य वापर करून टाळता येऊ शकते—जसे की सुरक्षित कोडिंग सर्वोत्तम पद्धती आणि पुरवठा साखळी सुरक्षा. या पारखलेल्या पद्धतींनी सुरक्षा धोका कमी करण्यात सर्वाधिक परिणामकारकता आहे.

MCP स्वीकारताना सुरक्षा धोके कसे हाताळायचे याकडे पाहूया.

> **Note:** खालील माहिती **२९ मे २०२५** पर्यंत बरोबर आहे. MCP प्रोटोकॉल सतत विकसित होत आहे, आणि भविष्यातील अंमलबजावणीत नवीन प्रमाणीकरण पद्धती आणि नियंत्रण येऊ शकतात. ताज्या अद्यतनांसाठी आणि मार्गदर्शनासाठी नेहमी [MCP Specification](https://spec.modelcontextprotocol.io/) आणि अधिकृत [MCP GitHub repository](https://github.com/modelcontextprotocol) व [security best practice page](https://modelcontextprotocol.io/specification/draft/basic/security_best_practices) पहा.

### समस्या विधान  
मूळ MCP तपशीलात असे गृहित धरले गेले होते की विकसक स्वतःचा प्रमाणीकरण सर्व्हर लिहतील. यासाठी OAuth आणि संबंधित सुरक्षा निर्बंधांची माहिती आवश्यक होती. MCP सर्व्हर OAuth 2.0 Authorization Servers प्रमाणे काम करत होते, जे वापरकर्त्यांचे प्रमाणीकरण थेट व्यवस्थापित करत होते, बाह्य सेवा जसे Microsoft Entra ID यांना सोपवण्याऐवजी. **२६ एप्रिल २०२५** पासून, MCP तपशीलात एक अद्यतन आले आहे ज्यामुळे MCP सर्व्हर वापरकर्त्यांचे प्रमाणीकरण बाह्य सेवेला सोपवू शकतात.

### धोके
- MCP सर्व्हरमधील चुकीची अधिकृतता लॉजिक संवेदनशील डेटाचा उघड होण्यास आणि चुकीच्या प्रवेश नियंत्रणास कारणीभूत ठरू शकते.
- स्थानिक MCP सर्व्हरवर OAuth टोकन चोरी होऊ शकते. चोरी झाल्यास, टोकनचा वापर करून MCP सर्व्हरची नक्कल करून त्या टोकनसाठी असलेल्या सेवांमधील संसाधन आणि डेटा वापरता येऊ शकतो.

#### Token Passthrough
Authorization specification मध्ये टोकन पासथ्रू स्पष्टपणे मनाई आहे कारण यामुळे अनेक सुरक्षा धोके निर्माण होतात, ज्यात:

#### सुरक्षा नियंत्रण टाळणे
MCP सर्व्हर किंवा डाउनस्ट्रीम API महत्त्वाचे सुरक्षा नियंत्रण जसे की rate limiting, विनंती पडताळणी, किंवा ट्रॅफिक मॉनिटरिंग लागू करतात, जे टोकन ऑडियन्स किंवा इतर क्रेडेन्शियल निर्बंधांवर अवलंबून असतात. जर क्लायंट्सना टोकन थेट डाउनस्ट्रीम API सोबत वापरण्याची परवानगी दिली गेली आणि MCP सर्व्हरने त्यांची योग्य पडताळणी केली नाही किंवा टोकन योग्य सेवेकरिता जारी झाले आहेत याची खात्री केली नाही, तर हे नियंत्रण टाळले जातात.

#### जबाबदारी आणि ऑडिट ट्रेल समस्या
MCP सर्व्हरला MCP क्लायंट्स ओळखणे किंवा वेगळे करणे शक्य नसते जेव्हा क्लायंट्स अपस्ट्रीम जारी केलेले अॅक्सेस टोकन वापरतात, जे MCP सर्व्हरला अस्पष्ट असू शकते.
डाउनस्ट्रीम रिसोर्स सर्व्हरच्या लॉगमध्ये असे विनंत्या दिसू शकतात ज्या वेगळ्या स्रोताकडून किंवा वेगळ्या ओळखीने आल्या आहेत, प्रत्यक्षात टोकन फॉरवर्ड करणारा MCP सर्व्हर नसून.
हे दोन्ही घटक घटना तपासणी, नियंत्रण, आणि ऑडिटिंग अधिक कठीण करतात.
जर MCP सर्व्हर टोकनच्या दाव्यांची (उदा. भूमिका, अधिकार, किंवा ऑडियन्स) किंवा इतर मेटाडेटाची पडताळणी न करता टोकन पास करतो, तर चोरी केलेल्या टोकनसह असलेला दुष्ट व्यक्ती डेटा चोरीसाठी सर्व्हरचा प्रॉक्सी म्हणून वापर करू शकतो.

#### विश्वास मर्यादा समस्या
डाउनस्ट्रीम रिसोर्स सर्व्हर विशिष्ट घटकांना विश्वास देतो. हा विश्वास मूळ किंवा क्लायंट वर्तनाच्या नमुन्यांवर आधारित असू शकतो. ही विश्वास मर्यादा मोडल्यास अनपेक्षित समस्या उद्भवू शकतात.
जर टोकन योग्य पडताळणीशिवाय अनेक सेवांनी स्वीकारले, तर एका सेवेला हल्ला करणारा त्या टोकनचा वापर करून इतर जोडलेल्या सेवांमध्ये प्रवेश करू शकतो.

#### भविष्यातील सुसंगतता धोका
आज MCP सर्व्हर “शुद्ध प्रॉक्सी” म्हणून सुरू झाला तरी, भविष्यात सुरक्षा नियंत्रण जोडण्याची गरज भासू शकते. योग्य टोकन ऑडियन्स वेगळेपणाने सुरुवात केल्यास सुरक्षा मॉडेल विकसित करणे सोपे होते.

### कमी करण्यासाठी नियंत्रण

**MCP सर्व्हरने असे कोणतेही टोकन स्वीकारू नयेत जे स्पष्टपणे MCP सर्व्हरसाठी जारी केलेले नाहीत**

- **अधिकृतता लॉजिक पुनरावलोकन आणि कडक करा:** तुमच्या MCP सर्व्हरच्या अधिकृतता अंमलबजावणीची काळजीपूर्वक तपासणी करा जेणेकरून फक्त इच्छित वापरकर्ते आणि क्लायंट्स संवेदनशील संसाधनांवर प्रवेश करू शकतील. व्यावहारिक मार्गदर्शनासाठी [Azure API Management Your Auth Gateway For MCP Servers | Microsoft Community Hub](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690) आणि [Using Microsoft Entra ID To Authenticate With MCP Servers Via Sessions - Den Delimarsky](https://den.dev/blog/mcp-server-auth-entra-id-session/) पहा.
- **सुरक्षित टोकन पद्धती लागू करा:** [Microsoft च्या टोकन पडताळणी आणि कालावधीसाठी सर्वोत्तम पद्धती](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens) वापरा जेणेकरून अॅक्सेस टोकनचा गैरवापर, पुनरावृत्ती किंवा चोरीचा धोका कमी होईल.
- **टोकन साठवण सुरक्षित करा:** टोकन नेहमी सुरक्षित ठिकाणी साठवा आणि ते विश्रांतीत आणि ट्रान्झिटमध्ये एन्क्रिप्शन वापरून संरक्षित करा. अंमलबजावणी टिप्ससाठी [Use secure token storage and encrypt tokens](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2) पहा.

# MCP सर्व्हरसाठी जास्त अधिकार

### समस्या विधान
MCP सर्व्हरला ज्या सेवा/संसाधनावर प्रवेश आहे, त्यासाठी त्यांना जास्त अधिकार दिले गेले असू शकतात. उदाहरणार्थ, एखादा MCP सर्व्हर जो एंटरप्राइझ डेटा स्टोअरशी जोडलेला AI विक्री अनुप्रयोगाचा भाग आहे, त्याला फक्त विक्री डेटावर प्रवेश असावा, स्टोअरमधील सर्व फाइल्सवर नाही. किमान अधिकाराचा तत्त्व (सुरक्षा तत्त्वांपैकी एक जुना तत्त्व) लक्षात घेता, कोणत्याही संसाधनाला त्याच्या कार्यासाठी आवश्यक त्या अधिकारांपेक्षा जास्त अधिकार असू नयेत. AI या क्षेत्रात लवचिकता सक्षम करण्यासाठी आवश्यक अधिकार निश्चित करणे आव्हानात्मक असते.

### धोके  
- जास्त अधिकार दिल्यामुळे MCP सर्व्हरला न दिलेला डेटा चोरी करणे किंवा बदलणे शक्य होते. जर डेटा वैयक्तिक ओळखण्यायोग्य माहिती (PII) असेल तर गोपनीयतेचा प्रश्न निर्माण होऊ शकतो.

### कमी करण्यासाठी नियंत्रण
- **किमान अधिकार तत्त्व लागू करा:** MCP सर्व्हरला फक्त आवश्यक कार्ये पार पाडण्यासाठी आवश्यक किमान अधिकार द्या. नियमितपणे या अधिकारांची पुनरावलोकने करा आणि अद्यतन करा जेणेकरून ते आवश्यकतेपेक्षा जास्त होऊ नयेत. सविस्तर मार्गदर्शनासाठी [Secure least-privileged access](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access) पहा.
- **भूमिकाधारित प्रवेश नियंत्रण (RBAC) वापरा:** MCP सर्व्हरला विशिष्ट संसाधने आणि क्रियांशी घट्ट संबंधित भूमिका द्या, व्यापक किंवा अनावश्यक अधिकार टाळा.
- **अधिकारांचे निरीक्षण आणि ऑडिट करा:** अधिकार वापर सतत तपासा आणि प्रवेश लॉगचे ऑडिट करा जेणेकरून जास्त किंवा न वापरलेले अधिकार त्वरीत ओळखून दुरुस्त करता येतील.

# अप्रत्यक्ष prompt injection हल्ले

### समस्या विधान

दुष्ट किंवा बिघडलेले MCP सर्व्हर ग्राहक डेटा उघड करण्याचा किंवा अनपेक्षित क्रिया करण्याचा धोका निर्माण करतात. हे धोके AI आणि MCP-आधारित कार्यभारांमध्ये विशेषतः महत्त्वाचे आहेत, जिथे:

- **Prompt Injection हल्ले**: हल्लेखोर prompt किंवा बाह्य सामग्रीमध्ये दुष्ट सूचना लपवतात, ज्यामुळे AI प्रणाली अनपेक्षित क्रिया करते किंवा संवेदनशील डेटा लीक होतो. अधिक वाचा: [Prompt Injection](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/)
- **Tool Poisoning**: हल्लेखोर टूल मेटाडेटा (उदा. वर्णन किंवा पॅरामीटर्स) बदलून AI च्या वर्तनावर परिणाम करतात, ज्यामुळे सुरक्षा नियंत्रण टाळले जाऊ शकतात किंवा डेटा चोरी होऊ शकते. तपशील: [Tool Poisoning](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks)
- **Cross-Domain Prompt Injection**: दुष्ट सूचना दस्तऐवज, वेब पेजेस, किंवा ईमेलमध्ये लपविल्या जातात, ज्यावर AI प्रक्रिया करून डेटा लीक किंवा फेरफार होतो.
- **Dynamic Tool Modification (Rug Pulls)**: वापरकर्त्याच्या मंजुरीनंतर टूल व्याख्या बदलल्या जाऊ शकतात, ज्यामुळे नवीन दुष्ट वर्तन येते.

हे असुरक्षिततेचे मुद्दे MCP सर्व्हर आणि टूल्स तुमच्या वातावरणात समाकलित करताना मजबूत पडताळणी, निरीक्षण, आणि सुरक्षा नियंत्रणांची गरज अधोरेखित करतात. अधिक सखोल माहिती वरील संदर्भांमध्ये पाहा.

![prompt-injection-lg-2048x1034](../../../translated_images/prompt-injection.ed9fbfde297ca877c15bc6daa808681cd3c3dc7bf27bbbda342ef1ba5fc4f52d.mr.png)

**अप्रत्यक्ष Prompt Injection** (ज्याला cross-domain prompt injection किंवा XPIA देखील म्हणतात) हे जनरेटिव्ह AI प्रणालींमधील एक गंभीर असुरक्षितता आहे, ज्यात Model Context Protocol (MCP) वापरले जाते. या हल्ल्यात, दुष्ट सूचना बाह्य सामग्रीमध्ये लपविल्या जातात—जसे की दस्तऐवज, वेब पेजेस, किंवा ईमेल. जेव्हा AI प्रणाली ही सामग्री प्रक्रिया करते, तेव्हा ती लपविलेल्या सूचनांना वैध वापरकर्ता आदेश समजून अनपेक्षित क्रिया करते, जसे की डेटा लीक, हानिकारक सामग्री निर्मिती, किंवा वापरकर्ता संवादात फेरफार. सविस्तर स्पष्टीकरण आणि प्रत्यक्ष उदाहरणांसाठी [Prompt Injection](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/) पहा.

या हल्ल्याचा एक विशेष धोकादायक प्रकार म्हणजे **Tool Poisoning**. येथे हल्लेखोर MCP टूल्सच्या मेटाडेटामध्ये (उदा. टूल वर्णन किंवा पॅरामीटर्स) दुष्ट सूचना इंजेक्ट करतात. मोठ्या भाषा मॉडेल्स (LLMs) या मेटाडेटावर अवलंबून असतात की कोणते टूल कॉल करायचे, त्यामुळे बिघडलेले वर्णन मॉडेलला अनधिकृत टूल कॉल करण्यास किंवा सुरक्षा नियंत्रण टाळण्यास प्रवृत्त करू शकते. हे फेरफार वापरकर्त्यांना दिसत नाहीत पण AI प्रणाली त्यांना समजून आणि अंमलात आणू शकते. हा धोका होस्ट केलेल्या MCP सर्व्हर वातावरणात अधिक वाढतो, जिथे वापरकर्त्याच्या मंजुरीनंतर टूल व्याख्या बदलल्या जाऊ शकतात—याला कधी कधी "[rug pull](https://www.wiz.io/blog/mcp-security-research-briefing#remote-servers-22)" म्हणतात. अशा परिस्थितीत, पूर्वी सुरक्षित असलेले टूल नंतर दुष्ट क्रिया करण्यासाठी बदलले जाऊ शकते, जसे की डेटा चोरी किंवा प्रणाली वर्तनात फेरफार, वापरकर्त्याच्या माहितीसह न करता. या हल्ल्याविषयी अधिक माहितीसाठी [Tool Poisoning](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks) पहा.

![tool-injection-lg-2048x1239 (1)](../../../translated_images/tool-injection.3b0b4a6b24de6befe7d3afdeae44138ef005881aebcfc84c6f61369ce31e3640.mr.png)

## धोके
अनपेक्षित AI क्रिया विविध सुरक्षा धोके निर्माण करतात ज्यात डेटा चोरी आणि गोपनीयतेचे उल्लंघन यांचा समावेश आहे.

### कमी करण्यासाठी नियंत्रण
### अप्रत्यक्ष Prompt Injection हल्ल्यांपासून संरक्षणासाठी prompt shields चा वापर
-----------------------------------------------------------------------------

**AI Prompt Shields** हे Microsoft द्वारे विकसित केलेले एक उपाय आहे जे थेट आणि अप्रत्यक्ष prompt injection हल्ल्यांपासून संरक्षण करतो. ते खालील प्रकारे मदत करतात:

1.  **शोध आणि फिल्टरिंग**: Prompt Shields प्रगत मशीन लर्निंग अल्गोरिदम आणि नैसर्गिक भाषा प्रक्रिया वापरून दस्तऐवज, वेब पेजेस, किंवा ईमेलमध्ये लपविलेल्या दुष्ट सूचनांचा शोध घेतात आणि त्यांना फिल्टर करतात.
    
2.  **Spotlighting**: ही तंत्रज्ञान AI प्रणालीला वैध प्रणाली सूचना आणि संभाव्य अविश्वसनीय बाह्य इनपुट यामध्ये फरक ओळखण्यास मदत करते. इनपुट मजकूराला अशा प्रकारे रूपांतरित करून जे मॉडेलसाठी अधिक संबंधित बनते, Spotlighting AI ला दुष्ट सूचनांना ओळखून दुर्लक्षित करण्यास सक्षम करते.
    
3.  **Delimiters आणि Datamarking**: सिस्टम मेसेजमध्ये delimiters समाविष्ट करून इनपुट मजकूराचा ठिकाण स्पष्टपणे दर्शविला जातो, ज्यामुळे AI प्रणाली वापरकर्ता इनपुट आणि संभाव्य हानिकारक बाह्य
सप्लाय चेन सुरक्षा AI युगात अत्यंत महत्त्वाची आहे, परंतु तुमच्या सप्लाय चेनचा व्याप्ती आता विस्तारित झाली आहे. पारंपरिक कोड पॅकेजेसव्यतिरिक्त, तुम्हाला आता सर्व AI-संबंधित घटक, जसे की फाउंडेशन मॉडेल्स, एम्बेडिंग सेवा, संदर्भ प्रदाते आणि तृतीय-पक्ष API यांची काटेकोरपणे पडताळणी आणि देखरेख करणे आवश्यक आहे. जर योग्य प्रकारे व्यवस्थापित केले नाही तर प्रत्येक घटकात असुरक्षितता किंवा धोके उद्भवू शकतात.

**AI आणि MCP साठी मुख्य सप्लाय चेन सुरक्षा पद्धती:**
- **एकत्रीकरणापूर्वी सर्व घटकांची पडताळणी करा:** यात फक्त ओपन-सोर्स लायब्ररीच नव्हे तर AI मॉडेल्स, डेटा स्रोत आणि बाह्य API देखील समाविष्ट आहेत. नेहमीच मूळ, परवाना आणि ज्ञात असुरक्षितता तपासा.
- **सुरक्षित डिप्लॉयमेंट पाइपलाइन राखा:** सुरुवातीपासूनच समस्या ओळखण्यासाठी स्वयंचलित CI/CD पाइपलाइनमध्ये सुरक्षा स्कॅनिंग समाकलित करा. फक्त विश्वासार्ह आर्टिफॅक्ट्सच उत्पादनात डिप्लॉय करा.
- **सतत देखरेख आणि ऑडिट करा:** सर्व अवलंबित्वांसाठी, ज्यात मॉडेल्स आणि डेटा सेवा यांचा समावेश आहे, सतत देखरेख करा जेणेकरून नवीन असुरक्षितता किंवा सप्लाय चेन हल्ले ओळखता येतील.
- **कमी अधिकार आणि प्रवेश नियंत्रण लागू करा:** मॉडेल्स, डेटा आणि सेवा यांचा प्रवेश फक्त MCP सर्व्हरच्या कार्यासाठी आवश्यक त्या मर्यादेतच ठेवा.
- **धोक्यांना त्वरीत प्रतिसाद द्या:** जर कोणता घटक बिघडला तर त्याचे दुरुस्तीकरण किंवा बदल करण्याची प्रक्रिया ठेवा, तसेच गुपिते किंवा प्रमाणपत्रे बदलण्याची व्यवस्था ठेवा.

[GitHub Advanced Security](https://github.com/security/advanced-security) मध्ये गुपित स्कॅनिंग, अवलंबित्व स्कॅनिंग आणि CodeQL विश्लेषण यांसारख्या वैशिष्ट्यांचा समावेश आहे. हे साधने [Azure DevOps](https://azure.microsoft.com/en-us/products/devops) आणि [Azure Repos](https://azure.microsoft.com/en-us/products/devops/repos/) सोबत एकत्रित होऊन कोड आणि AI सप्लाय चेन घटकांमधील असुरक्षितता शोधण्यात आणि कमी करण्यात मदत करतात.

मायक्रोसॉफ्ट देखील सर्व उत्पादनांसाठी अंतर्गतपणे व्यापक सप्लाय चेन सुरक्षा पद्धती लागू करते. अधिक जाणून घेण्यासाठी [The Journey to Secure the Software Supply Chain at Microsoft](https://devblogs.microsoft.com/engineering-at-microsoft/the-journey-to-secure-the-software-supply-chain-at-microsoft/) वाचा.


# तुमच्या MCP अंमलबजावणीच्या सुरक्षा स्थितीला उंचावणाऱ्या स्थापित सुरक्षा सर्वोत्तम पद्धती

कोणतीही MCP अंमलबजावणी तुमच्या संस्थेच्या आधीच्या सुरक्षा स्थितीवर अवलंबून असते, त्यामुळे MCP ला तुमच्या एकूण AI प्रणालीचा एक घटक म्हणून पाहताना, तुमची एकूण सुरक्षा स्थिती उंचावण्याचा विचार करणे शिफारसीय आहे. खालील स्थापित सुरक्षा नियंत्रण विशेषतः महत्त्वाचे आहेत:

- तुमच्या AI अनुप्रयोगात सुरक्षित कोडिंग सर्वोत्तम पद्धती वापरा - [OWASP Top 10](https://owasp.org/www-project-top-ten/) आणि [OWASP Top 10 for LLMs](https://genai.owasp.org/download/43299/?tmstv=1731900559) विरुद्ध संरक्षण करा, गुपिते आणि टोकन्ससाठी सुरक्षित वॉल्ट वापरा, सर्व अनुप्रयोग घटकांमध्ये एंड-टू-एंड सुरक्षित संवाद राबवा, इत्यादी.
- सर्व्हर हार्डनिंग करा - शक्य असल्यास MFA वापरा, पॅचेस वेळोवेळी अपडेट करा, प्रवेशासाठी तृतीय-पक्ष ओळख प्रदात्याशी सर्व्हर एकत्रित करा, इत्यादी.
- उपकरणे, इन्फ्रास्ट्रक्चर आणि अनुप्रयोग पॅचेससह अद्ययावत ठेवा.
- सुरक्षा देखरेख करा - AI अनुप्रयोग (MCP क्लायंट/सर्व्हर यांसह) चे लॉगिंग आणि मॉनिटरिंग राबवा आणि त्या लॉग्सना केंद्रीकृत SIEM कडे पाठवा जेणेकरून असामान्य क्रियाकलाप ओळखता येतील.
- झिरो ट्रस्ट आर्किटेक्चर वापरा - नेटवर्क आणि ओळख नियंत्रणाद्वारे घटकांना तार्किक पद्धतीने वेगळे करा जेणेकरून AI अनुप्रयोग बिघडल्यास साइडवे लक्षणीय हालचाल कमी करता येईल.

# मुख्य मुद्दे

- सुरक्षा मूलतत्त्वे अत्यंत महत्त्वाची आहेत: सुरक्षित कोडिंग, कमी अधिकार, सप्लाय चेन पडताळणी आणि सतत देखरेख MCP आणि AI कार्यभारांसाठी आवश्यक आहेत.
- MCP नवीन धोके आणते—जसे की प्रॉम्प्ट इंजेक्शन, टूल पॉइझनिंग, आणि जास्त अधिकार, ज्यासाठी पारंपरिक तसेच AI-विशिष्ट नियंत्रण आवश्यक आहेत.
- मजबूत प्रमाणीकरण, अधिकृतता आणि टोकन व्यवस्थापन पद्धती वापरा, शक्य असल्यास Microsoft Entra ID सारख्या बाह्य ओळख प्रदात्यांचा लाभ घ्या.
- अप्रत्यक्ष प्रॉम्प्ट इंजेक्शन आणि टूल पॉइझनिंगपासून संरक्षण करा, टूल मेटाडेटाची पडताळणी करा, डायनॅमिक बदलांवर लक्ष ठेवा, आणि Microsoft Prompt Shields सारखे उपाय वापरा.
- तुमच्या AI सप्लाय चेनमधील सर्व घटकांना—मॉडेल्स, एम्बेडिंग्ज, संदर्भ प्रदाते—कोड अवलंबित्वांप्रमाणेच काटेकोरपणे हाताळा.
- MCP स्पेसिफिकेशन्सच्या बदलांशी अद्ययावत रहा आणि सुरक्षित मानके तयार करण्यासाठी समुदायात योगदान द्या.

हा दस्तऐवज AI अनुवाद सेवा [Co-op Translator](https://github.com/Azure/co-op-translator) वापरून अनुवादित केला आहे. आम्ही अचूकतेसाठी प्रयत्नशील असलो तरी, कृपया लक्षात घ्या की स्वयंचलित अनुवादांमध्ये चुका किंवा अचूकतेची कमतरता असू शकते. मूळ दस्तऐवज त्याच्या स्थानिक भाषेत अधिकृत स्रोत मानला जावा. महत्त्वाच्या माहितीसाठी व्यावसायिक मानवी अनुवाद करण्याची शिफारस केली जाते. या अनुवादाच्या वापरामुळे उद्भवणाऱ्या कोणत्याही गैरसमजुती किंवा चुकीच्या अर्थलागी आम्ही जबाबदार नाही.
